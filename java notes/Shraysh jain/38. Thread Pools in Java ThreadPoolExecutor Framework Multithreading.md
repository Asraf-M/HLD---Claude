## What is a Thread Pool?

A **collection of pre-created threads** that are reused to execute submitted tasks.

```
  Tasks: T1, T2, T3, T4, T5

  Thread Pool (size = 3)          Queue
  ┌────────┬────────┬────────┐   ┌────┬────┐
  │Thread-1│Thread-2│Thread-3│   │ T4 │ T5 │
  │  (T1)  │  (T2)  │  (T3)  │   └────┴────┘
  └────────┴────────┴────────┘
       ↑                              │
       └── when free, picks from ─────┘
```

| Without Pool | With Pool |
|---|---|
| Create new thread per task | Reuse existing threads |
| Thread creation overhead each time | One-time creation cost |
| No control over thread count | Configurable min/max |
| Manual lifecycle management | Framework handles it |

---

## Advantages of Thread Pools

| Advantage | Detail |
|---|---|
| **Save creation time** | Threads are pre-created and reused — no per-task overhead |
| **Lifecycle management** | Framework handles thread states (no manual start/stop) |
| **Performance** | Fewer threads → less context switching → more CPU processing time |
| **Resource control** | Cap max threads to prevent resource exhaustion |

## ThreadPoolExecutor — Constructor & Parameters

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,        // min threads always alive
    maximumPoolSize,     // max threads allowed
    keepAliveTime,       // idle timeout for extra threads
    TimeUnit.MINUTES,    // time unit for keepAliveTime
    workQueue,           // queue to hold pending tasks
    threadFactory,       // custom thread creation
    rejectedHandler      // what to do when task can't be accepted
);
```

### Parameters Explained

| Parameter | Purpose | Example |
|---|---|---|
| `corePoolSize` | Min threads — always alive, even if idle | `3` |
| `maximumPoolSize` | Max threads — created only when core + queue are full | `5` |
| `keepAliveTime` | How long extra threads (beyond core) stay idle before termination | `5` |
| `timeUnit` | Unit for keepAliveTime | `TimeUnit.MINUTES` |
| `workQueue` | Holds tasks when all core threads are busy | `new ArrayBlockingQueue<>(10)` |
| `threadFactory` | Custom thread creation (name, priority, daemon) | `new CustomThreadFactory()` |
| `rejectedHandler` | Handler for tasks that can't be queued or assigned | `new AbortPolicy()` |

### allowCoreThreadTimeOut

```java
executor.allowCoreThreadTimeOut(true);
// Now even core threads are terminated if idle > keepAliveTime
// Default: false (core threads never die)
```

### Queue Types

| Queue | Type | Detail |
|---|---|---|
| `ArrayBlockingQueue` | Bounded | Fixed capacity — preferred ✅ |
| `LinkedBlockingQueue` | Unbounded | No limit — avoid (no backpressure) ❌ |

## Task Submission Flow

```
  New Task Arrives
       │
       ▼
  1. Any core thread free?
       ├── YES → assign task to thread ✅
       │
       └── NO → all core threads busy
                │
                ▼
           2. Any space in queue?
                ├── YES → add task to queue ✅
                │         (thread picks it when free)
                │
                └── NO → queue full
                         │
                         ▼
                    3. Can create more threads?
                       (current < maximumPoolSize)
                         ├── YES → create new thread, assign task ✅
                         │
                         └── NO → max reached
                                  │
                                  ▼
                             4. REJECT task (handler) ❌
```

### Example Walkthrough

```
  Config: core=3, max=5, queue=5

  T1-T3:  assigned to Thread 1-3 ✅ (core threads)
  T4-T8:  added to queue ✅ (5 slots)
  T9:     core busy + queue full → create Thread-4 ✅
  T10:    core busy + queue full → create Thread-5 ✅
  T11:    core busy + queue full + max reached → REJECT ❌
```

## Why Queue Before Max Threads?

**Design rationale:** Core pool size is set for the **average load**. Extra threads are only for **peak bursts**.

```
  Average load: 3 threads handle most requests
  Queue: absorbs temporary spikes with existing 3 threads
  Max threads: only for extreme peaks when queue overflows

  If we created threads immediately instead of queueing:
  → Extra threads complete task, return to pool, sit IDLE
  → Wasted resources on average load
  → Thread pool grows unnecessarily
```

## Rejected Execution Handlers

| Handler | Behavior |
|---|---|
| `AbortPolicy` (default) | Throws `RejectedExecutionException` |
| `DiscardPolicy` | Silently discards the task — no error |
| `CallerRunsPolicy` | Runs the task in the caller's thread (e.g., main) |
| `DiscardOldestPolicy` | Removes oldest task from queue, adds new one |

### Custom Handler

```java
class MyHandler implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        System.out.println("Task rejected: " + r.toString());
        // log, alert, save to DB, etc.
    }
}
```

## Thread Pool Lifecycle

```
  ┌──────────┐    shutdown()    ┌──────────┐     all done     ┌────────────┐
  │ RUNNING  │ ───────────────▶ │ SHUTDOWN │ ────────────────▶ │ TERMINATED │
  │          │                  │          │                   │            │
  │ accepts  │                  │ no new   │                   │ all threads│
  │ new tasks│                  │ tasks    │                   │ eliminated │
  └──────────┘                  │ existing │                   └────────────┘
       │                        │ continue │
       │    shutdownNow()       └──────────┘
       │                              │
       └──────────────────────────────┘
                  │
            ┌─────▼─────┐
            │   STOP    │ → force-stop existing + reject new → TERMINATED
            └───────────┘
```

| State | New Tasks? | Existing Tasks? |
|---|---|---|
| **RUNNING** | ✅ Accepted | ✅ Processing |
| **SHUTDOWN** | ❌ Rejected | ✅ Let them finish |
| **STOP** | ❌ Rejected | ❌ Force-stopped |
| **TERMINATED** | ❌ | ❌ All threads dead |

```java
executor.shutdown();         // graceful — finish existing, reject new
executor.shutdownNow();      // force — stop everything
executor.isTerminated();     // check if fully done
```

## Summary

| Concept | Key Point |
|---|---|
| **Thread Pool** | Pre-created threads reused for tasks — saves creation overhead |
| **corePoolSize** | Min threads always alive |
| **maximumPoolSize** | Max threads — only created when core + queue are full |
| **Queue** | Holds tasks when core threads busy — checked before creating extra threads |
| **keepAliveTime** | Idle timeout for threads beyond core count |
| **Rejected handler** | AbortPolicy (throw), DiscardPolicy (silent), CallerRuns, DiscardOldest |
| **Lifecycle** | RUNNING → SHUTDOWN → TERMINATED |
