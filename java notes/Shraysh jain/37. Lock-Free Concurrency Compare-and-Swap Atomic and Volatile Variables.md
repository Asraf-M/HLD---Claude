## Lock-Based vs Lock-Free Concurrency

```
  Concurrency Mechanisms
  ├── Lock-Based (slower, general-purpose)
  │   ├── synchronized
  │   ├── ReentrantLock
  │   ├── ReadWriteLock
  │   ├── StampedLock
  │   └── Semaphore
  │
  └── Lock-Free (faster, specific use cases)
      └── CAS (Compare-and-Swap)
          ├── AtomicInteger
          ├── AtomicBoolean
          ├── AtomicLong
          └── AtomicReference
```

| Feature | Lock-Based | Lock-Free |
|---|---|---|
| Speed | Slower (blocking) | Faster (no blocking) |
| Use case | Complex business logic | Simple read-modify-update |
| Alternative? | ❌ Lock-free is NOT a replacement | ✅ Only for specific scenarios |

## Compare-and-Swap (CAS) Operation

A **CPU-level atomic operation** supported by all modern processors. Inspired by optimistic locking.

### How CAS Works

```
  CAS(memory, expectedValue, newValue)
       │
       ▼
  1. Read value from memory
       │
       ▼
  2. Compare memory value with expectedValue
       │
       ├── MATCH → update memory with newValue ✅ (return true)
       │
       └── NO MATCH → do nothing ❌ (return false, retry)
```

### CAS vs Optimistic Locking

| Feature | CAS | Optimistic Locking |
|---|---|---|
| Level | CPU-level (hardware) | Database-level (row version) |
| Atomicity | Guaranteed by CPU across all cores | Guaranteed by DB transaction |
| Mechanism | Compare memory + expected value | Compare row version |
| Retry | Loop until success | Re-read, re-modify, re-update |

### CAS Example with Two Threads

```
  Memory: counter = 0

  Thread-1                    CAS                    Thread-2
       │                      │                          │
  read counter=0              │                     read counter=0
  expected=0                  │                     expected=0
       │                      │                          │
       ├── CAS(mem, 0, 1) ───▶│                          │
       │   read mem = 0       │                          │
       │   0 == 0? ✅ MATCH   │                          │
       │   update → 1         │                          │
       │   return true ✅     │                          │
       │                      │◄── CAS(mem, 0, 1) ──────┤
       │                      │   read mem = 1           │
       │                      │   1 == 0? ❌ NO MATCH    │
       │                      │   return false           │
       │                      │                     retry: │
       │                      │                     read counter=1
       │                      │                     expected=1
       │                      │◄── CAS(mem, 1, 2) ──────┤
       │                      │   1 == 1? ✅ → update 2  │
```

## ABA Problem

CAS checks only the **value**, not whether it changed and changed back.

```
  Memory: x = 10 (version 1)

  Thread-1: reads x = 10, expected = 10
  Thread-2: changes x to 12 (version 2)
  Thread-3: changes x back to 10 (version 3)
  Thread-1: CAS(mem, 10, 13) → 10 == 10 ✅ → updates to 13
            BUT the "10" isn't the original 10!
```

### Solution: Add Version Number

```
  Memory: x = 10, version = 1

  Thread-1: reads x=10, version=1
  Thread-2: updates → x=12, version=2
  Thread-3: updates → x=10, version=3

  Thread-1: CAS expects version=1, actual=3 → MISMATCH → retry ✅
```

> Java's `AtomicStampedReference` solves ABA by tracking stamps (versions).

## Atomic Variables

Provide **thread-safe operations** using CAS internally — no locks needed.

### Why `counter++` Is NOT Atomic

```java
counter++;  // actually 3 steps:
// 1. Read counter value
// 2. Increment by 1
// 3. Write back

// Two threads can read same value → lost update!
```

### AtomicInteger Solution

```java
AtomicInteger counter = new AtomicInteger(0);

counter.incrementAndGet();   // atomic! uses CAS internally
counter.addAndGet(5);        // atomic add
counter.get();               // read value
counter.compareAndSet(10, 20); // manual CAS
```

### How incrementAndGet() Works Internally

```java
// Pseudocode of incrementAndGet():
do {
    int expected = value;                    // read from memory (volatile)
} while (!CAS(memory, expected, expected + 1));  // retry until success
return expected + 1;
```

### Available Atomic Classes

| Class | For Type |
|---|---|
| `AtomicInteger` | `int` |
| `AtomicLong` | `long` |
| `AtomicBoolean` | `boolean` |
| `AtomicReference<T>` | Any object |

### When to Use

| Scenario | Use |
|---|---|
| Simple **read-modify-update** (counter, flag) | ✅ Atomic |
| Complex business logic, multiple shared resources | ❌ Use locks |

## Volatile Keyword

Ensures a variable is always **read from and written to main memory** — not CPU cache.

### The Problem Without Volatile

```
  CPU Core 1              CPU Core 2
  ┌──────────┐            ┌──────────┐
  │ L1 Cache │            │ L1 Cache │
  │ x = 11   │            │ (empty)  │
  └────┬─────┘            └────┬─────┘
       │                       │
  ┌────┴───────────────────────┴────┐
  │          L2 Cache               │
  └────────────┬────────────────────┘
               │
  ┌────────────┴────────────────────┐
  │      Main Memory: x = 10       │  ← stale!
  └─────────────────────────────────┘

  Thread-1 (Core 1): x++ → x=11 in L1 cache (not yet synced to memory)
  Thread-2 (Core 2): read x → goes to L2 → memory → gets 10 (STALE!)
```

### With Volatile

```java
volatile int x = 10;
// All reads/writes go directly to main memory — bypass cache
```

```
  Thread-1: x++ → writes 11 directly to MEMORY
  Thread-2: read x → reads directly from MEMORY → gets 11 ✅
```

| Feature | Without `volatile` | With `volatile` |
|---|---|---|
| Read from | CPU cache (possibly stale) | Main memory (always fresh) |
| Write to | CPU cache (sync later) | Main memory (immediately) |
| Visibility | ❌ Other threads may see stale data | ✅ All threads see latest value |

## Atomic vs Volatile

| Feature | Atomic | Volatile |
|---|---|---|
| **Thread safety** | ✅ Yes — CAS guarantees atomicity | ❌ No — only visibility |
| **Purpose** | Safe read-modify-update | Ensure read/write from memory |
| **Operations** | `incrementAndGet()`, `compareAndSet()` | Simple read/write |
| **Use case** | Counters, flags with concurrent updates | Flags read by multiple threads, written by one |
| **Internally uses** | CAS (CPU operation) + volatile | Memory barrier |

```java
// Volatile — visibility only (NOT safe for counter++)
volatile boolean running = true;    // one writer, many readers ✅
volatile int counter = 0;
counter++;                           // ❌ NOT atomic even with volatile!

// Atomic — thread-safe operations
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();           // ✅ atomic
```

## Summary

| Concept | Key Point |
|---|---|
| **CAS** | CPU-level atomic operation: compare expected → update if match |
| **ABA problem** | Value changed and changed back — solved with version/stamp |
| **AtomicInteger** | Lock-free thread-safe counter using CAS |
| **Volatile** | Read/write directly from main memory — visibility, NOT atomicity |
| **Lock-free** | Faster but only for simple read-modify-update scenarios |
| **Lock-based** | General purpose — complex logic, multiple resources |
