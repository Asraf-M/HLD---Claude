# System Design: API Gateway (Beginner-Friendly Guide)

---

## What Are We Building?

The "front door" for all client requests to your microservices platform. Instead of clients talking to 50 different backend services directly, they talk to one API Gateway:
- Gateway authenticates users
- Gateway validates requests
- Gateway routes to the right backend service
- Gateway handles rate limiting (prevent abuse)
- Gateway caches frequently-accessed data
- Gateway adds headers, transforms responses
- Gateway detects when services are down and fails gracefully

Think: Netflix app talks to Netflix API Gateway (not directly to 50 Netflix services). Gmail app talks to Google API Gateway (not directly to Gmail, Contacts, Calendar services).

**Key Engineering Challenges:**
- **Single Point of Failure** — If gateway goes down, entire API is down
- **Latency** — Gateway adds overhead; must be < 10ms per request
- **Scalability** — All traffic flows through gateway; must handle millions of requests/sec
- **Routing Complexity** — 500+ endpoints; route to correct backend service
- **Authentication burden** — Validate every single request (slow if done wrong)
- **Rate limiting** — Prevent one user from consuming all resources
- **Caching decisions** — What to cache? For how long? When to invalidate?

**What the Gateway handles:**

| Responsibility | Implementation |
|---|---|

## Step 3: Routing Strategies

**Path-Based Routing:**
```
GET /api/users/123 → User Service
GET /api/orders/456 → Order Service
POST /api/payments → Payment Service

Pattern matching:
/api/users/* → User Service
/api/orders/* → Order Service
/api/products/* → Product Service
```

**Host-Based Routing:**
```
api.amazon.com/orders → Order Service
api.amazon.com/products → Product Service

Or subdomain:
orders.api.amazon.com → Order Service
products.api.amazon.com → Product Service
```

**Header-Based Routing (Versioning):**
```
GET /api/users
Header: X-API-Version: v2
→ Route to User Service v2

GET /api/users
Header: X-API-Version: v1
→ Route to User Service v1

Allows gradual migration
```

**Example Routing Table:**
```
Method | Path              | Service        | Cache
-------|-------------------|----------------|---------
GET    | /api/users/:id    | User Service   | 5 min
POST   | /api/users        | User Service   | No
GET    | /api/orders/:id   | Order Service  | 1 min
POST   | /api/orders       | Order Service  | No
GET    | /api/products     | Product Service| 1 hour
```

## Step 4: Authentication & Authorization

**Authentication Flow:**
```
1. Client logs in:
   POST /auth/login
   Body: {username: "john", password: "secret"}
   
2. Gateway verifies password:
   - Query user DB
   - Check password hash
   - Generate JWT token
   Response: {access_token: "eyJ...", expires_in: 3600}

3. Client stores token

4. Future requests include token:
   GET /api/orders/123
   Header: Authorization: Bearer eyJ...

5. Gateway validates token:
   - Decode JWT (verify signature)
   - Check not expired
   - Extract user_id
   - Add header: X-User-ID: 456
   - Forward to Order Service
```

**JWT Token Example:**
```
Header:   {alg: HS256, typ: JWT}
Payload:  {user_id: 456, email: "john@example.com", exp: 1718704500}
Signature: HMAC-SHA256(secret)

Encoded: eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0NTZ9.xyz...
```

**Authorization (Role-Based):**
```
GET /api/users/123
- Gateway validates token → user_id=456
- Gateway queries: "What role does user 456 have?"
- Fetches from cache: user_456 → roles=[USER, ADMIN]
- Check endpoint: GET /api/users/123 requires [USER]
- User has USER role → Allowed
- Forward to User Service with headers:
  X-User-ID: 456
  X-User-Roles: USER,ADMIN
```

## Step 5: Rate Limiting

**Token Bucket Algorithm:**
```
Bucket Capacity: 1000 requests
Refill Rate: 100 requests per minute

Time 0:
- Bucket: 1000 tokens
- Request uses 1 token
- Bucket: 999

After 1 second:
- Tokens generated: 100/60 ≈ 1.67
- Bucket: 999 + 1.67 = 1000.67 (capped at 1000)

When bucket empty:
- User at rate limit
- Return 429 (Too Many Requests)
- Retry-After: 60 seconds
```

**Distributed Rate Limiting:**
```
Single Gateway: Easy, in-memory
Multiple Gateways: Need shared state

Solution: Use Redis
Gateway 1: Check Redis for user bucket
Gateway 2: Check Redis for user bucket
          (both see same count)

Redis command:
INCR user:123:rate_limit
→ Current count: 50
→ Allowed? 50 < 1000 ? Yes

EXPIRE user:123:rate_limit 60
→ Reset after 60 seconds
```

**Rate Limit Tiers:**
```
Free tier: 100 requests/hour
Pro tier: 10,000 requests/hour
Enterprise: 1,000,000 requests/hour

Check user's tier before applying limit
```

## Step 6: Caching Strategies

**Response Caching:**
```
GET /api/products/123
Before: Check cache
- Key = GET:/api/products/123
- Found in cache? Return immediately (< 5ms)
- Not found? Call Product Service

Receive response: {id: 123, name: "..."}
After: Store in cache with TTL
- Key = GET:/api/products/123
- Value = response JSON
- TTL = 1 hour (product doesn't change often)
```

**Cache Invalidation (Tricky!):**
```
Scenario:
1. GET /api/products/123 → Cached
2. PATCH /api/products/123 → Update name
3. GET /api/products/123 → Return old cached data (stale!)

Solutions:
a) Short TTL: 1 minute (data stale < 1 min)
b) Event-based: Product Service sends "cache.invalidate" event
c) Manual: Admin endpoint to clear cache

Recommended: Combination of (a) + (b)
```

**Cache Control Headers:**
```
Response from Product Service:
Cache-Control: max-age=3600, public
→ Cache for 1 hour

Cache-Control: max-age=0, must-revalidate
→ Don't cache; always validate with origin

Cache-Control: no-cache
→ Cache but validate before use

Cache-Control: private
→ Cache only for specific user
```

## Step 7: Circuit Breaker Pattern

**Problem:**
```
User Service temporarily slow/down
→ Requests pile up in queue
→ Gateway's thread pool exhausted
→ All requests fail with timeout
→ Entire system cascades

Solution: Circuit Breaker
```

**States:**
```
CLOSED (normal):
- Requests pass through
- Count successes/failures
- If failure rate > 50% over 10 requests → OPEN

OPEN (fail fast):
- Don't call service
- Return error immediately
- Prevent client timeout

HALF_OPEN (recovery test):
- Allow a few requests through
- If successful → CLOSED
- If fail → OPEN again
```

**Implementation:**
```
Transition to OPEN when:
- Failure rate > 50% (over 10+ requests)
- Timeout rate > 30%

Stay OPEN for:
- 30 seconds

Then try HALF_OPEN:
- Send 1 test request
- If success → return to CLOSED
- If fail → stay OPEN, try again in 30s
```

## Step 9: Load Balancing

**Round Robin:**
```
Request 1 → Gateway 1
Request 2 → Gateway 2
Request 3 → Gateway 3
Request 4 → Gateway 1
...

Simple, distributes evenly
```

**Least Connections:**
```
Track active connections per gateway
- Gateway 1: 10 active
- Gateway 2: 15 active
- Gateway 3: 5 active

Route to Gateway 3 (fewest connections)
Good for long-lived connections
```

**IP Hash:**
```
Hash(client_ip) → consistent gateway
- Client 1.2.3.4 → Gateway 1
- Client 5.6.7.8 → Gateway 2
- Client 1.2.3.4 → Gateway 1 (same)

Benefit: Connection affinity (client state)
Risk: If gateway down, some clients reroute to different gateway
```

## Step 12: Key Design Decisions & Tradeoffs

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Centralized Gateway | Single auth point; easy feature addition (rate limit, caching) | Single point of failure; all traffic through it |
| Gateway does auth | Fast (cached); protect backend | More code; security responsibility |
| Response caching | Reduce backend load 40-60%; faster responses | Stale data brief window; cache invalidation complexity |
| Circuit breaker | Fail fast; prevent cascades | Need graceful error handling; complexity |
| Rate limiting in gateway | Protect backend; fair usage | Adds 5-10ms latency; requires distributed state (Redis) |
| Multiple gateways | High availability; load distribution | More complex; requires load balancer |

## Interview Cheat Sheet Q&A

**Q: If the API Gateway has so much responsibility, won't it become a bottleneck?**  
A: Yes, if single instance. Solution: run multiple stateless gateway instances behind a load balancer. Each gateway handles requests independently. Add more gateways to handle more traffic. No session state needed in the gateway (JWT tokens are self-contained).

**Q: Why not let clients talk directly to microservices?**  
A: Then every client needs to know every service's address. If User Service moves to new server, all clients break. API Gateway abstracts this: clients only know gateway. When a service moves, only gateway config changes. Also: different clients need different response formats (web vs mobile), gateway handles transformation.

**Q: Doesn't caching cause stale data?**  
A: Briefly, yes. Solution: use short TTLs (1-5 minutes) for frequently-changing data, longer TTLs (1 hour) for static data. Also, use event-based invalidation: when backend updates data, it sends "invalidate cache" event. Best practice: TTL + event-based hybrid.

**Q: What if the gateway cache is full?**  
A: Use LRU (least recently used) eviction. Remove least-recently-accessed items. Or use probabilistic early expiration: refresh items near end of TTL, serve fresh data before entry is removed.

**Q: How to prevent DDoS attacks at the gateway?**  
A: Multiple layers:
1. Rate limiting (per user, per IP)
2. Circuit breaker (fail fast if overwhelmed)
3. Request validation (reject malformed)
4. IP reputation lists (block known bad IPs)
5. DDoS mitigation service (AWS Shield, Cloudflare)

**Q: Can gateway authenticate without checking database every time?**  
A: Yes! Use JWT tokens: token is self-signed. Gateway decodes and verifies signature (no DB needed). Extract user_id from token. Check token expiration. For role/permission changes, use short token TTL (1 hour) so changes take effect quickly.

## Full Flow (Start to End)

### Happy Path
1. Client request enters API Gateway and is authenticated/authorized.
2. Orchestrator service validates input and routing context.
3. Core service executes primary business logic and required checks.
4. Read path uses cache first; fallback goes to durable database/store.
5. Write path updates fast layer first (where applicable) and publishes async events.
6. Downstream consumers persist durable state and trigger secondary effects.
7. Response is returned to client with final status and metadata.

### Failure and Retry Paths
1. Cache miss: read from durable store, then repopulate cache.
2. Dependency timeout: retry with backoff or circuit-breaker fallback.
3. Async event failure: retry queue and dead-letter queue (DLQ) handling.
4. Duplicate request: idempotency key returns prior successful outcome.
5. Concurrent updates: version/lock conflict triggers re-read and safe retry.

### End-State Guarantees
- Low-latency user operations on the hot path.
- Durable correctness in the source-of-truth datastore.
- Eventual consistency for non-critical async side effects.
- Strict correctness at critical boundaries (commit/payment/finalization).

## Step 7: Circuit Breaker Pattern

### Preventing Cascading Failures

```
Problem:
Backend service slow/down
→ Gateway waits for timeout
→ User requests pile up
→ Gateway overloaded
→ All users affected

Solution: Circuit Breaker

States:
1. CLOSED: Normal operation
   - Requests pass through
   - Success/failure tracked

2. OPEN: Too many failures
   - Fail fast, don't call service
   - Return error to user
   - Prevent overload

3. HALF_OPEN: Recovery attempt
   - Allow some requests through
   - Test if service recovered
   - Return to CLOSED if success
```

### Circuit Breaker Thresholds

```
Transition to OPEN when:
- Failure rate > 50% (over 10 requests)
- Timeout rate > 30%
- Error count > 100 (last 5 min)

Transition to HALF_OPEN after:
- 30 seconds in OPEN state

Transition back to CLOSED when:
- 5 consecutive successful requests
```

## Step 8: Rate Limiting Implementation

### Token Bucket Algorithm

```
Bucket Capacity: 1000 tokens
Refill Rate: 100 tokens/minute

Time 0:
- Tokens: 1000
- Request: 100 tokens consumed
- Remaining: 900

Time 1 second:
- Tokens: 900 + (100/60) ≈ 901.67
- Request: 50 tokens
- Remaining: 851.67

Time exceeds capacity:
- Tokens: Max 1000 (no overflow)
```

### Implementation

```python
class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = capacity
        self.last_refill = time.time()
    
    def consume(self, tokens):
        self.refill()
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        return False
    
    def refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(
            self.capacity,
            self.tokens + elapsed * self.refill_rate
        )
        self.last_refill = now
```

### Distributed Rate Limiting

```
Problem: Multiple Gateway instances
- Gateway 1: 500 requests used
- Gateway 2: 400 requests used
- Combined: 900 (should be limited at 1000)
But counted separately!

Solution: Centralized Rate Limiter
- All gateways check Redis
- Redis atomic operations
- True limit across instances
- Slightly higher latency
```

## Interview Cheat Sheet Q&A

**Q: If the API Gateway has so much responsibility, won't it become a bottleneck?**  
A: Yes, if it's a single instance. Solution: run multiple gateway instances behind a load balancer. Each gateway is stateless (no session storage), so requests can route to any gateway. Add more gateways to handle more traffic.

**Q: Why not let clients talk directly to microservices?**  
A: Then every client needs to know every service's address and auth method. If service X moves to a new server, all clients break. API Gateway abstracts this: clients only know the gateway address. Gateway routes to services. When a service moves, only gateway config changes.

**Q: Doesn't caching in the gateway cause stale data issues?**  
A: Briefly, yes. Solution: use short TTLs (1-5 minutes) for data that changes frequently, longer TTLs (1 hour) for static data. For critical data, use event-based invalidation: when the data service updates a record, it sends a "delete cache" event to the gateway.

**Q: What if the gateway cache is full (1GB limit)?**  
A: LRU eviction: remove least recently used items. Or use probabilistic early expiration: refresh items before TTL expires, so you serve fresh data before the cache entry is removed.

**Q: How do we handle user authentication? Gateway or backend service?**  
A: Gateway: Verify JWT/session in the gateway, extract user_id. Add to request headers (X-User-ID). Send to backend. Backend trusts the gateway (internal traffic only). This saves the backend from validating each request.

**Q: What if one API endpoint is called 100x more than others (hot endpoint)?**  
A: Cache it heavily in the gateway (1-hour TTL). Also, consider a read-through cache: gateway checks cache before querying the backend. For a hot endpoint, 99% of requests hit the cache, backend barely sees them.
