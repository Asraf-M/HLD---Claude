## Why stop(), suspend(), resume() Are Deprecated

### stop()

Terminates the thread **abruptly** — no lock release, no resource cleanup.

```
  Thread-1 acquires Lock on Resource-1
       │
       ▼
  Thread-1.stop() called
       │
       ▼
  Thread DIES — lock NOT released!
       │
       ▼
  Thread-2 waiting for Resource-1 → DEADLOCK (forever waiting)
```

### suspend()

Puts thread on hold **without releasing locks** (unlike `wait()` which releases monitor locks).

```
  Thread-1 acquires Lock → suspended → lock NOT released
  Thread-2 wants Lock → waits forever → DEADLOCK
```

### resume()

Resumes a suspended thread. Since `suspend()` is deprecated, `resume()` must be too.

| Method | Problem | Alternative |
|---|---|---|
| `stop()` | No lock release, no cleanup | `interrupt()` + graceful shutdown |
| `suspend()` | No lock release → deadlock | `wait()` (releases lock) |
| `resume()` | Paired with `suspend()` | `notify()` / `notifyAll()` |

## Thread Joining

`join()` makes the **current thread wait** for the target thread to finish before proceeding.

```java
Thread t1 = new Thread(() -> {
    // long-running task (e.g., 8 seconds)
});
t1.start();

System.out.println("Main waiting for t1...");
t1.join();   // main thread BLOCKS here until t1 finishes
System.out.println("t1 finished, main continues");
```

```
  Main Thread           Thread-1
       │                    │
       ├── t1.start() ────▶│
       │                    │ (doing work...)
       ├── t1.join() ──────▶│
       │  (BLOCKED)         │ (still working...)
       │  (BLOCKED)         │ (done!)
       │◄───────────────────┘
       │ (resumes)
       ▼
```

### Use Cases

| Scenario | How |
|---|---|
| Wait for one thread | `t1.join()` |
| Wait for multiple | `t1.join(); t2.join();` — waits for both before proceeding |
| Timed join | `t1.join(3000)` — wait max 3 seconds, then continue anyway |

## Thread Priority

An integer **hint** to the thread scheduler (1–10). **Not guaranteed** to be followed.

```java
Thread t1 = new Thread(() -> { /* task */ });
t1.setPriority(Thread.MAX_PRIORITY);   // 10
t1.setPriority(Thread.NORM_PRIORITY);  // 5 (default)
t1.setPriority(Thread.MIN_PRIORITY);   // 1
```

| Property | Detail |
|---|---|
| Range | 1 (lowest) to 10 (highest) |
| Default | 5 (`NORM_PRIORITY`) — inherits from parent |
| Guaranteed order? | ❌ No — just a hint to OS scheduler |
| Use in production? | ❌ Avoid — unreliable |

> JVM doesn't guarantee priority order. 9 out of 10 runs may ignore the priority completely.

## Daemon Thread

A background thread that **automatically terminates** when all user threads finish.

### User Thread vs Daemon Thread

```
  Main (user) ─────────────────────┐
       │                           │ main finishes
       ├── User Thread ────────────┤─── keeps running ✅
       │                           │
       ├── Daemon Thread ──────────┤─── DIES automatically ❌
       │                           │
```

```java
Thread t1 = new Thread(() -> {
    while (true) { /* background task */ }
});
t1.setDaemon(true);   // must set BEFORE start()
t1.start();
```

| Feature | User Thread | Daemon Thread |
|---|---|---|
| Default | ✅ All threads are user by default | Must call `setDaemon(true)` |
| JVM shutdown | JVM waits for all user threads to finish | JVM does NOT wait — kills daemon |
| Lifecycle | Lives independently | Dies when all user threads finish |
| Use cases | Business logic, request handling | GC, auto-save, logging |

### Real-World Examples

| Example | Why Daemon? |
|---|---|
| **Garbage Collector** | Runs in background, stops when JVM stops |
| **Auto-save** | Periodically saves — no point if app is closed |
| **Logging** | Background log writer — dies with app |

### Important Rules

```java
Thread t = new Thread(task);
t.setDaemon(true);    // ✅ must be before start()
t.start();

// t.setDaemon(true); // ❌ IllegalThreadStateException — already started!
```

| Rule | Detail |
|---|---|
| Set before `start()` | ❌ Cannot change after thread starts |
| `isDaemon()` | Check if thread is daemon |
| Child inherits | New thread inherits daemon status from parent |

## Summary

| Concept | Key Point |
|---|---|
| **stop()** | Deprecated — no lock release, causes deadlocks |
| **suspend()/resume()** | Deprecated — use `wait()`/`notify()` instead |
| **join()** | Current thread waits for target to finish |
| **Priority** | Hint only (1–10) — never rely on it |
| **Daemon thread** | Background thread — auto-killed when all user threads finish |
| **Daemon examples** | Garbage Collector, auto-save, logging |
