## Overview

Java has **two categories** of data types:

| Category | Also Known As | Examples |
|---|---|---|
| **Primitive** | Value types | `int`, `float`, `char`, `boolean`, `byte`, `short`, `long`, `double` |
| **Non-Primitive** | **Reference types** | `Class`, `String`, `Interface`, `Array` |

> **Key insight:** A reference variable does **not** store the actual data — it stores a **reference (address)** to the object in **heap memory**.

```
┌────────────────────────────┐
│     Reference Data Types   │
├────────────┬───────────────┤
│  Class     │  String       │
├────────────┼───────────────┤
│  Interface │  Array        │
└────────────┴───────────────┘
```

## 1. Class

A class is the most fundamental reference data type. It contains **properties** and **methods**.

```java
class Employee {
    int employeeId;

    int getEmployeeId() {
        return employeeId;
    }

    void setEmployeeId(int employeeId) {
        this.employeeId = employeeId;
    }
}
```

### Creating an Object (How References Work)

```java
Employee empObj = new Employee();
```

When `new Employee()` executes:

```
  Stack                          Heap Memory
┌──────────────┐          ┌──────────────────────┐
│  empObj ─────┼────────▶ │  Employee Object     │
│  (reference) │          │  employeeId = 0      │
└──────────────┘          └──────────────────────┘
```

- `new Employee()` allocates space in the **heap memory**
- `empObj` is a **reference variable** — it holds the **address** of the heap object, not the object itself
- Accessing `empObj.employeeId` follows the reference to the actual memory

### Multiple References to Same Object

```java
Employee obj1 = new Employee();
obj1.employeeId = 10;

Employee obj2 = obj1;    // obj2 now points to the SAME object
obj2.employeeId = 30;    // changes the SAME memory

System.out.println(obj1.employeeId);  // 30 (not 10!)
```

```
  Stack                          Heap Memory
┌──────────────┐          ┌──────────────────────┐
│  obj1 ───────┼────┐     │  Employee Object     │
└──────────────┘    ├───▶ │  employeeId = 30     │
┌──────────────┐    │     └──────────────────────┘
│  obj2 ───────┼────┘
└──────────────┘
```

> Both `obj1` and `obj2` point to the **same memory** — changing one reflects in the other.

## Pass by Value in Java

> **Java has NO pointers and NO pass-by-reference. Everything is passed by value.**

But because reference types hold an **address**, passing a reference variable copies the **address** (not the object), giving similar behavior to pass-by-reference in C++.

### Example — Reference Type Achieves Reflection

```java
void modify(Employee emp) {
    emp.employeeId = 20;       // modifies the ACTUAL object in heap
}

// In calling method:
Employee empObj = new Employee();
empObj.employeeId = 10;
modify(empObj);                // passes copy of the reference

System.out.println(empObj.employeeId);  // Output: 20 ✅
```

```
  Before modify()                After modify()
┌─────────┐    ┌──────────┐   ┌─────────┐    ┌──────────┐
│ empObj ──┼──▶ │ id = 10  │   │ empObj ──┼──▶ │ id = 20  │
└─────────┘    └──────────┘   └─────────┘    └──────────┘
                                    ▲
┌─────────┐                   ┌────┴────┐
│ emp ────┼──────────────────▶│(same obj)│
└─────────┘                   └─────────┘
```

### Example — Primitive Type Does NOT Reflect

```java
void modify(int x) {
    x = 20;                    // modifies only the LOCAL copy
}

int a = 10;
modify(a);
System.out.println(a);         // Output: 10 ❌ (unchanged)
```

| | Primitive | Reference |
|---|---|---|
| Stored in | **Stack** | Object in **Heap**, reference in Stack |
| Passed as | Copy of **value** | Copy of **reference (address)** |
| Changes in method reflect? | **No** | **Yes** (same object) |

## 2. String

Even though `String` is built into Java (like primitives), it is a **reference data type** because string variables hold a **reference** to an object in heap memory.

### String Constant Pool (SCP)

Inside the **heap memory**, Java maintains a special area called the **String Constant Pool**.

```
┌─────────────────────── Heap Memory ───────────────────────┐
│                                                           │
│   ┌──────────── String Constant Pool ──────────────┐      │
│   │                                                │      │
│   │   ┌─────────┐    ┌───────────────┐             │      │
│   │   │ "hello" │    │ "hello world" │   ...       │      │
│   │   └─────────┘    └───────────────┘             │      │
│   └────────────────────────────────────────────────┘      │
│                                                           │
│   ┌───────────────────────┐                               │
│   │ String Object (new)   │  ← created via new keyword   │
│   │ value = "hello"       │                               │
│   └───────────────────────┘                               │
└───────────────────────────────────────────────────────────┘
```

### String Literals

A **string literal** is a value directly assigned using double quotes:

```java
String s1 = "hello";   // string literal → goes into SCP
String s2 = "hello";   // reuses the SAME literal from SCP
```

```
  Stack                   String Constant Pool
┌────────┐              ┌─────────┐
│ s1 ────┼─────────┐    │ "hello" │
└────────┘         ├──▶ │         │
┌────────┐         │    └─────────┘
│ s2 ────┼─────────┘
└────────┘
```

> When creating a literal, Java first **checks the SCP** — if the literal already exists, it returns a reference to the existing one instead of creating a new copy. This saves memory.

### `new String()` — Creates Object Outside SCP

```java
String s3 = new String("hello");  // creates new object in heap (NOT in SCP)
```

```
  Stack              Heap Memory
┌────────┐         ┌──────────────────┐
│ s3 ────┼───────▶ │ String Object    │   ← separate from SCP
│        │         │ value = "hello"  │
└────────┘         └──────────────────┘
```

### `==` vs `.equals()` — Critical Difference

| Operator | Checks | Use Case |
|---|---|---|
| `==` | Whether both references point to the **same memory location** | Reference comparison |
| `.equals()` | Whether the **content/value** is the same | Content comparison |

```java
String s1 = "hello";
String s2 = "hello";
String s3 = new String("hello");

// == checks: same memory address?
s1 == s2       // ✅ true  (both point to same literal in SCP)
s1 == s3       // ❌ false (s3 points to a different heap object)

// .equals() checks: same content?
s1.equals(s2)  // ✅ true  (both contain "hello")
s1.equals(s3)  // ✅ true  (both contain "hello")
```

### Strings Are Immutable

Once a `String` object is created, its value **cannot be changed**. Any "modification" creates a **new** string.

```java
String s1 = "hello";       // "hello" literal created in SCP
s1 = "hello world";        // "hello world" is a NEW literal in SCP
                            // "hello" still exists — it is NOT modified
```

```
  Before                            After
                                   
  s1 ──▶ "hello"                   s1 ──▶ "hello world"  (new literal)
                                          "hello"          (still exists, unchanged)
```

> **Why immutable?** Because multiple references (e.g., `s1`, `s2`) can point to the same literal in SCP. If one reference could modify it, all others would be affected — this would break the whole pool mechanism.

## 3. Interface

An interface is a reference data type, very similar to a class. It defines a **blueprint** of methods without implementation.

```java
interface Person {
    String profession();
}

class Teacher implements Person {
    @Override
    public String profession() {
        return "Teaching";
    }
}

class Engineer implements Person {
    @Override
    public String profession() {
        return "Software Engineer";
    }
}
```

### Creating Objects with Interface References

```java
// ✅ Valid — parent interface holds reference to child object
Person p1 = new Engineer();
Person p2 = new Teacher();

// ✅ Valid — same type reference
Teacher t1 = new Teacher();
Engineer e1 = new Engineer();

// ❌ INVALID — cannot instantiate an interface!
Person p3 = new Person();   // Compilation Error!
```

> **Why can't you instantiate an interface?**  
> An interface only defines a blueprint (abstract methods). It has **no implementation**. Only the child classes that `implement` the interface have actual code, so only they can be instantiated.

```
  Stack                          Heap Memory
┌──────────────┐          ┌──────────────────────┐
│ p1 (Person) ─┼────────▶ │  Engineer Object     │
└──────────────┘          └──────────────────────┘

┌──────────────┐          ┌──────────────────────┐
│ p2 (Person) ─┼────────▶ │  Teacher Object      │
└──────────────┘          └──────────────────────┘
```

> A **parent reference** can hold a **child object** because the reference is just an address — the actual object type in heap determines the behavior (polymorphism).

## 4. Array

An array is a **fixed-size, contiguous block of memory** that stores elements of the **same data type**. It is a reference type — the variable holds a reference to the memory block in the heap.

### 1D Array

```java
// Method 1: Declare size, then assign values
int[] a = new int[5];
a[0] = 10;
a[3] = 40;

// Method 2: Declare and initialize together
int[] arr = {30, 20, 10, 40, 50};
```

> Both `int[] a` and `int a[]` are valid syntax.

```
  Stack                    Heap Memory
┌────────┐          ┌─────┬─────┬─────┬─────┬─────┐
│ arr ───┼────────▶ │  30 │  20 │  10 │  40 │  50 │
└────────┘          ├─────┼─────┼─────┼─────┼─────┤
                    │ [0] │ [1] │ [2] │ [3] │ [4] │
                    └─────┴─────┴─────┴─────┴─────┘
```

### 2D Array

```java
// Method 1: Declare size, then assign values
int[][] a = new int[5][4];   // 5 rows × 4 columns
a[2][2] = 20;
a[1][3] = 30;

// Method 2: Inline initialization
int[][] b = {
    {1, 5, 7},
    {4, 2, 3}
};  // 2 rows × 3 columns
```

```
  2D Array — int[5][4] (default values = 0)

       Col 0   Col 1   Col 2   Col 3
      ┌───────┬───────┬───────┬───────┐
Row 0 │   0   │   0   │   0   │   0   │
      ├───────┼───────┼───────┼───────┤
Row 1 │   0   │   0   │   0   │  30   │  ← a[1][3] = 30
      ├───────┼───────┼───────┼───────┤
Row 2 │   0   │   0   │  20   │   0   │  ← a[2][2] = 20
      ├───────┼───────┼───────┼───────┤
Row 3 │   0   │   0   │   0   │   0   │
      ├───────┼───────┼───────┼───────┤
Row 4 │   0   │   0   │   0   │   0   │
      └───────┴───────┴───────┴───────┘
```

Accessing: `b[1][2]` → Row 1, Col 2 → value `3`

> Arrays are reference types because the variable (`a`, `arr`) holds a **reference to the contiguous memory block** in the heap. Any changes via the reference modify the actual data.

## Wrapper Classes

For each of the **8 primitive types**, Java provides a corresponding **wrapper class** (reference type):

| Primitive | Wrapper Class | Example |
|---|---|---|
| `int` | `Integer` | `Integer x = 10;` |
| `char` | `Character` | `Character c = 'A';` |
| `short` | `Short` | `Short s = 5;` |
| `byte` | `Byte` | `Byte b = 1;` |
| `long` | `Long` | `Long l = 100L;` |
| `float` | `Float` | `Float f = 1.5f;` |
| `double` | `Double` | `Double d = 3.14;` |
| `boolean` | `Boolean` | `Boolean flag = true;` |

### Why Do We Need Wrapper Classes?

**1. Reference capability** — Wrappers are objects stored in the heap. Passing them to methods allows changes to reflect back (same as any reference type).

**2. Collections require objects** — Java Collections (`ArrayList`, `HashMap`, etc.) work **only with objects**, not primitives.

```java
// ❌ INVALID — Collections don't accept primitives
ArrayList<int> list = new ArrayList<>();

// ✅ VALID — Use wrapper class
ArrayList<Integer> list = new ArrayList<>();
```

### Autoboxing & Unboxing

| Term | Direction | Example |
|---|---|---|
| **Autoboxing** | Primitive → Wrapper | `Integer a = 10;` (auto-converts `int` → `Integer`) |
| **Unboxing** | Wrapper → Primitive | `int b = a;` (auto-converts `Integer` → `int`) |

```java
// Autoboxing: primitive → wrapper
int a = 10;
Integer a1 = a;        // int → Integer (autoboxing)

// Unboxing: wrapper → primitive
Integer x = 20;
int x1 = x;            // Integer → int (unboxing)
```

```
  Autoboxing                          Unboxing
  ──────────                          ────────
  int (stack)                         Integer (heap)
      │                                   │
      ▼                                   ▼
  Integer (heap)                      int (stack)
  
  Primitive ──▶ Wrapper               Wrapper ──▶ Primitive
```

## Constant Variables

A **constant variable** in Java is declared using `static final`.

```java
class Employee {
    public static final int EMPLOYEE_ID = 10;
}
```

### Breakdown

| Keyword | Meaning |
|---|---|
| `static` | Only **one copy** exists, shared across all objects (belongs to the class, not instances) |
| `final` | The value **cannot be changed** after initialization |
| `static final` | One copy + immutable = **Constant** |

```java
Employee obj1 = new Employee();
Employee obj2 = new Employee();

// Both share the same EMPLOYEE_ID = 10
// Any attempt to change it causes a COMPILATION ERROR:
Employee.EMPLOYEE_ID = 20;   // ❌ Error: cannot assign to final variable
```

### Naming Convention

Constants use **UPPER_SNAKE_CASE**:

```java
public static final int MAX_RETRY_COUNT = 3;
public static final String DEFAULT_NAME = "Unknown";
public static final double PI = 3.14159;
```

## Summary

```
┌──────────────────────────────────────────────────────┐
│              Java Data Types                         │
├──────────────────────┬───────────────────────────────┤
│   Primitive          │   Reference (Non-Primitive)   │
│   ─────────          │   ────────────────────────    │
│   int, float,        │   Class, String,              │
│   char, boolean,     │   Interface, Array            │
│   byte, short,       │                               │
│   long, double       │   + Wrapper Classes           │
│                      │   (Integer, Character, etc.)  │
├──────────────────────┼───────────────────────────────┤
│   Stored in: Stack   │   Object in: Heap             │
│   Holds: actual value│   Holds: reference (address)  │
│   Pass: copy of value│   Pass: copy of reference     │
└──────────────────────┴───────────────────────────────┘
```

| Concept | Key Takeaway |
|---|---|
| **Reference** | Variable stores address of object in heap, not the data itself |
| **String Pool** | Literals are reused from SCP; `new String()` creates separate heap object |
| **`==` vs `.equals()`** | `==` compares addresses; `.equals()` compares content |
| **Immutability** | Strings cannot be modified — reassignment creates a new literal |
| **Interface** | Cannot be instantiated; parent ref can hold child object |
| **Array** | Fixed-size contiguous memory; reference to heap block |
| **Wrapper Class** | Object equivalent of primitives; needed for collections |
| **Autoboxing** | Primitive → Wrapper (automatic) |
| **Unboxing** | Wrapper → Primitive (automatic) |
| **Constant** | `static final` — one copy, immutable, UPPER_SNAKE_CASE |
| **Pass by Value** | Java always passes by value; reference types pass copy of address |
