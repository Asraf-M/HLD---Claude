# System Design: Food Delivery Backend (Beginner-Friendly Guide)

---

## What Are We Building?

Imagine Uber Eats or DoorDash — a food delivery platform where:
- Customers search nearby restaurants and browse menus
- Place orders with customizations (no onions, extra sauce)
- Track their order in real-time (cooking, on the way)
- Delivery drivers pick up and deliver

The interesting engineering challenges hidden in this flow:
- **Real-time updates** — customer sees driver location live; even one minute lag feels broken
- **Geospatial queries** — "Find restaurants within 5km of my location" must be fast (<100ms)
- **Driver assignment** — assign 1000 orders to drivers optimally (minimize distance, balance workload)
- **Concurrency** — what if 2 customers order the last portion of food simultaneously?
- **Surge pricing** — during lunch rush, how much should delivery cost to balance supply/demand?
- **Reliable location tracking** — driver phones lose signal; GPS is sometimes inaccurate

## Step 3: Database Choice

| Consideration | SQL | NoSQL |
|---------------|-----|-------|
| Restaurant/Menu data | ✅ Relational | ❌ Unnatural |
| Orders | ✅ ACID, consistency | ❌ Eventual consistency risky |
| Real-time queries | ✅ Good | ✅ Acceptable |
| Geospatial queries | ✅ Good (PostGIS) | ✅ Good (MongoDB geo) |
| Write pattern | 60 writes/sec | Better for extreme writes |

**Decision: Hybrid approach**
- **PostgreSQL** (with PostGIS extension) for restaurants, orders, users → geospatial + ACID
- **Redis** cache for search results (expensive computation)
- **Kafka** stream for real-time driver locations → fast writes

## Step 5: Restaurant Search & Geospatial Queries

### Geohashing vs PostGIS

```
GEOHASHING approach:
- Encode location as string: "ezs42"
- Find restaurants with same prefix
- Limited precision

POSTGIS approach (better):
- Store location as POINT(latitude, longitude)
- Index using GIST index (R-tree spatial index)
- Query: distance <= 5km

Query example:
SELECT id, name, rating
FROM restaurants
WHERE ST_DWithin(
  location,
  ST_MakePoint(37.7749, -122.4194),
  5000  -- 5km in meters
)
ORDER BY ST_Distance(location, ST_MakePoint(...))
LIMIT 20;
```

### Caching Popular Searches

```
User searches: "Pizza in San Francisco"
  ↓
Check cache: "location:SFO:cuisine:pizza"
  ↓ Cache miss (first time)
Query database (expensive)
  ↓
Store in Redis with 1 hour TTL
  ↓
Next user searches same thing
  ↓ Cache hit
Return instantly from Redis
```

## Step 7: Driver Assignment Algorithm

### Real-Time Matching

```
Order ready for pickup (food prepared):

1. Find nearby drivers:
   SELECT * FROM drivers
   WHERE status = 'AVAILABLE'
   AND ST_DWithin(location, restaurant_location, 2000)
   AND rating > 3.5

2. Score drivers:
   score = 100
   score -= distance_km * 5  // Closer drivers preferred
   score -= (5 - rating) * 10  // Better rated drivers preferred
   if (currently_delivering) score -= 20
   
   Ranking:
   Driver A: distance 0.5km, rating 4.8 → score 93
   Driver B: distance 1.2km, rating 4.2 → score 80
   Driver C: distance 0.3km, rating 3.8, busy → score 55

3. Send request to top 3 drivers:
   - Driver A gets request
   - 20 second timeout
   - If A accepts: assign order
   - If timeout: try B and C

4. If no acceptance:
   - Expand radius (try 3km drivers)
   - Increase delivery fee (surge pricing)
   - Attract more drivers
```

### Bulk Assignment (Batch Optimization)

```
Peak hours (12-1pm):
- 1000 orders ready simultaneously
- 500 available drivers

Problem: Assign optimally
- Minimize total distance (cost)
- Balance driver workload
- Meet customer time expectations

Solution: Hungarian Algorithm
- Input: 1000 orders × 500 drivers (cost matrix)
- Output: Optimal 1:1 assignments
- Computation: ~5 seconds

Benefit: Save 10-15% on delivery distance vs greedy assignment
```

## Step 9: Handling Failures

### Network Issues

```
Driver loses signal mid-delivery:
1. Location updates stop
2. Customer sees "Last known location"
3. Driver regains signal → location resumes

Restaurant system crashes:
1. Orders can't be confirmed
2. Queue new orders in Kafka
3. When restaurant back up, process queued orders
4. Notify customers: "Restaurant delayed, new ETA"
```

### Cancellation Policy

```
Customer cancels < 5 min after order:
- Full refund
- No cancellation fee

Customer cancels 5-15 min after order:
- Refund minus $2 platform fee
- Restaurant keeps prep costs

Customer cancels > 15 min after order:
- Refund minus delivery fee
- Driver gets compensation ($3-5)

Incentivizes early cancellation (less waste).
```

## Step 11: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| WebSocket + long polling fallback | Real-time updates are critical; fallback for reliability | WebSocket complexity; consume server connections |
| Geospatial database (PostGIS) | Fast <100ms searches; exact distance queries | Requires PostgreSQL + extension knowledge |
| Batch driver assignment (every 2min) | Optimize globally (minimize total distance) | 2min delay (drivers know assignments late); complex algorithms |
| Surge pricing algorithm | Auto-adjust to balance supply/demand | Risk: price too high → customers angry; too low → drivers unavailable |
| Separate order confirmation from driver assignment | Fast order confirmation (seconds); driver assign later | Customer sees "Finding driver" state; wait for actual assignment |

## Interview Cheat Sheet Q&A

**Q: What if two customers order the last portion of food at the same time?**  
A: Race condition. Solution: database lock or optimistic locking. When ordering, check: "is this item still available?" If yes, decrement inventory and confirm order atomically. Second customer loses race: "Sorry, out of stock. Try similar item?"

**Q: Why WebSocket instead of just polling?**  
A: Polling = client asks "Is order ready?" every 5 seconds. Inefficient. WebSocket = server pushes "Order ready!" immediately when status changes. One order × 1000 customers × 4 polls/min = 4000 requests/min. WebSocket = 1 broadcast when status changes.

**Q: What if driver app crashes and location stops updating?**  
A: Last known location shown to customer (30 seconds old). Customer sees "Driver location unavailable." When driver restarts app, reconnects, sends latest location. Customer sees updated location. Temporary stall, not a disaster.

**Q: Can we predict delivery time better than "25-45 minutes"?**  
A: Yes, ML model: train on historical data (distance, restaurant prep time, traffic, time of day, weather). Predict actual time. But ML has error ~5 minutes. Show range: "25-30 minutes" (more accurate than 45).

**Q: What if restaurant accepts order but kitchen is too busy to prepare?**  
A: Happens. Options: (1) Restaurant marks order as delayed, notifies customer +15min ETA; (2) Support team calls customer, offers refund + $5 credit if cancel; (3) Customer sees delay, chooses to wait or cancel. No perfect solution.

**Q: Why not use location updates every 1 second instead of every 10 seconds?**  
A: Battery drain on driver phone (GPS is power-hungry). Network overhead (1M drivers × 1 update/sec = network saturated). 10 seconds is good balance: smooth updates (no jumping), battery friendly, network manageable.

## Step 6: Geospatial Queries

### Finding Nearby Restaurants

```
Using Geohashing:
- Encode user location: "ezs42"
- Find restaurants with same prefix
- Also check adjacent cells

Using R-tree:
- Spatial indexing structure
- Find restaurants in bounding box
- Fast range queries

Query:
SELECT * FROM restaurants
WHERE latitude BETWEEN lat-0.1 AND lat+0.1
  AND longitude BETWEEN lng-0.1 AND lng+0.1
  AND ST_Distance(location, point(lat, lng)) <= 10
ORDER BY distance ASC
LIMIT 50
```

### Delivery Area Management

```
Restaurant service area defined as polygon:
- Vertices: (lat1, lng1), (lat2, lng2), ...
- When user places order, check if delivery location inside polygon
- If yes: Can deliver
- If no: Cannot deliver (show "out of range")

Elasticsearch geo-polygon query:
GET /restaurants/_search
{
  "query": {
    "bool": {
      "filter": {
        "geo_polygon": {
          "location": {
            "points": [[lat1, lng1], [lat2, lng2], ...]
          }
        }
      }
    }
  }
}
```

## Interview Cheat Sheet Q&A

**Q: Why can't restaurant-to-driver assignment happen automatically?**  
A: It can, but restaurants prefer control. "I don't want this driver; he's been slow." Allowing opt-out makes restaurants happy. In reality, most accept automatically, but the option exists.

**Q: What if driver picks up order but never goes to customer (drives away)?**  
A: After 10 minutes without reaching customer, customer app shows warning: "Driver hasn't started delivery, contact support." Support team calls driver: "Where are you?" If unreachable, reassign to another driver, refund customer delivery fee, mark driver as unreliable.

**Q: How do we handle orders from multiple restaurants in one trip (batch delivery)?**  
A: Algorithmically determine: is it on the driver's route? Will it add < 5 minutes? If yes, add it to the trip. Show customer: "Your order will arrive with 1 other order." Saves money, but customers might wait longer (acceptable as long as < 30 min total).

**Q: What if a restaurant kitchen is behind and can't prepare 100 orders?**  
A: Queue orders. Show customer: "Estimated prep time 45 minutes." If too long, customer cancels (refunded). Restaurant sees backlog in their app. They hire temporary staff or issue refund to clear queue. At 8pm on Saturday, this happens. It's normal.

**Q: Why not let customers tip anonymously?**  
A: You could, but you don't! Show driver the tip before they accept. High tip = driver accepts faster, better service. No tip = driver delays or declines. This is behavioral economics: incentivize good service upfront.

**Q: What if a driver marks order as "delivered" but customer didn't receive it (fraud)?**  
A: Customer can immediately dispute in app with photo/proof. Support investigates: check GPS history of driver (was at location?), review previous disputes (is this driver a repeat offender?). If driver at fault, refund customer, deactivate driver. If customer lying, no refund.
