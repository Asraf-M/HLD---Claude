### Key Differences

| Feature | Stack | Heap |
|---|---|---|
| Stores | Primitives, references, method frames | Objects, string pool |
| Scope | Variables visible only within their block | Objects accessible via references |
| Lifecycle | Deleted when scope ends (LIFO) | Deleted by Garbage Collector |
| Thread | Each thread has its own | Shared across all threads |
| Size | Smaller | Larger |
| Error | `StackOverflowError` | `OutOfMemoryError` |

## Memory Allocation Example

```java
public class MemoryManagement {
    public static void main(String[] args) {
        int primitiveVar = 10;                    // в‘  primitive
        Person personObj = new Person();           // в‘Ў object
        String strLiteral = "24";                  // в‘ў string literal
        MemoryManagement memObj = new MemoryManagement(); // в‘Ј object
        memObj.test(personObj);                    // в‘Ө method call
    }

    void test(Person personObj) {
        Person personObj2 = personObj;             // в‘Ҙ same reference
        String strLiteral2 = "24";                 // в‘Ұ reuses from pool
        String strObj = new String("24");          // в‘§ new heap object
    }
}
```

### Cleanup Order (LIFO)

1. `test()` method ends вҶ’ its frame is popped: `strObj`, `strLiteral2`, `personObj2`, `personObj` removed from stack
2. `main()` method ends вҶ’ its frame is popped: `memObj`, `strLiteral`, `personObj`, `primitiveVar` removed
3. Stack is now empty
4. **Heap objects with no references** вҶ’ eligible for Garbage Collection

## Garbage Collector

The GC **automatically** removes unreferenced objects from the heap.

```
  Stack references removed  вҶ’  Heap objects become "unreferenced"
                                     вҶ“
                              Garbage Collector deletes them
```

### Key Points

| Point | Detail |
|---|---|
| **Purpose** | Free heap memory by removing unreferenced objects |
| **Triggered by** | JVM decides when to run (automatic) |
| **`System.gc()`** | Suggests GC run вҖ” **no guarantee** JVM will comply |
| **Also called** | Automatic Memory Management |

> JVM monitors heap usage. If heap is filling fast вҶ’ GC runs more frequently. If plenty of space вҶ’ GC runs less often.

## Types of References

### 1. Strong Reference (default)

```java
Person pObj = new Person();  // strong reference
```

GC **will NOT** delete the object as long as a strong reference exists.

### 2. Weak Reference

```java
WeakReference<Person> weakRef = new WeakReference<>(new Person());
```

GC **will delete** the object as soon as it runs, even if a weak reference exists. Accessing it after GC returns `null`.

### 3. Soft Reference

```java
SoftReference<Person> softRef = new SoftReference<>(new Person());
```

GC **may delete** if memory is critically low. Otherwise keeps it alive. More lenient than weak.

| Reference | GC Behavior | Use Case |
|---|---|---|
| **Strong** | Never deleted while referenced | Normal usage |
| **Weak** | Deleted on next GC run | Caches, metadata |
| **Soft** | Deleted only when memory is urgent | Memory-sensitive caches |

### How References Change

```java
// Making an object eligible for GC:
pObj = null;                    // в‘  remove reference вҶ’ eligible

// Reassigning reference:
Person pObj2 = new Person();    // new object
pObj = pObj2;                   // old object loses reference вҶ’ eligible
```

### Object Lifecycle in Heap

```
Step 1: New objects вҶ’ Eden
        obj1, obj2, obj3, obj4, obj5 вҶ’ Eden

Step 2: Minor GC runs (Mark & Sweep)
        вҖў Mark unreferenced objects (obj2, obj5)
        вҖў Sweep: delete marked objects
        вҖў Move survivors to S0, set age = 1
        
        Eden: empty  |  S0: obj1(1), obj3(1), obj4(1)  |  S1: empty

Step 3: More objects created вҶ’ Eden
        obj6, obj7 вҶ’ Eden

Step 4: Minor GC runs again
        вҖў Mark: obj7, obj4 unreferenced
        вҖў Sweep: delete them
        вҖў Move survivors to S1 (alternate!), increment age
        
        Eden: empty  |  S0: empty  |  S1: obj6(1), obj1(2), obj3(2)

Step 5: When age reaches threshold (e.g., 3)
        вҶ’ Object gets PROMOTED to Old Generation

        Old Gen: obj1 (promoted вҖ” survived long enough)
```

### Minor GC vs Major GC

| | Minor GC | Major GC |
|---|---|---|
| **Where** | Young Generation | Old Generation |
| **Speed** | Fast | Slow |
| **Frequency** | Very frequent | Infrequent |
| **Objects** | Short-lived, rapidly created/deleted | Long-lived, many references |

### Metaspace (replaces PermGen)

| Feature | PermGen (pre-Java 8) | Metaspace (Java 8+) |
|---|---|---|
| Part of heap? | Yes | **No** (non-heap) |
| Expandable? | No вҶ’ `OutOfMemoryError` | **Yes** вҖ” grows as needed |
| Stores | Class metadata, constants | Same data |

## GC Algorithms & Types

### Mark & Sweep

1. **Mark** вҖ” identify all objects with no references
2. **Sweep** вҖ” delete marked objects, free memory

### Mark & Sweep with Compaction

Same as above, plus:
3. **Compact** вҖ” move surviving objects together to eliminate memory fragmentation

### GC Implementations

| GC Type | Threads | App Pause | Compaction | Notes |
|---|---|---|---|---|
| **Serial GC** | 1 GC thread | Long pause | Yes | Simplest, slowest |
| **Parallel GC** | Multiple GC threads | Shorter pause | Yes | Java 8 default |
| **CMS** (Concurrent Mark & Sweep) | Concurrent with app | Minimal pause | No | App threads mostly don't stop |
| **G1** (Garbage First) | Concurrent + parallel | Very low pause | Yes | Modern default (Java 9+) |

> **Stop-the-World:** When GC runs, all application threads **pause**. Less pause time = better throughput and lower latency.

## Summary

| Concept | Key Takeaway |
|---|---|
| **Stack** | Per-thread, stores primitives + references, LIFO cleanup |
| **Heap** | Shared, stores objects, cleaned by GC |
| **Strong Ref** | GC won't delete while reference exists |
| **Weak Ref** | GC deletes on next run |
| **Soft Ref** | GC deletes only under memory pressure |
| **Eden** | Where new objects are born |
| **Survivor (S0/S1)** | Alternating spaces for surviving objects |
| **Old Gen** | Promoted long-lived objects |
| **Metaspace** | Non-heap, expandable, stores class metadata |
| **Minor GC** | Fast, frequent, Young Gen |
| **Major GC** | Slow, infrequent, Old Gen |
| **Mark & Sweep** | Identify + delete unreferenced objects |
| **Compaction** | Defragment memory after sweep |
