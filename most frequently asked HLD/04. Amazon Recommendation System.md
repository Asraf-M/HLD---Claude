# System Design: Recommendation Engine (Beginner-Friendly Guide)

---

## What Are We Building?

Think of Netflix or Amazon's "Recommended For You" section — a recommendation engine that:
- Suggests products each user might want to buy
- Personalizes experience based on browsing, clicks, and purchase history
- Increases conversion rate and average order value
- Scales to millions of users and products simultaneously

The interesting engineering challenges hidden in this simple flow:
- **Cold-start problem** — new users have no history; how do we recommend anything?
- **Scale** — millions of users × millions of products = 1 trillion possible pairs to score
- **Latency** — recommendations must appear within 200ms (users won't wait)
- **Staleness** — old recommendations are useless; freshness is critical
- **Diversity** — don't recommend 10 identical items; vary the results
- **Business goals** — maximize sales, but also user satisfaction (not just profit-chasing)
- **Model updates** — retrain models frequently to catch trends and seasonality

## Step 3: Recommendation Algorithms

### Algorithm 1: Collaborative Filtering
**Idea:** "If you and I like similar products, recommend to you what I bought"

**User-User Collaborative Filtering:**
```
       Prod1  Prod2  Prod3  Prod4
User A   ★★★★★   ★★★★    ?      ★★
User B   ★★★★★   ★★★★   ★★★★★   ★★  (very similar to User A)
User C   ★★      ★★     ★★★★    ★★★★
```

User A and User B are similar (both rated Prod1, Prod2 high). 
User B liked Prod3 highly, so recommend Prod3 to User A.

**Pros:** No product metadata needed; finds hidden gems
**Cons:** Cold-start (new users have no history); sparse matrices (most users rate few products)

### Algorithm 2: Content-Based Filtering
**Idea:** "If you like Action movies, recommend other Action movies"

**Steps:**
1. Extract product features: category, price, brand, rating, description keywords
2. Build user preference profile: "User likes expensive action movies with high ratings"
3. Find products matching user's preferences
4. Recommend

**Pros:** Works for new products (no rating history needed); handles new users (demographics)
**Cons:** Requires product metadata; limited serendipity (recommends similar items only)

### Algorithm 3: Hybrid (BEST)
```
Final Score = 0.5 × Collaborative + 
              0.3 × ContentBased + 
              0.1 × Popularity + 
              0.1 × BusinessMetrics
```

Combines strengths: Collaborative finds hidden gems, content adds diversity, popularity provides safety net.

## Step 4: Matrix Factorization at Scale

### The Problem
User-Product interaction matrix is HUGE: 500M users × 500M products = 250 trillion cells!
Can't store 250 trillion numbers in memory.

### Solution: Matrix Factorization
Instead of storing the full matrix, factorize it:

```
User-Product Matrix (500M × 500M)
              ≈
User Embeddings (500M × 128) × Product Embeddings (500M × 128)

Size reduced from 250TB → 128 GB (2000x smaller!)
```

**How it works:**
```
Each user represented as 128-dimensional vector (embedding)
Each product represented as 128-dimensional vector
Recommendation score = dot_product(user_embedding, product_embedding)

Find top-K products with highest scores
```

### Algorithms
- **ALS (Alternating Least Squares):** Fast, works offline
- **SGD (Stochastic Gradient Descent):** Incremental, handles streaming
- **Deep Learning:** Neural networks learn better embeddings

## Step 5: Deep Learning: Two-Tower Model

### Architecture

```
USER TOWER                              PRODUCT TOWER
┌────────────────┐                    ┌─────────────────┐
│ User ID        │                    │ Product ID      │
│ Browse history │                    │ Category        │
│ Purchase hist. │                    │ Price           │
│ Ratings        │                    │ Description     │
│ Demographics   │                    │ Images (CNN)    │
└────────┬───────┘                    └────────┬────────┘
         │                                     │
      Embedding Layer                    Embedding Layer
    (user → vector)                      (product → vector)
         │                                     │
      Dense Layers                       Dense Layers
    (transform)                          (transform)
         │                                     │
    Final Embedding                    Final Embedding
      (d = 64)                            (d = 64)
         └──────────────┬──────────────────┘
                        │
                   Dot Product
                        │
                  Relevance Score
                   (0 to 1)
```

### Why Two-Tower?

**Advantages:**
- Separates user and product towers → can update each independently
- Fast inference: pre-compute all product embeddings, then just dot products
- Supports content features: description, images
- Scalable: easy to add new features

## Step 6: Real-Time Scoring Pipeline

### Challenge: Speed
1M recommendation requests/second × 1000 products to score each = 1 billion dot products/second

### Solution: Caching + Approximation

**Approach 1: Pre-compute & Cache (Fast)**
```
Offline (daily):
1. Compute recommendations for top 10M users
2. Store in Redis cache
3. Return from cache at runtime (< 1ms)

Online (new users):
1. Fetch user embedding
2. Fetch top 100 product embeddings
3. Score via dot product
4. Return top 10
```

**Approach 2: Approximate Nearest Neighbors (Fast + Scalable)**
```
Exact:       dot_product(user_vec, product_vec) — accurate but slow
Approximate: Use FAISS or Annoy library
             → 10x faster, 99% accuracy
```

### Model Serving

```
Request → Load Balancer
           ↓
      TensorFlow Serving Cluster
      (10 GPU servers)
           ↓
      Batch predictions (GPU optimized)
           ↓
      Results → Redis Cache
           ↓
      Client
```

## Step 8: Addressing Cold-Start Problem

### New User (No History)

**Solutions (in priority order):**
1. **Popular/Trending items** — everyone likes bestsellers
2. **Demographic-based** — users in same age/location get similar recs
3. **Survey** — ask "What are you interested in?"
4. **Content-based** — if user visits category X, recommend from X
5. **Exploration** — show random items, learn from reactions

### New Product (No Interactions)

**Solutions:**
1. **Content-based matching** — similar products to ones users like
2. **Boost visibility** — temporary recommendation lift
3. **Monitor via A/B test** — does this product sell?
4. **Cross-category promotion** — if user likes A, show similar B

### New User + New Product

**Solution:**
- Content-based: Match product features to user demographics/category preference

## Step 9: Ranking & Diversity

### Problem
Top-K recommendations might be 10 identical products (all iPhones, all from Amazon brand, all $900).

### Solution: Re-ranking

After generating top 100 candidates:
```
1. Remove exact duplicates
2. Enforce category diversity (max 2 per category)
3. Price diversity (mix $50, $500, $5000)
4. Brand diversity (don't recommend 5 Samsung products)
5. Apply business rules (boost high-margin items)
6. Add serendipity (10% random exploration)
```

### Scoring Function

```
Final Score = 
  0.6 × RelevanceScore +
  0.2 × DiversityScore +
  0.1 × BusinessMetrics +
  0.1 × FreshnessBoost
```

## Step 10: Evaluation Metrics

### Offline Metrics (Model Evaluation)

```
Precision@K:   What % of recommended items were actually relevant?
Recall@K:      What % of all relevant items appeared in recommendations?
NDCG@K:        Ranking quality (ranking matters; #1 spot better than #10)
MAP@K:         Mean average precision across users
Diversity:     How varied are recommendations?
```

### Online Metrics (A/B Testing - Real User Impact)

```
Click-Through Rate (CTR): % users who click recommended item
Conversion Rate (CVR):    % users who purchase
Average Order Value (AOV):How much users spend
Add-to-Cart Rate:         Items added to cart
Repeat Purchase Rate:     Users buying again
User Engagement Time:     Time spent browsing
```

### Running A/B Test

```
Split users:
- Control (50%):   Old recommendations
- Treatment (50%): New recommendations

Monitor metrics for 2 weeks:
- If CTR improvement statistically significant? → Deploy
- If worse? → Rollback
- Undecided? → Run longer
```

## Step 11: Model Retraining & Updates

### Scheduling

**Collaborative Filtering:**
```
Frequency: Daily (overnight, 2am-6am)
Process:
1. Collect past 30 days events
2. Train on Spark cluster (parallel, ~1 hour)
3. Evaluate on validation set
4. Compare to production model
5. Deploy if better; rollback if worse
6. Keep 3 previous versions for fallback
```

**Content-Based Features:**
```
Frequency: Real-time when product metadata changes
Updates: Product price change → immediately refresh
```

**Deep Learning Models:**
```
Frequency: Weekly (expensive to train)
Fine-tune: Daily on recent data (cheap, incremental updates)
```

## Step 12: Caching & Scalability

### Redis Cache Layers

```
Layer 1: User Recommendations
- Key: recs:{userId}
- Value: [productId1, productId2, ...]
- TTL: 24 hours
- Size: ~100 GB (10M users × 100 bytes)

Layer 2: Popular Items
- Key: popular:{category}
- Value: [productId1, productId2, ...]
- TTL: 6 hours

Layer 3: User Embeddings
- Key: user_emb:{userId}
- Value: 128-dim vector
- TTL: 7 days
```

### Fallback Strategy

```
If recommendation service is down:
1. Try Redis cache (might be old, but still usable)
2. Fall back to trending/popular products
3. Fall back to category bestsellers
4. Worst case: Return generic "editors' choice" list
```

## Step 13: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Hybrid (collab + content) | Combines pattern discovery with explainability | More complex; harder to debug |
| Daily batch training | Scalable to 500M users; risk-managed | 24hr staleness; misses hourly trends |
| Cache recommendations | Sub-100ms response; no model server latency | Requires pre-computation; less personalized |
| Two-tower model | Separates user/item; fast inference | More infrastructure; harder to interpret |
| A/B testing | Measure real user impact, not just metrics | Slower iteration; some users see worse recs |

## Interview Cheat Sheet

**Q: Why not just show most-bought products to everyone?**  
A: Most-bought items are already discoverable. Recommendations shine by showing personalized items. User A (loves gaming) and User B (loves cooking) should get different recommendations even if they buy from same site.

**Q: Can we recommend in real-time (not cache)?**  
A: Theoretically yes, but slow. Computing user embedding + scoring 500M products = 500M dot products. With GPU, still 100+ ms. Caching pre-computed recs + serving from Redis = <10ms. Speed matters for UX.

**Q: What if collaborative filtering is wrong?**  
A: That's why hybrid. Collaborative might recommend random item (noise in data). Content-based checks: "Does this match user's preferences?" Diversity rules ensure we don't recommend 10 identical items.

**Q: How to handle 10B events/day?**  
A: Kafka for event streaming, Spark for aggregation. Don't store raw events; aggregate (e.g., daily click count per product). Weekly full retraining with aggregated data.

**Q: How to detect recommendation bias (always recommending expensive items)?**  
A: Track metrics by user segment (low-income vs high-income). If low-income users get worse recommendations, add fairness constraints in model or post-processing. Test regularly.

## Full Flow (Start to End)

### Happy Path
1. Client request enters API Gateway and is authenticated/authorized.
2. Orchestrator service validates input and routing context.
3. Core service executes primary business logic and required checks.
4. Read path uses cache first; fallback goes to durable database/store.
5. Write path updates fast layer first (where applicable) and publishes async events.
6. Downstream consumers persist durable state and trigger secondary effects.
7. Response is returned to client with final status and metadata.

### Failure and Retry Paths
1. Cache miss: read from durable store, then repopulate cache.
2. Dependency timeout: retry with backoff or circuit-breaker fallback.
3. Async event failure: retry queue and dead-letter queue (DLQ) handling.
4. Duplicate request: idempotency key returns prior successful outcome.
5. Concurrent updates: version/lock conflict triggers re-read and safe retry.

### End-State Guarantees
- Low-latency user operations on the hot path.
- Durable correctness in the source-of-truth datastore.
- Eventual consistency for non-critical async side effects.
- Strict correctness at critical boundaries (commit/payment/finalization).

- ✅ Continuous A/B testing
