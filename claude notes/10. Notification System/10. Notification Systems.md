# System Design: Notification System (Beginner-Friendly Guide)

---

## What is a Notification System?

Think of it like a **postal service inside software**. When something important happens — your Uber arrives, your Amazon package ships, your bank detects a transaction — the system needs to *tell you*. That's the notification system's job.

**Real-world examples:**
- WhatsApp message ping on your phone
- "Your order has shipped!" email from Amazon
- OTP SMS from your bank
- YouTube "new video" push notification

---

## Step 1: Understand What We're Building

### Requirements

| Type | Volume |
|------|--------|
| Push Notifications (iOS/Android) | 10 million/day |
| SMS | 1 million/day |
| Emails | 5 million/day |

**Total: ~16 million notifications per day.**

> **Why does scale matter?**  
> If you're sending 10 million push notifications per day, that's ~116 per second on average. During peak hours (like a big sale or news event), it could spike to 500+/sec. A simple `for loop` calling an API won't cut it — you need an architecture that *distributes* the load.

### Supported Platforms
- iOS devices
- Android devices
- Desktop/Web (email)

### Other requirements:
- Users should be able to **opt out** of notifications
- System should be **soft real-time** (best effort, small delay is okay)
- Notifications can be triggered by **client apps or scheduled server jobs** (e.g., "send this promo at 9am")

---

## Step 2: The Three Notification Channels

### Why three separate channels?

> **Question:** Can't we just send everything through one system?  
> **Answer:** No. Each channel has a completely different delivery mechanism, protocol, and third-party provider. They're as different as postal mail, a phone call, and a fax.

### Channel 1: Push Notifications (Mobile)

**iOS** uses **APNs** (Apple Push Notification Service)  
**Android** uses **FCM** (Firebase Cloud Messaging, by Google)

> **Why can't iOS and Android share the same push service?**  
> Apple and Google have completely separate infrastructure. Apple requires all iOS push notifications to go through *their* servers (APNs). You can't bypass it. Same with Google's FCM for Android. It's like two different countries with different postal systems.

**How it works:**
1. User installs your app → app asks APNs/FCM for a **device token** (a unique ID for that device + app combo)
2. Your app sends that token to your backend: *"Hey, this is how you reach me"*
3. When you want to notify the user, your server sends the notification + device token to APNs/FCM
4. APNs/FCM delivers it to the device

> **Analogy:** The device token is like a mailbox address. APNs/FCM is the postman. You hand the postman a letter + address, they deliver it.

### Channel 2: SMS

Uses third-party services like **Twilio** or **Nexmo**.

> **Why not build your own SMS sender?**  
> SMS delivery requires agreements with telecom carriers in every country, managing short codes, handling delivery reports, etc. Services like Twilio have already done all of this. Using them is like using FedEx instead of building your own delivery fleet.

**How it works:**
1. Your server calls Twilio's API with the user's phone number + message
2. Twilio routes it through the appropriate telecom network
3. User receives SMS

### Channel 3: Email

Uses services like **SendGrid** or **Mailchimp** (or AWS SES).

> **Why not use your own SMTP server?**  
> If you send millions of emails from an unknown IP, Gmail/Outlook will mark you as spam. Email providers like SendGrid have "warm" IPs with established trust. They also handle unsubscribe links, bounce management, and spam compliance (legally required). Building this from scratch is months of work.

## Step 3: Collecting Contact Info

Before you can notify anyone, you need their contact details.

```
User signs up / installs app
        |
        v
   Your Backend
        |
   Stores in DB:
   - device_tokens table   → for push notifications
   - users table           → email, phone number
```

> **Question:** What if a user has 3 phones and 2 email addresses?  
> **Answer:** Your `device_tokens` table has a many-to-one relationship with users. One user can have multiple device tokens. When notifying, you send to ALL their active tokens.

**`device_tokens` table:**
| user_id | device_token | platform | created_at |
|---------|-------------|----------|------------|
| 101 | abc123... | iOS | 2024-01-01 |
| 101 | xyz789... | Android | 2024-02-01 |
| 102 | def456... | iOS | 2024-01-15 |

## Step 4: Basic High-Level Design (First Draft)

```
  [Trigger Services]
  (order service, marketing cron job, etc.)
         |
         | "Send this notification"
         v
  [Notification Server]
  - Validates the request
  - Fetches user contact info from DB
  - Calls third-party services directly
         |
    _____|_____
   |     |     |
  APNs  FCM  Twilio  SendGrid
   |     |     |        |
  iOS  Android SMS    Email
```

### What are "Trigger Services"?

These are the parts of your system that *decide* a notification needs to be sent:
- **Order Service** → "Your order shipped" (event-driven)
- **Marketing Cron Job** → "Weekend sale starts now!" (scheduled)
- **Security Service** → "Suspicious login detected" (alert)

> **Question:** Why does the Notification Server need to validate the request?  
> **Answer:** Any internal service can call the Notification Server. Without validation, a bug in the order service could send 1 million emails accidentally. Basic checks: Is the email format valid? Does the user exist? Is the message template valid?

---

## Step 5: Problems with the Basic Design

> **Question:** What's wrong with the design above?  
> **Answer:** Three major problems:

### Problem 1: Single Point of Failure (SPOF)

If the one Notification Server crashes, **zero notifications get sent**. For a system handling millions of messages, this is unacceptable.

> **Analogy:** Imagine a city with only one postal worker. If they call in sick, all mail stops.

### Problem 2: Can't Scale Independently

The DB, cache, and notification processing are all crammed into one server. If email sending is slow, it blocks push notifications too.

### Problem 3: No Buffer for Spikes

What if marketing triggers 5 million emails at 9am? The notification server gets slammed and starts dropping requests.

---

## Step 6: Improved Design (Production-Ready)

```
  [Trigger Services]
         |
         v
  [Notification Servers] ← Multiple instances (horizontal scaling)
         |
   Validates + fetches user data from [Cache] / [DB]
         |
    _____|_____________________
   |           |               |
[Push Queue] [SMS Queue]  [Email Queue]  ← Message Queues (Kafka/SQS)
   |           |               |
[Push Workers] [SMS Workers] [Email Workers]  ← Process independently
   |           |               |
  APNs/FCM   Twilio        SendGrid
```

### Why Message Queues?

> **Question:** Why add Kafka/SQS between the servers and workers?  
> **Answer:** A message queue is a **buffer**. Think of it like a ticket dispenser at a government office. People take a number and wait — the office serves them at their own pace without chaos. 

Benefits:
- **Decoupling:** The Notification Server doesn't care if APNs is slow. It just drops the message in the queue and moves on.
- **Resilience:** If a worker crashes, messages stay in the queue and another worker picks them up.
- **Absorbs spikes:** 5 million emails at 9am? Queue fills up and workers drain it steadily. Nothing crashes.

### Why Separate Queues per Channel?

> **Question:** Can't we use one queue for everything?  
> **Answer:** No. If SMS sending is slow (network issue with Twilio), it would block push notifications too. Separate queues mean each channel is **isolated** — a problem in one doesn't affect others.

### Why Multiple Notification Servers?

> **Question:** Why run 5 notification servers instead of one big one?  
> **Answer:** Horizontal scaling. If one server gets 100 req/sec and crashes, you have 4 others still running. You can also add more servers as traffic grows without changing code. This is more cost-effective than buying one massive server.

## Step 7: Handling Reliability

### Problem: What if a notification is lost?

> **Scenario:** The worker picks up a notification from the queue, calls APNs, but APNs returns an error. What happens to that notification?

### Solution: Notification Log + Retry

```
Worker picks up notification
        |
        v
  [Save to Notification Log DB]  ← "I'm about to send this"
        |
        v
  Call APNs/FCM/Twilio/SendGrid
        |
     Success? ─── Yes → Mark as "delivered" in DB
        |
       No
        |
        v
  Retry with exponential backoff:
  Wait 1s → retry → Wait 2s → retry → Wait 4s → retry...
        |
  After 5 failures → move to Dead Letter Queue (DLQ)
```

> **Question:** What is exponential backoff?  
> **Answer:** Instead of retrying every 1 second (which hammers the service), you wait longer between each retry: 1s, 2s, 4s, 8s, 16s... This gives the failing service time to recover.

> **Question:** What is a Dead Letter Queue (DLQ)?  
> **Answer:** A "failed messages graveyard." Notifications that couldn't be delivered after N retries go here for manual inspection. An engineer can investigate: was it a bad phone number? A provider outage? And decide whether to retry or discard.

## Step 8: Preventing Duplicate Notifications

> **Scenario:** A worker processes a notification and calls APNs successfully. But before updating the DB, the worker crashes. When it restarts, it sees the notification is still "pending" and sends it again. The user gets the same notification twice!

### Solution: Idempotency Keys

Assign every notification a **unique ID** (e.g., `order:12345:shipped:push`).

```
Worker picks up notification
        |
        v
  Check Redis: "Have I processed ID=order:12345:shipped:push before?"
        |
   Yes → Skip it (already sent)
        |
   No → Send notification → Store ID in Redis with TTL
```

> **Analogy:** Like a "seen stamps" system in a post office. Each letter gets a unique stamp. If the same letter arrives twice, it gets rejected the second time.

## Step 9: Additional Features

### 1. Notification Templates

> **Question:** Why use templates instead of sending raw text?  
> **Answer:** Consistency and efficiency. Instead of every service constructing full HTML emails, they just say: "Use template `order_shipped`, fill in `{order_id}` and `{user_name}`."

Stored in DB:
| template_id | channel | locale | subject | body |
|-------------|---------|--------|---------|------|
| order_shipped | email | en-US | "Your order shipped!" | "Hi {{name}}, order #{{id}}..." |
| order_shipped | email | es-MX | "¡Tu pedido salió!" | "Hola {{name}}..." |

### 2. User Notification Preferences (Opt-out)

Users can turn off specific channels:

| user_id | channel | type | enabled |
|---------|---------|------|---------|
| 101 | push | marketing | false |
| 101 | email | transactional | true |
| 102 | sms | all | false |

Before dispatching: check this table. If `enabled = false`, drop the notification.

> **Cache this in Redis!** You don't want a DB query for every single notification. Cache: `user:101:push:marketing → false` with 10 min TTL.

### 3. Rate Limiting

> **Problem:** Marketing team sends 20 push notifications per day per user. Users get annoyed and uninstall the app.

> **Solution:** Implement per-user, per-channel limits.

Using Redis counters:
```
INCR user:101:push:today
EXPIRE user:101:push:today  86400  ← reset after 24 hours

If count > 5 → drop this notification
```

### 4. Do Not Disturb (DND)

Store per-user DND windows:
| user_id | dnd_start | dnd_end | timezone |
|---------|-----------|---------|----------|
| 101 | 22:00 | 08:00 | Asia/Kolkata |

Before sending: convert current UTC time to user's timezone. If within DND window → either discard or **delay** (put into a time-delayed queue, deliver at 8:00am).

> **Exception:** Security alerts (OTP, suspicious login) bypass DND — these are always urgent.

### 5. Event Tracking / Analytics

> **Question:** How does the marketing team know if notifications are working?  
> **Answer:** Track open rates and click-through rates.

How:
- Include a **unique tracking ID** in each notification
- When user taps notification → app sends `notification_opened` event to analytics server
- For emails: embed a **1×1 tracking pixel** (when email opens, it loads the image = we know it was opened)
- Store events in analytics warehouse (BigQuery/Redshift)

Metrics to track:
- **Delivery rate:** What % actually reached the device?
- **Open rate:** What % of delivered notifications were tapped?
- **CTR (Click-Through Rate):** What % clicked a link inside?

## Step 10: Final Architecture (Full Picture)

```
 [Trigger Services: Order, Marketing, Security, etc.]
                    |
                    | POST /api/v1/notify
                    v
          ┌─────────────────────┐
          │  Notification Server │  ← Multiple instances (Load Balanced)
          │  - Auth (AppKey)     │
          │  - Validate request  │
          │  - Check user prefs  │
          │  - Fetch templates   │
          └──────────┬──────────┘
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
    [Push Queue] [SMS Queue] [Email Queue]   ← Kafka / AWS SQS
          │          │          │
    [Push Workers] [SMS W.] [Email Workers]
          │          │          │
     APNs/FCM     Twilio     SendGrid
          │          │          │
       iPhones    SMS inbox   Email inbox

    ─────────────────────────────────────────
    Supporting Components:
    - [Notification Log DB]     ← Persist all notification events
    - [Redis Cache]             ← User prefs, rate limits, dedup keys
    - [User/Device Token DB]    ← Who to notify + how
    - [Analytics Service]       ← Open/click tracking
    - [Dead Letter Queue]       ← Failed notifications for review
```

### Security

> **Question:** What stops anyone from calling your Notification Server API and sending spam?  
> **Answer:** **AppKey + AppSecret** authentication. Only internal services with valid credentials can call the API. Rotate these secrets regularly and store them in a secrets manager (AWS Secrets Manager, HashiCorp Vault) — never hardcode them.

## Step 11: Scaling to 1 Billion Notifications/Day

> **Math:** 1B/day ÷ 86,400 seconds = ~11,574/sec average. Peak could be 50,000+/sec.

What changes:
1. **More Kafka partitions + more worker pods** → horizontal scale
2. **Shard the device token DB** by `user_id` → avoid single DB bottleneck
3. **APNs/FCM HTTP/2 multiplexing** → one connection handles thousands of concurrent requests
4. **Geo-distributed workers** → run workers in AWS Mumbai for Indian users, AWS US-East for US users — reduces latency to APNs/FCM
5. **CDN for notification images** → don't serve from your origin server

## Summary: The "Why" Behind Each Decision

| Decision | Why? |
|----------|------|
| Separate queues per channel | Isolate failures; email slowdown doesn't block push |
| Message queues at all | Decouple producers from consumers; absorb traffic spikes |
| Multiple notification servers | No single point of failure; horizontal scaling |
| Redis for dedup + rate limiting | Sub-millisecond lookups; TTL support built-in |
| Notification log DB | Audit trail; enable retries; prevent data loss |
| Third-party providers (Twilio, SendGrid) | Deliverability, compliance, carrier relationships |
| Exponential backoff retries | Give failing services time to recover; don't hammer them |
| Device token per install | Required by APNs/FCM; the only way to reach a specific device |
| Separate transactional/marketing infra | Protect transactional email deliverability from spam blacklisting |

## Quick Interview Q&A Cheat Sheet

**Q: Why not call APNs directly from the order service?**  
A: Tight coupling. If APNs is slow, the order service hangs. Put a queue in between so the order service just publishes an event and moves on.

**Q: Why use Redis for deduplication instead of the main DB?**  
A: Speed. Redis lookups are O(1) at sub-millisecond. A DB query for every notification at millions/day would create a bottleneck.

**Q: What happens when a device token becomes invalid?**  
A: APNs/FCM return a specific error code (e.g., `DeviceTokenNotForTopic`). The worker catches this and marks the token as inactive in the DB. No more sends to that token.

**Q: What's the difference between a push notification and an in-app notification?**  
A: Push = delivered via APNs/FCM, appears on the lock screen even when app is closed. In-app = a bell icon inside the app showing a list of notifications, stored server-side, visible only when the user opens the app. Most systems implement both.

**Q: How do you handle a 9am marketing blast to 50M users across time zones?**  
A: Store each user's timezone. Use a scheduled job that produces messages **per user** at their local 9am. Spread the sends over 24 hours across different timezones so you don't blast all 50M at once.
