## Why Custom Locks?

`synchronized` puts a **monitor lock on an object**. If threads use **different objects**, they bypass each other's locks.

```java
SharedResource r1 = new SharedResource();
SharedResource r2 = new SharedResource();

// Thread-1: r1.produce() → lock on r1
// Thread-2: r2.produce() → lock on r2 (DIFFERENT!)
// Both enter critical section simultaneously ❌
```

**Custom locks** don't depend on objects — they depend on the **lock object itself**.

```
  ┌────────────────────────────────────────────────────┐
  │  synchronized       → lock per object              │
  │  ReentrantLock      → lock per Lock object         │
  │  ReadWriteLock      → shared/exclusive locking     │
  │  StampedLock        → read/write + optimistic      │
  │  Semaphore          → N threads can enter at once  │
  └────────────────────────────────────────────────────┘
```

## 1. ReentrantLock

Lock that does **not depend on the object** — any number of objects can share the same lock.

```java
ReentrantLock lock = new ReentrantLock();

class SharedResource {
    void produce(ReentrantLock lock) {
        lock.lock();                     // acquire
        try {
            System.out.println("Lock acquired by " + Thread.currentThread().getName());
            Thread.sleep(4000);
        } finally {
            lock.unlock();               // ALWAYS unlock in finally
        }
    }
}

// Same lock passed to different objects → still synchronized
SharedResource r1 = new SharedResource();
SharedResource r2 = new SharedResource();
ReentrantLock lock = new ReentrantLock();

new Thread(() -> r1.produce(lock)).start();  // acquires lock
new Thread(() -> r2.produce(lock)).start();  // WAITS — same lock
```

| Feature | synchronized | ReentrantLock |
|---|---|---|
| Lock depends on | Object | Lock object |
| Multiple objects | ❌ Different locks | ✅ Same lock shared |
| Unlock location | Auto (end of block) | Manual (`finally` block) |
| Try-lock | ❌ | ✅ `tryLock()` |
| Fairness | ❌ | ✅ `new ReentrantLock(true)` |

## 2. ReadWriteLock

Implements **shared lock (read)** and **exclusive lock (write)**.

### Shared vs Exclusive Lock Rules

```
  ┌──────────────────────────────────────────────────────────┐
  │  Thread holds    │ Another thread can acquire:           │
  │──────────────────┼───────────────────────────────────────│
  │  Shared (Read)   │ Shared (Read) ✅  │ Exclusive (Write) ❌│
  │  Exclusive (Write)│ Shared (Read) ❌  │ Exclusive (Write) ❌│
  │  No lock          │ Shared (Read) ✅  │ Exclusive (Write) ✅│
  └──────────────────────────────────────────────────────────┘
```

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();

// READ — multiple threads allowed
void readData(ReadWriteLock lock) {
    lock.readLock().lock();
    try {
        // read-only operations
    } finally {
        lock.readLock().unlock();
    }
}

// WRITE — exclusive access
void writeData(ReadWriteLock lock) {
    lock.writeLock().lock();
    try {
        // modify data
    } finally {
        lock.writeLock().unlock();
    }
}
```

| Lock Type | Alias | Concurrent? | Can Modify? |
|---|---|---|---|
| Read Lock | Shared | ✅ Multiple threads | ❌ Read only |
| Write Lock | Exclusive | ❌ One thread only | ✅ Read + Write |

> **When to use:** Applications with **high reads, low writes** — readers don't block each other.

## 3. StampedLock

Provides **Read/Write lock** + **Optimistic Read** (no actual lock acquired).

### Optimistic Locking (Lock-Free Read)

```java
StampedLock lock = new StampedLock();

// Optimistic read — no lock acquired
long stamp = lock.tryOptimisticRead();     // saves "version"
// perform read operations
int value = sharedValue;

if (lock.validate(stamp)) {                // check if anyone wrote in between
    // no write happened → use value ✅
} else {
    // someone wrote → rollback, retry ❌
}
```

```
  Thread-1                        StampedLock                  Thread-2
       │                                                          │
       ├── tryOptimisticRead() ──▶ stamp = version 1              │
       │   (no lock acquired)                                     │
       │   reading value...                                       │
       │                          ◄── writeLock() ────────────────┤
       │                              version → 2                 │
       │                          ◄── writeUnlock() ──────────────┤
       ├── validate(stamp) ──────▶ stamp=1, current=2 → INVALID! │
       │   → ROLLBACK                                             │
```

### Read/Write Lock (Same as ReadWriteLock)

```java
// Read lock with stamp
long stamp = lock.readLock();
try { /* read */ } finally { lock.unlockRead(stamp); }

// Write lock with stamp
long stamp = lock.writeLock();
try { /* write */ } finally { lock.unlockWrite(stamp); }
```

| Feature | ReadWriteLock | StampedLock |
|---|---|---|
| Read lock | ✅ | ✅ |
| Write lock | ✅ | ✅ |
| Optimistic read | ❌ | ✅ (no lock, validate later) |
| Stamp (version) | ❌ | ✅ Used for validation |

## 4. Semaphore

Allows **N threads** to enter a critical section simultaneously (configurable permits).

```java
Semaphore semaphore = new Semaphore(2);  // 2 permits

void task() {
    semaphore.acquire();     // takes 1 permit (blocks if none available)
    try {
        // critical section — max 2 threads at a time
    } finally {
        semaphore.release();  // returns 1 permit
    }
}
```

```
  Permits = 2

  Thread-1: acquire() → permit 1 taken → enters ✅
  Thread-2: acquire() → permit 2 taken → enters ✅
  Thread-3: acquire() → no permits → WAITS ⏳
  Thread-1: release() → permit freed
  Thread-3: acquire() → permit available → enters ✅
```

### Use Cases

| Scenario | Permits |
|---|---|
| **Printer pool** (2 printers) | `new Semaphore(2)` |
| **DB connection pool** (5 connections) | `new Semaphore(5)` |
| **Rate limiting** | `new Semaphore(maxConcurrent)` |

## Condition — Inter-Thread Communication

`wait()` / `notify()` work only with `synchronized` (monitor lock). For custom locks, use **Condition**.

| Synchronized | Custom Lock |
|---|---|
| `wait()` | `condition.await()` |
| `notify()` | `condition.signal()` |
| `notifyAll()` | `condition.signalAll()` |

```java
ReentrantLock lock = new ReentrantLock();
Condition condition = lock.newCondition();  // create Condition from lock

void produce() {
    lock.lock();
    try {
        while (itemAvailable) {
            condition.await();        // wait on this lock's condition
        }
        itemAvailable = true;
        condition.signal();           // wake up one waiting thread
    } finally {
        lock.unlock();
    }
}

void consume() {
    lock.lock();
    try {
        while (!itemAvailable) {
            condition.await();
        }
        itemAvailable = false;
        condition.signal();
    } finally {
        lock.unlock();
    }
}
```

## Summary

| Lock Type | Key Feature | Depends On |
|---|---|---|
| **synchronized** | Monitor lock | Object |
| **ReentrantLock** | Object-independent lock | Lock object |
| **ReadWriteLock** | Multiple readers, exclusive writer | Lock object |
| **StampedLock** | Read/Write + optimistic read | Stamp (version) |
| **Semaphore** | N concurrent threads | Permits count |
| **Condition** | `await()`/`signal()` for custom locks | Lock object |
