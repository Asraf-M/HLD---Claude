# System Design: Real-Time Analytics Platform (Beginner-Friendly Guide)

---

## What Are We Building?

Imagine Mixpanel or Amplitude — an analytics platform that tracks user behavior in real-time. When someone:
- Clicks a button → event sent
- Completes a purchase → event sent
- Abandons cart → event sent

The platform:
- Collects millions of events per second
- Computes dashboards: "How many users active now?" (live counter)
- Analyzes trends: "What's my retention rate?" (complex SQL queries)
- Alerts on anomalies: "Traffic just spiked 2x" (automated alert)

The interesting engineering challenges hidden in this flow:
- **Massive volume** — 1M events/sec = 86B events/day = petabytes of data
- **Real-time dashboards** — queries must return in <1 second even searching 100TB
- **Complex analytics** — "Funnel analysis" requires joining 3 event tables, complex logic
- **Cost efficiency** — hot (recent) data is searchable but expensive; cold (old) data is archived/cheap
- **Privacy** — events contain PII (emails, behavior); must mask/delete on request
- **Duplicate prevention** — same event sent twice (network retry); must deduplicate

## Step 3: Database Choice

| Consideration | SQL | OLAP (ClickHouse) | Time-Series |
|---------------|-----|---|---|
| Events (1M/sec writes) | ❌ Slow | ✅ Built for it | ✅ OK |
| Aggregations (GROUP BY) | ✅ OK but slow | ✅ Fast | ❌ Not ideal |
| Retention (1 year data) | ❌ Expensive | ✅ Compressed | ✅ OK |
| Complex joins | ✅ Good | ✅ Good | ❌ Limited |

**Decision: ClickHouse + Kafka**
- **Kafka** for event stream (buffer spikes; 1M events/sec easy)
- **ClickHouse** for analytics (OLAP DB; designed for this workload)
- **S3** for long-term cold storage
- **Redis** for real-time metrics (live dashboard counters)

### Deduplication

```
Problem: Network glitch → client retries event → sent twice

Solution:
1. Each event has unique eventId (UUID)
2. First request: Store eventId, process event
3. Retry request: Check eventId in cache
   - If found: Return success (duplicate)
   - If not found: Process event

Implementation:
- Redis cache: "eventId:evt_123456" → processed
- TTL: 24 hours (cover retry window)
```

## Step 7: Analytics Queries (ClickHouse)

### Query Examples

```sql
-- Revenue by country (last 7 days)
SELECT country, SUM(amount) as revenue
FROM events
WHERE event = 'purchase'
  AND date >= today() - 7
GROUP BY country
ORDER BY revenue DESC;

-- Funnel analysis
SELECT
  COUNT(DISTINCT CASE WHEN event='page_view' THEN userId END) as viewed,
  COUNT(DISTINCT CASE WHEN event='add_to_cart' THEN userId END) as added_cart,
  COUNT(DISTINCT CASE WHEN event='purchase' THEN userId END) as purchased
FROM events
WHERE date = today();

-- User retention (% of day 1 users active on day 7)
WITH day1_users AS (
  SELECT DISTINCT userId
  FROM events
  WHERE toDate(timestamp) = '2024-06-11'
)
SELECT
  COUNT(DISTINCT e.userId) as retained,
  (SELECT COUNT(*) FROM day1_users) as total,
  ROUND(100 * COUNT(*) / (SELECT COUNT(*) FROM day1_users)) as retention_pct
FROM events e
WHERE toDate(e.timestamp) = '2024-06-18'
  AND userId IN (SELECT userId FROM day1_users);
```

### Performance Optimization

```
ClickHouse optimizations:

1. Columnar storage:
   Instead of storing rows, store columns
   - All user_ids together
   - All amounts together
   - Query "SUM(amount) WHERE event='purchase'"
   - Only reads amount column (90% less I/O)

2. Partitioning by date:
   Old data (2023) on slow storage
   Recent data (2024) on fast storage

3. Pre-computed aggregations:
   Instead of "SUM revenue per country" (slow, full scan)
   Pre-compute nightly: "daily_revenue_by_country" table
   Query: 10ms vs 10 seconds
```

```

- Avg revenue/hour: $50K
- StdDev: $5K

- False positives: Too many alerts → engineers ignore
```

## Step 11: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Kafka queue | Decouple; buffer spikes; never lose data | Added complexity; harder to debug; eventual consistency |
| ClickHouse OLAP | Built for analytics; columnar = fast; handles 1M events/sec | Not ACID; requires data warehousing expertise |
| Real-time aggregation (Flink) | Live dashboard (users now, revenue today) | Complexity; eventual consistency with batch (temporary lag) |
| Pre-computed aggregations | Queries instant (10ms vs 10 sec) | Stale by 1 hour; manual logic for new metrics |
| Event deduplication (24h window) | Prevent double-counting from retries | Edge case: event retried after 24h counts twice |

## Interview Cheat Sheet Q&A

**Q: Why not use a traditional database for analytics?**  
A: Traditional DB (SQL Server, MySQL) optimized for rows: "Get user 123's record." Analytics needs: "Sum revenue for 1B purchases grouped by country." Row-oriented DB scans all rows for each column. ClickHouse scans only the needed column (100x faster).

**Q: What if 2M events/sec arrive (2x normal)?**  
A: Kafka buffers them (10-minute backlog, no data loss). ClickHouse ingests from Kafka at its pace. Dashboard updates lag slightly (shows 5-minute-old data), but no crashes. Auto-scale: add more consumers (read from Kafka in parallel).

**Q: Can we query data from 1 year ago?**  
A: Yes, but slow (cold storage). Querying 365 days of archived data: 5-30 seconds. For real-time: cache recent 30 days (expensive SSD). For historical: Archive to S3 (cheap, slow). User expects: "Real-time dashboard" = fast; "Historical report" = wait 30 seconds.

**Q: How do we prevent alerts from crying wolf (false positives)?**  
A: Threshold tuning. Alert: "Revenue dropped 50% for 10 minutes" = maybe transient. Alert: "Revenue dropped 50% for 1 hour" = real problem. Set duration threshold (for: 1h) to filter out noise.

**Q: What if ClickHouse crashes?**  
A: Kafka has the data (replicated, durable). Restart ClickHouse → re-ingest from Kafka (backfill). Takes 1-2 hours to restore. In meantime: dashboards show stale data (from 2 hours ago). Not ideal, but data isn't lost.

**Q: Why partition by userId in Kafka?**  
A: Ensures all events from one user go to same partition → preserve order. User's journey: page_view → add_to_cart → purchase. If events out of order: looks like purchase before page view (wrong). Partitioning maintains per-user causality.

### Event Deduplication

```
Stream deduplication:
1. Event arrives with id: evt_123
2. Check if seen before (within 24 hours)

3. If not seen:
   - Process event

4. If seen:
   - Drop event (duplicate)

- Use Redis for short-term (1-2 hours)
- Use persistent DB for longer
```

## Step 7: Complex Analytics Queries

### Cohort Analysis

```sql
-- Create cohort of users who signed up in June
WITH june_cohorts AS (
  SELECT
    user_id,
    DATE_TRUNC('week', signup_date) as cohort_week,
    DATE_TRUNC('week', event_date) as event_week,
    DATEDIFF(week, signup_date, event_date) as week_num
  FROM users u
  JOIN events e ON u.user_id = e.user_id
  WHERE DATE_TRUNC('month', signup_date) = '2024-06-01'
    AND event = 'purchase'
)
SELECT
  cohort_week,
  week_num,
  COUNT(DISTINCT user_id) as active_users
FROM june_cohorts
GROUP BY cohort_week, week_num
ORDER BY cohort_week, week_num;
```

### Retention Analysis

```sql
-- Calculate day-over-day retention
WITH users_by_day AS (
  SELECT
    DATE(event_timestamp) as event_date,
    user_id
  FROM events
  WHERE event IN ('login', 'purchase')
)
SELECT
  a.event_date,
  b.event_date,
  COUNT(DISTINCT b.user_id) as returning_users,
  COUNT(DISTINCT a.user_id) as total_users,
  ROUND(100 * 
    COUNT(DISTINCT b.user_id) / 
    NULLIF(COUNT(DISTINCT a.user_id), 0)
  ) as retention_percent
FROM users_by_day a
LEFT JOIN users_by_day b
  ON a.user_id = b.user_id
  AND b.event_date = a.event_date + INTERVAL 1 DAY
WHERE a.event_date >= CURRENT_DATE - 30
GROUP BY a.event_date, b.event_date;
```

```
Hot Storage (0-30 days):
- Elasticsearch or ClickHouse
- Fast queries (< 1 second)
- Full indexing
- Cost: $1.00/GB/month

Warm Storage (30-365 days):
- Parquet files in S3
- Slower queries (5-30 seconds)
- Columnar format
- Cost: $0.10/GB/month

Cold Storage (> 1 year):
- Glacier / Deep Archive
- For compliance/audit
- Cost: $0.01/GB/month
```

```
- Reduce network transfer
- Example: Query uses 5% of columns
```

## Step 12: Key Design Decisions & Tradeoffs

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Separate real-time and batch processing | Real-time for dashboards (low latency); batch for accuracy | Complexity; eventual consistency |
| OLAP database for analytics | Fast aggregations (sum, count, group-by); columnar | Slower for point lookups ("get user by ID"); need separate DB |
| Event deduplication with bloom filter | Prevent double-counting events | Bloom filters have false positives (tiny %, acceptable) |
| Tiered storage (hot/warm/cold) | Cost savings; older data 100x cheaper | Slow queries on old data; complexity |
| Pre-computed aggregations (materialized views) | Dashboard loads instantly (cached aggregates) | Stale by 1 hour; manual logic for new metrics |

## Interview Cheat Sheet Q&A

**Q: Why not just use your main database for analytics?**  
A: Main database is optimized for transactional queries: "Get user 123's orders." Analytics needs aggregations: "Sum revenue by country." These use different indexing. Running big scans on prod DB slows down the app. Analytics needs a separate, specialized system (OLAP).

**Q: What if a user sends the same event twice?**  
A: Without deduplication, you count it twice. "User X viewed product 100 times" when it was really 50. Solution: give each event a unique ID. Check if seen before (Redis or Bloom filter). If not, process; if yes, drop.

**Q: How long does real-time processing lag behind actual events?**  
A: Usually 10-30 seconds. Event happens → sent to Kafka (1 sec) → Flink processes (5 sec) → writes to cache (5 sec) → dashboard refreshes (10 sec) = 20 sec lag. Good for dashboards (shows "live" data). Not good for fraud detection (too slow). For fraud, you need <1 sec.

**Q: Can we customize a metric that wasn't pre-computed?**  
A: Yes, but it's slow (query the raw data warehouse). Pre-computed aggregates are instant. Custom queries take 5-30 seconds. For a dashboard you view once, that's fine. For a metric you check 100x/day, pre-compute it.

**Q: How do we handle user deletion (GDPR)?**  
A: Real-time: remove from live streams, don't process new events. Batch: delete from data warehouse. But aggregates are tricky (can't "un-sum"). Solution: mark user as deleted in database, exclude from new reports ("data from active users only"). Old reports are archived, left as-is.

**Q: What if we want to analyze events that happened 5 years ago?**  
A: They're in cold storage (S3 Glacier, expensive). Takes hours to retrieve. For rare requests, it's worth it. For frequent access, move to warm storage. Most analytics queries look at recent data ("last 30 days," "last year"), not 5 years ago.
