# System Design: Logging & Monitoring System (Beginner-Friendly Guide)

---

## What Are We Building?

Imagine platforms like Datadog or Elastic — a centralized system that collects logs and metrics from thousands of services and servers. When you have 100 microservices, each writing millions of log lines per day, you need:
- Collect logs from all services (without crashing them)
- Index them for fast search ("Show all errors from user-service in the last hour")
- Track system metrics (CPU, memory, disk)
- Alert engineers when something breaks (before customers complain)
- Create dashboards showing system health

The interesting engineering challenges hidden in this flow:
- **Massive volume** — if each service writes 10,000 logs/sec, and you have 100 services, that's 1M logs/sec = 86TB/day. Can't store all of it forever.
- **Low-latency search** — "Find all errors" must return in <1 second, even searching across 100TB.
- **Real-time alerting** — if error rate spikes, alert engineer within 1 minute (not 10 minutes later).
- **Data retention** — keep recent logs for quick debugging (expensive); older logs in archive (cheaper).
- **Privacy** — logs contain PII (emails, phone numbers); must mask or encrypt them.

**Key features:**
- Collect logs from 100+ microservices

- Real-time alerting on anomalies

**Non-functional requirements:**
- Handle 1M logs/sec throughput
- Search latency <1 second

## Step 3: Database Choice

| Consideration | SQL | Elasticsearch | Time-Series DB |
|---------------|-----|---|---|
| Full-text search | ❌ Slow | ✅ Fast (1-2 sec) | ❌ Not designed for it |
| High writes (1M/sec) | ❌ Slow | ✅ Built for logs | ✅ Very fast |
| Aggregations | ✅ SQL | ✅ Good | ✅ Designed for metrics |
| Time-range queries | ✅ OK | ✅ Fast | ✅ Very fast |

**Decision:**
- **Elasticsearch** for log storage and search (full-text + high writes)
- **Prometheus** for metrics (time-series data)
- **S3** for archival (cold storage after 30 days)

## Step 5: Log Ingestion

### Collection Agent (Fluentd/Logstash)

```
Runs on each host:

1. Tail log files:
   /var/log/user-service.log
   /var/log/order-service.log
   ...

2. Parse logs (extract structure):
   Raw: "2024-06-18T10:30:45Z ERROR user-service Failed to fetch"
   Structured: {
     "timestamp": "2024-06-18T10:30:45Z",
     "level": "ERROR",
     "service": "user-service",
     "message": "Failed to fetch"
   }

3. Enrich logs:
   Add hostname: "server-001"
   Add env: "production"
   Add version: "1.2.3"

4. Buffer in memory:
   Batch 1000 logs together
   Send to Kafka every 5 seconds
   
5. Retry logic:
   If Kafka down, buffer locally
   Retry when Kafka back up
```

### Preventing Logs from Blocking Application

```
Application writes log:
  logger.info("User logged in")
  ↓
Goes to file (fast, asynchronous)
  ↓
Fluentd tails file (non-blocking)
  ↓
Application doesn't wait for Fluentd/Kafka

Benefit: Logging never slows down application
```

## Step 6: Elasticsearch Indexing & Storage

### Index-per-Day Strategy

```
Indices created daily:
- logs-2024-06-16
- logs-2024-06-17
- logs-2024-06-18
- logs-2024-06-19

Benefits:
- Old index (06-16) can be deleted/archived in bulk
- New index (06-19) optimized for writes
- Parallel writes to different indices

Alias for querying:
- Query "logs" alias → searches latest indices
```

### Tiered Storage

```
HOT (0-3 days):
- Fast SSD storage
- Fully indexed
- Cost: $0.50/GB/month
- Use case: Debugging recent issues

WARM (3-30 days):
- HDD storage
- Indexed but read-only
- Cost: $0.10/GB/month
- Use case: Historical analysis

COLD (30-365 days):
- Archive (S3, Glacier)
- Not indexed
- Cost: $0.01/GB/month
- Use case: Compliance, rare queries

FROZEN (>365 days):
- Deep Archive (tape)
- Cost: $0.001/GB/month
- For compliance only
```

### Compression

```
Original logs: 86TB/day
After compression (gzip): 4.3TB/day (20:1 ratio)

Storage cost reduction:
Hot (7 days): 4.3TB × 7 × $0.50/TB = $15/day
vs Uncompressed: 86TB × 7 × $0.50/TB = $300/day

Savings: $285/day = $100K+/year
```

## Step 7: Querying & Search

### Full-Text Search

```
Find all errors in user-service (last 24h):

GET /logs/search
{
  "query": {
    "bool": {
      "must": [
        {"match": {"service": "user-service"}},
        {"match": {"level": "ERROR"}},
        {"range": {"timestamp": {
          "gte": "now-24h"
        }}}
      ]
    }
  }
}

Elasticsearch:
1. Reads index logs-2024-06-18, logs-2024-06-17
2. Inverts index: ERROR → [doc_1, doc_2, doc_3]
3. Returns 250 matching logs
4. Sorts by timestamp descending

Latency: 350ms for 100TB index (fast!)
```

```
Count errors by service (last hour):
```

## Step 8: Metrics Collection (Prometheus)

### Metric Types

```
1. COUNTER (always increasing):
   http_requests_total: 1000 → 1001 → 1002
   Use for: requests, errors, transactions

2. GAUGE (current value):
   active_connections: 42
   Use for: CPU, memory, connections

3. HISTOGRAM (distribution):
   request_latency_ms: [100, 200, 150, 180, ...]
   Buckets: 0-100, 100-500, 500-1000, ...
   Use for: latencies, sizes

4. SUMMARY (percentiles):
   request_latency_quantile{quantile="0.95"}: 350
   Use for: P50, P95, P99 latencies
```

### Collection Flow

```
Application:
metrics.inc('requests_total')
metrics.set('active_connections', 42)
metrics.observe('request_latency', 150)

Prometheus Server:
1. Scrapes /metrics endpoint every 15 seconds
2. Stores data in TSDB (Time-Series Database)
3. Data retention: 15 days (configurable)
4. Evaluation: Alerting rules every 30 seconds

Query:
rate(requests_total[5m])  → average requests/sec in last 5 minutes
histogram_quantile(0.95, request_latency) → 95th percentile latency
```

## Step 9: Real-Time Alerting

### Alert Rules

```yaml
- alert: HighErrorRate
  expr: |
    rate(requests{status="500"}[5m]) > 10
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "High error rate ({{ $value }} errors/sec)"

- alert: HighLatency
  expr: |
    histogram_quantile(0.95, request_latency) > 500
  for: 10m
  labels:
    severity: warning
```

### Alert Flow

```
1. Prometheus evaluates rule every 30 seconds
2. Condition met (e.g., error rate > 10/sec for 5 mins)
3. Alert fired → sent to AlertManager
4. AlertManager:
   - Deduplicates (same alert, multiple Prometheus replicas)
   - Groups related alerts
   - Routes to notification channel

5. Notification channels:
   - Email (slow, reliable)
   - Slack (fast, interactive)
   - PagerDuty (escalates if not acknowledged)
   - Webhook (custom integration)
```

## Step 11: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Elasticsearch for logs | Full-text search; high ingestion | Expensive; requires tuning; index explosion |
| Kafka for buffering | Decouple services from Elasticsearch; handle spikes | Added complexity; eventual consistency |
| Index-per-day strategy | Easy deletion; parallel writes; old data optimization | Need alias; manually manage indices |
| Tiered storage (hot/warm/cold) | Cost savings (100x cheaper for old data) | Complexity; slower queries on cold data |
| Prometheus for metrics | Lightweight; standard in industry; good for alerts | Doesn't replace APM (application performance monitoring) |

## Interview Cheat Sheet Q&A

**Q: Why not just log to a single database?**  
A: 1M logs/sec × 86 SQL statements/sec = SQL DB overloaded. Elasticsearch is built for log-like data: writes are parallelized, data is denormalized, indices optimized for text search. SQL is optimized for relational queries ("Get user 123"), not bulk ingestion.

**Q: What if Kafka queue fills up?**  
A: Logs are dropped (lossy). But that's OK: logs are analytics, not critical. Losing some logs is fine. Drop oldest logs first (ring buffer). If critical: alert ops, they increase Kafka capacity or scale Elasticsearch.

**Q: How many Elasticsearch nodes for 1M logs/sec?**  
A: ~20 nodes (varies by config). Each node ~50K logs/sec. With replication (2 replicas), 1M logs/sec needs 60 shards across 20 nodes = 3 shards/node. Complex tuning required.

**Q: Can we search 365 days of cold storage logs?**  
A: Technically yes, but slow (hours). Cold storage isn't indexed. You'd download from S3, decompress, re-index temporarily. Expensive. Use for rare compliance queries, not daily use.

**Q: What if we accidentally log a password?**  
A: Immediately: mask before sending to Elasticsearch ("password: ****"). Or filter at Fluentd (drop sensitive fields). Once in Elasticsearch, can't undo (stored). Requires data deletion (GDPR compliance).

**Q: How do we prevent alert fatigue (engineers ignoring alerts)?**  
A: Set thresholds carefully. Alert if error_rate > 5% for 5 min (not every spike). Group related alerts (5 errors = 1 alert, not 5). Escalation: after 5 min no acknowledgment, alert manager → page on-call engineer. Make alerts actionable.


## Step 10: Log Retention & Compliance

### Data Retention Policy

```
Production Logs:
- Hot: 7 days (searchable)
- Warm: 30 days (less searchable)
- Cold: 365 days (archive)
- Delete: > 365 days

Cost optimization:
- Hot: 1000 GB × 30 days × $0.5/GB = $15,000/month
- Warm: 10 TB × 365 days × $0.2/GB = $73,000/year
- Cold: 100 TB × indefinite × $0.05/GB

Instead:
- Hot: 50 GB × 7 days = $100/month
- Warm: 500 GB × 23 days = $2,300/month
- Cold: 50 TB × 365 days = $2,500/year
```

### Compliance & Data Privacy

```
GDPR Compliance:
- PII in logs: Email, phone, SSN
- Mask in Elasticsearch: username-xxxx@xxx.com
- Delete logs on user deletion
- Audit who accessed logs

SOC 2 Compliance:
- Immutable audit logs
- Log access tracked
- Retention for 2 years
- Encryption in transit and at rest
```

### High Volume Ingestion

```
1M events/second:

1. Kafka cluster:
   - Multiple brokers
   - Partitioned by service
   - Replication factor: 3

2. Elasticsearch:
   - Multiple nodes
   - Index shards: 5-10 per day
   - Replicas: 1-2
   - Bulk indexing: 5000 docs/batch

3. Ingestion pipeline:
   - Multiple Logstash instances
   - Consumer groups for parallelism
   - Batch processing (not per-message)
```

## Step 12: Key Design Decisions & Tradeoffs

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Kafka for log ingestion | Handle 1M events/sec; buffer spikes | Added complexity; need Kafka ops expertise |
| Elasticsearch for search | Fast full-text search; aggregations | High resource usage; requires cluster management |
| Tiered storage (hot/warm/cold) | Cost savings; older data cheaper | Complexity; slower queries on old data |
| Real-time metrics with Prometheus | Sub-second freshness; good for alerting | High cardinality can explode storage |
| Distributed tracing | See request flow across services; debug better | Overhead (each request adds 10KB tracing data); complexity |

## Interview Cheat Sheet Q&A

**Q: Why not just log to a file and grep through logs?**  
A: Works for 1 server. For 1000 servers, you have 1000 log files. Grepping all of them is slow (minutes). You want: one central place (Elasticsearch), search in 1 second across all servers.

**Q: Why Elasticsearch instead of a database?**  
A: Databases are row-oriented (good for "select * where id=123"). Elasticsearch is document-oriented (good for text search). Searching for "ERROR" in all logs requires filtering millions of rows. Elasticsearch indexes words, so "ERROR" search is instant.

**Q: How much data do 1M events/second generate?**  
A: Each event ~1KB. 1M/sec × 1KB = 1GB/sec = 86TB/day. Storage is cheap (~$0.10/GB), but processing is expensive. That's why tiering: hot (expensive, searchable) for recent data, cold (cheap, slow) for old data.

**Q: What if someone accidentally logs a password?**  
A: Hash it before logging (can't decrypt). Or mask in logs (show only last 4 chars). But best: don't log secrets. Use structured logging: `logging.info({"action": "login", "user_id": "123"})` without the password field.

**Q: Can we real-time alert on errors?**  
A: Yes, via Prometheus: "If error_rate > 5% for 5 minutes, alert." But beware of false positives. If you alert on every spike, engineers ignore alerts (alert fatigue). Set thresholds carefully, test before deploying.

**Q: How do we prevent logs from being tampered with?**  
A: Immutable logging: write logs to append-only storage (can't modify old logs). Also, sign log entries: log = hash(previous_log + current_entry). If someone modifies an old log, the hash breaks. For compliance audits, this is important.

A logging and monitoring system requires:

- ✅ Alerting on anomalies
- ✅ Distributed tracing for debugging
- ✅ Tiered storage for cost efficiency
- ✅ Dashboards and visualization (Grafana)
- ✅ Compliance and retention policies
- ✅ Scalability for massive volume
