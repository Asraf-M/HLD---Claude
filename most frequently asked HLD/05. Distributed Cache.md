# System Design: Distributed Cache (Beginner-Friendly Guide)

---

## What Are We Building?

Think of Redis or Memcached — an in-memory distributed caching layer that sits between your application and database. It provides:
- Ultra-fast access to frequently used data (< 1ms vs 10ms from database)
- Reduced database load (caching absorbs 80% of reads)
- Scalable horizontal architecture (add nodes as users grow)
- High availability (survives node failures)
- Flexible data structures (strings, lists, sets, hashes, sorted sets)
- Optional persistence (recover from restarts)

The interesting engineering challenges hidden in this simple flow:
- **Scale** — millions of concurrent users reading/writing simultaneously; single cache node handles only 100K operations/sec
- **Hot keys** — what if 1M users all request the same key (e.g., celebrity product on Black Friday)?
- **Cache stampede** — popular key expires, 10K users all hit database simultaneously
- **Consistency** — cache must stay in sync with database; stale data is worse than no cache
- **Eviction** — when memory fills up, which cached items to discard?
- **Persistence** — how to recover after restart without losing cached data?
- **Failure** — if cache node fails, how to rebuild without overloading database?

## Step 1: Design Scope

**Scale:**
| Parameter | Value |
|-----------|-------|
| Concurrent connections | 1 million |
| QPS (peak) | 1-10 million ops/sec |
| Data size (typical) | 10-100 GB per node |
| Memory per node | 100-500 GB |
| P99 latency | < 5ms |
| Availability | 99.99% uptime |
| Fault tolerance | Can lose 1 node without data loss |

**Key feature:** Cache must serve reads much faster than database (1ms vs 10ms) while staying synchronized with actual data.

**QPS Funnel:**
```
Database read latency:     ~10ms
Cache hit latency:         <1ms (10x faster)
Cache miss latency:        ~20ms (includes DB fetch + cache store)
```

**Non-functional requirements:**
- Sub-millisecond read latency
- Support thousands of concurrent connections
- Horizontal scaling (add nodes easily)
- Survive node failures without data loss
- High throughput (millions of ops/sec)

## Step 2: API Design

**Core Cache Operations:**
```
GET key                    ← retrieve value
SET key value              ← store value
SET key value EX 3600      ← store with TTL (1 hour)
DEL key                    ← delete value
EXISTS key                 ← check if exists
TTL key                    ← get remaining TTL
EXPIRE key 3600            ← set TTL
```

**Example: Session Cache**
```
SET session:{userId} {sessionData} EX 86400
→ Stores user session, auto-expires after 24 hours

GET session:{userId}
→ Returns session object

DEL session:{userId}
→ Logs user out
```

**Example: Rate Limiting**
```
INCR requests:{userId}:minute
EXPIRE requests:{userId}:minute 60
→ Tracks requests per minute

If requests > 100, reject request
```

**Example: Leaderboard (Sorted Set)**
```
ZADD leaderboard 1000 "alice" 950 "bob" 900 "charlie"
ZREVRANGE leaderboard 0 9
→ Returns top 10 by score

ZRANK leaderboard "alice"
→ Returns alice's rank (1st)
```

## Step 4: Caching Algorithms & Eviction Policies

### Problem: Cache Memory is Finite

Cache node has 100 GB RAM. After 100 GB filled, what happens?

### Solution 1: LRU (Least Recently Used) — MOST COMMON

```
Evict the item accessed least recently

Example:
Cache access order: A, B, C, A, B, D, E
Memory limit: 4 items
Cache state: [D, E, A, B] (C evicted, was least recently used)

Access A:
Cache state: [D, E, B, A] (A moved to most recent)
```

**Pros:** Simple, effective for most workloads
**Cons:** Requires tracking access time (overhead)

**Real-world example:**
- User 1 accesses key at 10:00 AM
- User 2 accesses key at 10:15 AM
- If eviction needed, User 1's data evicted (less recent)

### Solution 2: LFU (Least Frequently Used)

```
Evict the item accessed least frequently

Example:
Access frequency: A(100 times), B(50), C(1), D(1)
If need space, evict C or D (accessed only once)
Keep A and B (accessed many times)
```

**Pros:** Better for repeated access patterns
**Cons:** More complex to track frequency

### Solution 3: TTL (Time-To-Live) — MOST PRACTICAL

```
Instead of evicting, set expiration time

SET key value EX 3600
→ Automatically deletes after 1 hour

Pros: Predictable, no wasted memory after expiry
Cons: Must choose right TTL (too short = cache useless, too long = stale data)
```

## Step 5: Cache Invalidation Strategies

### Challenge: Data Changes Faster Than Cache TTL

Original cache value: $100
Product price updated to $95
Cache still shows $100 for next hour
User pays $100 instead of $95
Lost revenue!

### Solution 1: TTL-Based (Simple but Stale)

```
SET price:{productId} 100 EX 300
→ Expires after 5 minutes
→ Worst staleness: 5 minutes
→ Trade-off: Fresh data every 5 min, or stale for up to 5 min

Good for: Non-critical data (trending products, popular items)
Bad for: Critical data (prices, inventory, payment info)
```

### Solution 2: Event-Based Invalidation (Fresh but Complex)

```
When product price changes:
1. Update database
2. Publish event: "price:updated:productId:789"
3. Cache subscriber receives event
4. Deletes from cache: DEL price:{productId}
5. Next request fetches fresh value from DB

Freshness: Immediate (< 1 second)
Complexity: Requires event infrastructure (Kafka, RabbitMQ)
```

### Solution 3: Write-Through Caching (Consistent but Slow)

```
User updates product price:
1. Write to cache:  SET price:{productId} 95
2. Wait for confirmation
3. Write to database (synchronous)
4. Return success

Pros: Cache and DB always consistent
Cons: Every write goes to both (slower, 2x latency)
```

### Solution 4: Write-Behind Caching (Fast but Risky)

```
User updates product price:
1. Write to cache: SET price:{productId} 95
2. Return success immediately
3. Asynchronously write to database (fire-and-forget)

Pros: Fast writes (only cache latency), improved UX
Cons: If cache crashes before DB sync, data lost

Risk: Acceptable for non-critical data (analytics, recommendations)
Unacceptable for critical data (payments, inventory)
```

## Step 6: Distributed Cache Architecture

### Single Node (Not Scalable)

```
App 1 → Cache Node 1 ← App 2
App 3 ↗            ↖ App 4

Problem:
- Cache node becomes bottleneck
- Can only handle 100K ops/sec
- If node dies, all cached data lost
```

### Replicated Cache (High Availability)

```
App Servers
    ↓
Master Cache Node (reads + writes)
    ↓
Slave Cache Node 1 (reads, syncs from master)
Slave Cache Node 2 (reads, syncs from master)

If master dies:
→ Promote slave to master (automatic failover via Sentinel)
→ Apps switch to new master
→ Downtime: < 1 second
```

### Distributed Cache Cluster (Horizontal Scaling)

```
Consistent Hashing Ring:

                    Node A
                   /      \
             Node D        Node B
                   \      /
                    Node C

Key "user:1" hashes to Node A
Key "user:2" hashes to Node B
Key "user:3" hashes to Node C
Key "user:4" hashes to Node D

Each node stores 1/4 of data
Total capacity = 4× single node

Adding Node E:
→ Only ~1/4 of keys rehash
→ Other 3/4 unaffected (minimal disruption)
```

## Step 7: Handling Hot Keys

### Problem: Single Popular Key

```
Black Friday: Everyone buys "prod_bestseller"
10 million requests/second for same key
Single cache node can handle 1 million ops/sec
→ 10x overload
→ Cache becomes bottleneck
```

### Solution 1: Local In-Memory Cache

```
Application-level cache (inside app memory):
cache = {
  "bestseller": {product data},
  "trending_1": {...}
}

For hot keys:
1. Check app-level cache first (< 0.1ms)
2. Cache miss? Go to distributed cache (< 1ms)
3. Distributed cache miss? Hit database (10ms)

Result: Hot key served in sub-ms
```

### Solution 2: Hot Key Replication

```
Normal keys stored on 1 node
Hot keys replicated to multiple nodes

bestseller:
- Replica on Node A
- Replica on Node B
- Replica on Node C

Client randomly picks replica:
→ 1/3 probability to Node A (1/3 of traffic)
→ 1/3 probability to Node B (1/3 of traffic)
→ 1/3 probability to Node C (1/3 of traffic)

Load balanced across 3 nodes
```

### Solution 3: Probabilistic Early Recomputation

```
Key TTL = 10 minutes
Instead of waiting 10 minutes:

At 8 minutes (80% of TTL):
  With 10% probability, recompute value
  Background refresh
  
Result:
- Most requests get fresh value
- No stampede at 10-minute mark
- Gradual refresh instead of thundering herd
```

## Step 8: Cache Stampede Prevention

### Problem: Cache Stampede (Thundering Herd)

```
Popular key: "homepage_html"
Cached data: Latest HTML for homepage
TTL: 60 seconds (refreshed every minute)

Timeline:
10:00:00 - Key cached with TTL=60s
10:01:00 - TTL expires, key deleted from cache
10:01:00.001 - User A requests: CACHE MISS
10:01:00.002 - User B requests: CACHE MISS
10:01:00.003 - User C requests: CACHE MISS
...
10:01:01.000 - 10,000 simultaneous requests
             - ALL hit database
             - Database overloaded
             - Response time: 10 seconds
             - Users wait for HTML
             - Poor experience
```

### Solutions

**Option 1: Stale-While-Revalidate**
```
Key expires at 60s, but stays in cache with "stale" marker
Serve stale data immediately while recomputing in background

GET homepage_html:
  Value is stale (expired at 10:01:00)
  BUT still in cache
  Return immediately
  Trigger background refresh
  
Result: No cache miss, no stampede, users see instant response
```

**Option 2: Locking/Serialization**
```
GET homepage_html:
  Cache miss
  TRY acquire lock:
    Lock acquired → Recompute and cache
    Lock failed → Wait for other process to finish
  Return value from cache

Only 1 process recalculates, others wait briefly
```

**Option 3: Probabilistic Refresh**
```
Key TTL = 60s
At 48s (80% of TTL):
  With 10% probability, trigger background refresh
  Refresh independently, gradually
  
Result:
  Most key accesses get fresh value
  No synchronized expiration
  No stampede
```

## Step 10: Persistence & Durability

### Problem: Cache Lost on Restart

Cache contains critical data. Server restarts. All data gone!
→ Database overloaded as apps rebuild cache
→ Worst case: cascade failure

### Solution 1: RDB (Snapshot)

```
Every 60 seconds:
  Snapshot entire cache to disk
  Atomic operation
  Compressed format (~10% original size)

Recovery:
  Restart cache
  Load from disk file
  Takes 10 seconds

Tradeoff:
  Pro: Fast startup, compact size
  Con: Max 60 seconds of data loss
```

### Solution 2: AOF (Append-Only File)

```
Every write operation:
  Append to file: "SET key value"
  Every command recorded

Recovery:
  Replay all commands from file
  Full recovery, no data loss

Tradeoff:
  Pro: No data loss, durable
  Con: Large file, slow recovery (replaying all commands)
```

### Solution 3: Hybrid (RDB + AOF)

```
Both:
  RDB for fast startup
  AOF for durability

Recovery:
  1. Load RDB snapshot (fast, ~10 seconds)
  2. Replay AOF commands after snapshot (incremental)
  3. Full recovery with minimal startup time

Best of both worlds
```

## Step 11: Cluster Mode

### Redis Cluster (Distributed Sharding)

```
16,384 hash slots distributed across nodes

Cluster (6 nodes):
Node A: Slots 0-2,731
Node B: Slots 2,732-5,462
Node C: Slots 5,463-8,191
Node D: Slots 8,192-10,922
Node E: Slots 10,923-13,652
Node F: Slots 13,653-16,383

Key mapping:
1. Key → Hash value
2. Hash value % 16,384 → Slot
3. Slot → Node

Example:
Key "user:1" → CRC16 hash → 5,000 → Node B
```

### Cluster Benefits

```
Horizontal Scaling:
  Add more nodes → distribute load
  
High Availability:
  Each node has replica
  Master fails → promote replica
  
Failure Isolation:
  One node fails → other 5 still work
  Only 1/6 of data affected
```

## Step 12: Monitoring & Observability

### Key Metrics

```
Hit Ratio: % requests served from cache
→ Target: > 90%
→ If < 80%, cache too small or TTLs too short

Eviction Rate: Keys removed per second
→ If high, memory pressure, need bigger cache

Memory Usage: Current usage vs max
→ Keep < 80% to avoid thrashing

Latency (p50, p99):
→ p50: 0.5ms (median)
→ p99: 2ms (worst case for 1% of requests)
→ Alert: p99 > 5ms

Replication Lag: Master-replica sync delay
→ Should be < 100ms
→ If > 1s, replica can't keep up
```

### Alerts to Set

```
- Hit ratio < 80%
- Memory usage > 90%
- Eviction rate > 100/sec
- Replication lag > 1 second
- Node offline
- P99 latency spike
```

## Interview Cheat Sheet

**Q: Why not just use a larger database instead of caching?**  
A: Databases optimize for durability (ACID). Caches optimize for speed. Database scales vertically (bigger machine, expensive). Cache scales horizontally (add cheap nodes). For read-heavy workloads, cache is 10-100x cheaper.

**Q: What if cache has stale data?**  
A: Depends on use case. For product listings, 1-minute staleness acceptable. For prices, must be < 1 second (use event-based invalidation). For payments, must be real-time (don't cache). Choose TTL accordingly.

**Q: How to handle cache stampede?**  
A: Use probabilistic early recomputation (refresh at 80% of TTL with low probability). Or use locking (only 1 process recomputes, others wait). Or serve stale-while-revalidate (return stale value, refresh in background).

**Q: Can cache replace database?**  
A: No. Cache loses data on restart. Database is source of truth. Cache is optimization layer. Always write to database, then update cache.

**Q: What's the difference between Redis and Memcached?**  
A: Memcached is simpler, faster, distributed by default. Redis is more feature-rich (data structures, persistence, pub/sub). Memcached for simple caching. Redis for complex caching + session store + message queue.

## Full Flow (Start to End)

### Happy Path
1. Client request enters API Gateway and is authenticated/authorized.
2. Orchestrator service validates input and routing context.
3. Core service executes primary business logic and required checks.
4. Read path uses cache first; fallback goes to durable database/store.
5. Write path updates fast layer first (where applicable) and publishes async events.
6. Downstream consumers persist durable state and trigger secondary effects.
7. Response is returned to client with final status and metadata.

### Failure and Retry Paths
1. Cache miss: read from durable store, then repopulate cache.
2. Dependency timeout: retry with backoff or circuit-breaker fallback.
3. Async event failure: retry queue and dead-letter queue (DLQ) handling.
4. Duplicate request: idempotency key returns prior successful outcome.
5. Concurrent updates: version/lock conflict triggers re-read and safe retry.

### End-State Guarantees
- Low-latency user operations on the hot path.
- Durable correctness in the source-of-truth datastore.
- Eventual consistency for non-critical async side effects.
- Strict correctness at critical boundaries (commit/payment/finalization).

## Step 3: Key Algorithms & Data Structures

### Caching Algorithms

#### 1. LRU (Least Recently Used)
```
When memory full, evict least recently used item

Example:
Access order: A, B, C, A, B, D
If capacity = 3:
[A, B, D] (C evicted, A most recent)
```

**Pros:** Simple, effective
**Cons:** High overhead to track access time

#### 2. LFU (Least Frequently Used)
```
When memory full, evict least frequently used item

Example:
Access frequency: A(5), B(2), C(1), D(1)
If need space, evict C or D (lowest frequency)
```

**Pros:** Better for access patterns
**Cons:** More complex

#### 3. FIFO (First In First Out)
```
When memory full, evict oldest item

Example:
Order: A, B, C, D
If need space, evict A
```

**Pros:** Simplest, low overhead
**Cons:** Ignores access patterns

#### 4. Random Replacement
```
When memory full, evict random item
```

**Pros:** Lowest overhead
**Cons:** Worst hit ratio

## Step 4: Cache Invalidation Strategies

### Time-Based Expiration (TTL)

```
SET key value EX 3600  // Expire in 1 hour

Pros: Simple, automatic
Cons: Stale data before expiry, unused memory until TTL expires
```

### Event-Based Invalidation

```
When data changes:
1. Update database
2. Publish invalidation event to message queue
3. Cache consumer receives event
4. Delete cache entry

Pros: Always fresh data
Cons: Complex, events may be lost
```

### Write-Through

```
Write Request:
1. Write to cache
2. Wait for cache write
3. Write to database
4. Return response

Pros: Cache always consistent
Cons: Slower writes (2 writes)
```

### Write-Behind (Write-Back)

```
Write Request:
1. Write to cache
2. Return immediately
3. Asynchronously write to database

Pros: Faster writes
Cons: Risk of data loss if cache fails before DB write
```

### Write-Around

```
Write Request:
1. Write directly to database
2. Don't update cache (let it expire)
3. Next read fetches from DB and updates cache

Pros: Avoids cache pollution
Cons: First read after write is slow
```

## Step 6: Cache Partitioning (Sharding)

### Consistent Hashing

```
Problem: Adding/removing cache nodes causes rehashing

Solution: Consistent Hashing
- Place nodes on hash ring
- Map keys to closest node clockwise
- Adding node only affects ~1/n keys

Example:
Ring: 0 - 360

Nodes: A(45), B(120), C(280)
Key "user:1" hash=100 → maps to B
Key "user:2" hash=250 → maps to C
Key "user:3" hash=350 → maps to A

Adding Node D(180):
- Only keys between C(280) and D(180) need remapping
- ~25% of keys affected (not 100%)
```

### Virtual Nodes
```
Instead of 1 node per position, use multiple virtual nodes

Improves load distribution
Easier rebalancing
```

## Step 7: Replication & Fault Tolerance

### Master-Slave Replication

```
Master Node:
- Accepts reads and writes
- Replicates writes to slaves
- Single point of write

Slave Nodes:
- Read-only replicas
- Replicate data from master
- Improve read throughput
- Fallback for reads if master down
```

### Replication Process
```
1. Client writes to master
2. Master stores in buffer
3. Master sends to all slaves
4. Master acknowledges after N slaves confirm
5. Async replication for faster writes
```

### Sentinel (Monitoring & Failover)

```
┌──────────────┐
│   Sentinel   │ Monitors health
│   Sentinel   │ Detects failures
│   Sentinel   │ Triggers failover
└──────────────┘

If master fails:
1. Sentinel detects no response
2. Performs health check
3. Promotes highest-ranked slave to master
4. Updates other slaves to replicate from new master
5. Notifies clients of new master address
```

## Step 8: Handling Cache Stampede

### Problem: Cache Stampede (Thundering Herd)

```
Popular key expires
Thousands of requests simultaneously
All hit database at once
Database overload

Scenario:
- Cache key expires
- 1000 requests waiting for that key
- All hit database in parallel
- Database crashes
```

### Solutions

#### 1. Probabilistic Early Expiration
```
Key TTL = 10 minutes
With probability, refresh at 8 minutes
Rather than wait for 10 minutes and all expire together
```

#### 2. Locking Pattern
```
GET cache_key
If expired:
  TRY Lock for recomputation:
    LOCK acquired → Recompute and update cache
    LOCK failed → Wait for other process, retry GET
Return value from cache
```

#### 3. Two-Layer Expiration
```
Active TTL: 5 minutes (considered fresh)
Passive TTL: 10 minutes (served if freshness expired)

Before passive expires:
  Background job recomputes value
  All users get fresh value
```

## Step 10: Persistence Options

### AOF (Append-Only File)
```
Write every command to disk
Pros: Durable, precise
Cons: Slower writes, large files

Format:
*3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n
```

### RDB (Snapshots)
```
Periodically dump entire dataset to disk
Pros: Fast startup, compact
Cons: Data loss between snapshots, CPU intensive

Example:
- Snapshot every 60 seconds
- Or when 10,000 keys change
```

### Hybrid (AOF + RDB)
```
Use RDB for fast startup
Use AOF for durability
Pros: Best of both
Cons: More complex
```

## Step 12: Key Design Decisions & Tradeoffs

| Decision | Why? | Tradeoff |
|----------|------|----------|
| In-memory storage | Sub-millisecond latency; perfect for frequently accessed data | Data lost on restart (unless persisted); expensive (RAM costs more than disk) |
| Consistent hashing | Adding nodes doesn't require rehashing all data | More complex implementation; requires understanding hash rings |
| TTL-based expiration | Automatic cleanup; no manual invalidation needed | Stale data briefly; unused entries take up space until TTL |
| Master-slave replication | High availability; read scaling | Replication lag; potential inconsistency during failover |
| LRU eviction | Simple, effective; removes least useful data | May evict items you wanted to keep if access patterns shift |

## Interview Cheat Sheet Q&A

**Q: Why use Redis instead of just caching in the application memory (HashMap)?**  
A: Application memory is per-process. If you have 5 servers, each has its own cache. User A's request hits server 1 (cache hit), next request hits server 2 (cache miss). With Redis, all servers share one cache, so data is consistent. Also, restarting a server doesn't lose cache (it's external).

**Q: What if Redis crashes? Don't we lose everything?**  
A: By default, yes. That's why you set up replication: master Redis and slave Redis both hold the data. If master crashes, slave promotes to master (automatic via Sentinel). Also, enable AOF/RDB for persistence: write to disk periodically, so you can recover on restart.

**Q: Why not just cache everything?**  
A: RAM is expensive. You can cache $10,000 worth of SSD, or $1,000 worth of RAM. So cache only hot data (frequently accessed). Cold data (rarely accessed) goes to the database.

**Q: How do we prevent cache stampede?**  
A: When a key expires, 1000 requests come in simultaneously and all hit the database. Use probabilistic early expiration: refresh the key at 80% TTL in the background, so by the time 100% is reached, the refresh is done. Or use locking: only the first request refreshes, others wait for the result.

**Q: What if a hot key (user:1:data) is accessed 1M times/sec?**  
A: Single cache node can't handle it. Replicate that key to multiple nodes, and the client library uses consistent hashing to pick one. Or use local caching on each server for that specific key, accept eventual consistency.

**Q: Can we trust LRU eviction to always keep the right data?**  
A: Not always. If access patterns shift unexpectedly, LRU might evict something useful. Alternative: LFU (Least Frequently Used), which tracks how often items are accessed, not just when. But LFU is more complex and slower.

- ✅ Persistence options
- ✅ Cluster mode for scalability
