## What is a Constructor?

A constructor is used to:
1. **Create an instance** (object) of a class
2. **Initialize instance variables** with values

```java
class Employee {
    int employeeId;
    
    // Constructor
    Employee() {
        System.out.println("Object created!");
    }
}

Employee emp = new Employee();  // calls the constructor
```

### What Does `new` Do?

`new` does **NOT** create the object itself — it tells the JVM at runtime: **"call the constructor, not a method"** (since methods can also have the same name as the class).

```
new Employee()
 │       │
 │       └── calls the Constructor (not a method named Employee)
 └── tells JVM: "this is a constructor call"
```

## Constructor vs Method

| Feature | Constructor | Method |
|---|---|---|
| Name | **Same as class name** | Any valid name |
| Return type | **None** (implicitly returns the object) | Must declare (`int`, `void`, etc.) |
| `static` | ❌ Not allowed | ✅ Allowed |
| `final` | ❌ Not allowed | ✅ Allowed |
| `abstract` | ❌ Not allowed | ✅ Allowed |
| Purpose | Create & initialize objects | Perform tasks |

```java
class Employee {
    Employee() { ... }           // Constructor — no return type
    int Employee() { ... }       // Method — has return type (int)
}
```

> **How to distinguish?** Constructor has **no return type** at all. Methods always have one (even `void`).

## FAQ

### Why can't constructors be `final`?

`final` prevents overriding. But constructors **cannot be inherited** → cannot be overridden → `final` is meaningless.

> **Why can't constructors be inherited?** If `Employee` constructor were inherited by `Manager`, the constructor name would be `Employee` inside `Manager` class — violating the rule that constructor name = class name. So it would be treated as a method, not a constructor.

### Why can't constructors be `abstract`?

`abstract` means "child must implement." But constructors aren't inherited → child can't implement → `abstract` is meaningless.

### Why can't constructors be `static`?

`static` methods can only access `static` variables. Constructors need to **initialize instance variables** — a static constructor couldn't do this.

Also, constructor chaining uses `super()` which requires instance context (covered below).

### Can interfaces have constructors?

**No.** You can't create objects of interfaces → constructors serve no purpose for interfaces.

## Types of Constructors

### 1. Default Constructor

When you define **no constructor**, Java automatically provides one.

```java
class Calculation {
    String name;
    // No constructor written
}

// Internally, Java adds:
// Calculation() { name = null; }  ← default constructor
```

- Sets all instance variables to their **default values** (`null`, `0`, `false`, etc.)
- **Disappears** if you define any constructor manually

### 2. No-Argument Constructor

Looks similar to default, but you **explicitly define it**:

```java
class Calculation {
    String name;

    Calculation() {           // manually defined
        name = "";
    }
}
```

### 3. Parameterized Constructor

Accepts parameters to initialize instance variables:

```java
class Calculation {
    String name;

    Calculation(String employeeName) {
        this.name = employeeName;
    }
}

Calculation c = new Calculation("Shreyansh");
// name = "Shreyansh"
```

> **Important:** Once you define ANY constructor, Java no longer provides a default one.

```java
class Calculation {
    Calculation(String name) { ... }
}

Calculation c = new Calculation();  // ❌ Error! No default constructor exists
```

## Constructor Overloading

Same as method overloading — **same name, different parameters**:

```java
class Employee {
    String name;
    int employeeId;

    Employee(String name) {
        this.name = name;
        this.employeeId = 0;
    }

    Employee(String name, int id) {
        this.name = name;
        this.employeeId = id;
    }
}

Employee e1 = new Employee("Shreyansh");       // calls 1st constructor
Employee e2 = new Employee("Shreyansh", 101);  // calls 2nd constructor
```

> **No constructor overriding** — constructors cannot be inherited, so they cannot be overridden.

## Private Constructor

Making a constructor `private` prevents object creation from **outside the class**.

```java
class Calculation {
    private Calculation() { }  // private!

    public static Calculation getInstance() {
        return new Calculation();  // only this class can call it
    }
}

// From outside:
// new Calculation();                    ❌ Error!
Calculation c = Calculation.getInstance(); // ✅ OK
```

> **Use case:** Singleton Pattern — control object creation, ensure only one instance.

## Constructor Chaining

Calling one constructor from another. Two mechanisms: `this()` and `super()`.

### Chaining with `this()` (Within Same Class)

```java
class Employee {
    String name;
    int employeeId;

    Employee(int id) {
        this("SJ", id);           // calls the 2-param constructor
    }

    Employee(String name, int id) {
        this.name = name;
        this.employeeId = id;
    }
}

new Employee(10);
// Flow: Employee(10) → this("SJ", 10) → Employee("SJ", 10)
```

```
  Constructor Chaining with this()
  
  Employee(10) ──this("SJ", 10)──▶ Employee("SJ", 10)
                                     name = "SJ"
                                     employeeId = 10
```

### Chaining with `super()` (Parent → Child)

When creating a child object, Java **always calls the parent constructor first**.

```java
class Person {
    Person() {
        System.out.println("Person constructor");
    }
}

class Manager extends Person {
    Manager() {
        // super(); ← hidden! Java adds this automatically
        System.out.println("Manager constructor");
    }
}

new Manager();
// Output:
// Person constructor     ← parent first!
// Manager constructor    ← then child
```

```
  new Manager()
       │
       ▼
  Manager() {
      super();   ← automatically added by Java (hidden)
      ...
  }
       │
       ▼
  Person() {
      ...        ← parent constructor runs first
  }
```

### Parameterized `super()`

If parent has a **parameterized constructor** (no no-arg), you must explicitly call `super(args)`:

```java
class Person {
    int employeeId;

    Person(int id) {               // parameterized — no default!
        this.employeeId = id;
    }
}

class Manager extends Person {
    int age;

    Manager(int id, int age) {
        super(id);                 // MUST explicitly call parent
        this.age = age;
    }
}

new Manager(101, 30);
// Person constructor: employeeId = 101
// Manager constructor: age = 30
```

| Scenario | `super()` needed? |
|---|---|
| Parent has no-arg constructor | Auto-added (hidden) |
| Parent has only parameterized constructor | Must call `super(args)` explicitly |

## Summary

| Concept | Key Point |
|---|---|
| **Constructor** | Creates instance + initializes variables |
| **`new` keyword** | Tells JVM to call constructor, not method |
| **No return type** | Differentiates constructor from method |
| **Cannot be** | `static`, `final`, `abstract` |
| **Default** | Auto-provided when no constructor is defined |
| **No-Arg** | Manually defined, looks like default |
| **Parameterized** | Accepts arguments; kills default constructor |
| **Overloading** | Same name, different params ✅ |
| **Overriding** | Not possible (not inherited) ❌ |
| **Private** | Controls object creation (Singleton pattern) |
| **`this()`** | Chain constructors within same class |
| **`super()`** | Chain to parent constructor (auto-added for no-arg) |
