## 1. Singleton Class

**Purpose:** Ensure only **ONE instance** of a class exists throughout the application.

**Use case:** Database connections, configuration managers, logging вҖ” where multiple objects are wasteful or harmful.

```
  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
  в”Ӯ         Singleton Class Rules            в”Ӯ
  в”Ӯ  1. Private constructor                  в”Ӯ
  в”Ӯ  2. Private static instance              в”Ӯ
  в”Ӯ  3. Public static getInstance() method   в”Ӯ
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### 1. Eager Initialization

Object is created **at class loading time** вҖ” before anyone requests it.

```java
public class DbConnection {
    // в‘  Private static instance вҖ” created eagerly
    private static final DbConnection conObj = new DbConnection();

    // в‘Ў Private constructor вҖ” no external instantiation
    private DbConnection() { }

    // в‘ў Public static getter
    public static DbConnection getInstance() {
        return conObj;
    }
}
```

| Pros | Cons |
|---|---|
| Simple, thread-safe | Object created even if **never used** вҶ’ wastes memory |

---

### 2. Lazy Initialization

Object created **only when first requested**.

```java
public class DbConnection {
    private static DbConnection conObj;  // null initially

    private DbConnection() { }

    public static DbConnection getInstance() {
        if (conObj == null) {           // create only on first call
            conObj = new DbConnection();
        }
        return conObj;
    }
}
```

| Pros | Cons |
|---|---|
| No wasted memory | вқҢ **NOT thread-safe** вҖ” two threads can create two objects simultaneously |

```
  Thread 1: conObj == null? вҶ’ YES вҶ’ creating...
  Thread 2: conObj == null? вҶ’ YES вҶ’ creating...  вҶҗ RACE CONDITION!
  Result: TWO objects created!
```

---

### 3. Synchronized Method

Add `synchronized` to prevent concurrent access.

```java
public static synchronized DbConnection getInstance() {
    if (conObj == null) {
        conObj = new DbConnection();
    }
    return conObj;
}
```

| Pros | Cons |
|---|---|
| Thread-safe | вқҢ **Slow** вҖ” lock/unlock on EVERY call, even after object exists |

---

### 4. Double-Checked Locking

Synchronize **only the critical section**, check twice.

```java
public class DbConnection {
    private static volatile DbConnection conObj;  // volatile!

    private DbConnection() { }

    public static DbConnection getInstance() {
        if (conObj == null) {                     // Check 1 (no lock)
            synchronized (DbConnection.class) {
                if (conObj == null) {              // Check 2 (with lock)
                    conObj = new DbConnection();
                }
            }
        }
        return conObj;
    }
}
```

```
  Thread 1                    Thread 2
  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ                    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  Check 1: null? YES          Check 1: null? YES
  Acquires lock в”Җв”Җв”Җв”Җв”Җв”Җв”җ       Waits for lock...
  Check 2: null? YES  в”Ӯ
  Creates object      в”Ӯ
  Releases lock в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                              Acquires lock
                              Check 2: null? NO вҶҗ already created!
                              Returns existing object
```

### Why `volatile`?

```
  в”Ӯ L1 Cache  в”Ӯ                  в”Ӯ L1 Cache  в”Ӯ
  в”Ӯ conObj=ref в”Ӯ  вҶҗ not synced   в”Ӯ conObj=nullв”Ӯ вҶҗ stale!

        в”Ӯ                              в”Ӯ
  в”Ңв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”җ
```

> `volatile` forces all reads/writes to go through **main memory**, not CPU cache вҖ” preventing stale data.

| Pros | Cons |
|---|---|
| Thread-safe, fast after first call | Slightly slow due to `volatile` (bypasses cache) |

### 5. Bill Pugh Solution

Uses a **static nested class** вҖ” loaded only when referenced (lazy + thread-safe without `synchronized`/`volatile`).

```java
public class DbConnection {
    private DbConnection() { }

    // Static nested class вҖ” NOT loaded until getInstance() is called
    private static class Helper {
        private static final DbConnection INSTANCE = new DbConnection();
    }

    public static DbConnection getInstance() {
        return Helper.INSTANCE;
    }
}
```

**Why it works:**
- Nested classes are **not loaded** until they are referenced
- First call to `getInstance()` вҶ’ loads `Helper` вҶ’ creates `INSTANCE`
- JVM guarantees class loading is thread-safe

| Pros | Cons |
|---|---|
| Lazy, thread-safe, fast, no `synchronized`/`volatile` | вҖ” |

> **Most recommended approach** for Singleton in Java.

### 6. Enum Singleton

Enums are inherently singleton per JVM.

```java
public enum DbConnection {
    INSTANCE;

    public void connect() {
        System.out.println("Connected to DB");
    }
}

// Usage
DbConnection.INSTANCE.connect();
```

**Why it works:**
- Enum constructors are **always private**
- JVM guarantees **one instance per enum constant**
- Thread-safe by default

| Pros | Cons |
|---|---|
| Simplest, thread-safe, serialization-safe | Cannot use lazy loading |

## 2. Immutable Class

A class whose **state cannot be changed** after object creation (like `String`).

### Rules to Create an Immutable Class

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ              Immutable Class Checklist                        в”Ӯ
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ  1. Declare class as final (no subclass can alter behavior)  в”Ӯ
в”Ӯ  2. All fields private + final                               в”Ӯ
в”Ӯ  3. Initialize fields ONLY via constructor (one-time)        в”Ӯ
в”Ӯ  4. NO setter methods                                        в”Ӯ
в”Ӯ  5. Getter for collections вҶ’ return a COPY, not original     в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Implementation

```java
public final class ImmutablePerson {
    private final String name;
    private final List<String> petNames;

    public ImmutablePerson(String name, List<String> petNames) {
        this.name = name;
        this.petNames = new ArrayList<>(petNames);  // defensive copy in
    }

    public String getName() {
        return name;  // String is already immutable вҖ” safe
    }

    public List<String> getPetNames() {
        return new ArrayList<>(petNames);  // return COPY вҖ” critical!
    }
}
```

### Why Return a Copy for Collections?

`final` on a collection means the **reference can't change** вҖ” but the **contents still can**!

```java
ImmutablePerson p = new ImmutablePerson("SJ", List.of("SJ", "PJ"));

List<String> names = p.getPetNames();   // gets a COPY
names.add("Hello");                      // modifies the COPY only

System.out.println(p.getPetNames());     // still ["SJ", "PJ"] вң…
```

```
  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җ final List<String> petNames в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
  в”Ӯ                                            в”Ӯ
  в”Ӯ  final = reference cannot change           в”Ӯ
  в”Ӯ  petNames в”Җв”Җв–¶ [SJ, PJ]  (this is fixed)   в”Ӯ
  в”Ӯ                                            в”Ӯ
  в”Ӯ  BUT contents CAN change:                  в”Ӯ
  в”Ӯ  petNames.add("X") вҶ’ [SJ, PJ, X]  вқҢ bad! в”Ӯ
  в”Ӯ                                            в”Ӯ
  в”Ӯ  Solution: return new copy in getter       в”Ӯ
  в”Ӯ  вҶ’ caller gets separate list               в”Ӯ
  в”Ӯ  вҶ’ original stays untouched        вң…      в”Ӯ
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Why Each Rule Matters

| Rule | Without It |
|---|---|
| `final` class | Subclass could add setters or override methods |
| `private final` fields | External code could modify fields |
| Constructor-only init | No way to change values after creation |
| No setters | State modification blocked |
| Return copy in getter | Prevents mutation via returned collection reference |

| Concept | Key Point |
