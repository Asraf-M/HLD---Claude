## Process vs Thread

### Process

A **process** is an instance of a program that is being executed.

```
  javac Main.java        →  bytecode generated
  java Main              →  JVM starts a new PROCESS
                              ↓
                          OS allocates resources (memory, etc.)
                              ↓
                          Process has its own heap, code segment, etc.
```

| Property | Detail |
|---|---|
| Created when | You execute a program (`java Main`) |
| Resources | Own heap, own memory — **not shared** with other processes |
| Independence | Two processes are **completely independent** |
| JVM instance | Each process gets a **new JVM instance** |

### Thread

A **thread** is the smallest sequence of instructions that are executed by CPU independently.

```
  Process
  ┌──────────────────────────────────────┐
  │  Thread 1 (main)   ← auto-created   │
  │  Thread 2          ← user-created    │
  │  Thread 3          ← user-created    │
  │                                      │
  │  Shared: Heap, Code Segment, Data    │
  └──────────────────────────────────────┘
```

| Property | Detail |
|---|---|
| Also called | Lightweight process |
| Initial thread | `main` thread — created automatically when process starts |
| Multiple threads | Created from `main` to perform tasks concurrently |

```java
public class Main {
    public static void main(String[] args) {
        System.out.println(Thread.currentThread().getName());  // "main"
    }
}
```

## JVM Memory Model Per Process

When a process is created, a **new JVM instance** is allocated to it with all memory areas.

```
  Process 1                           Process 2
  ┌────────────────────┐              ┌────────────────────┐
  │  JVM Instance 1    │              │  JVM Instance 2    │
  │  ┌──────────────┐  │              │  ┌──────────────┐  │
  │  │ Heap         │  │              │  │ Heap         │  │
  │  │ Code Segment │  │              │  │ Code Segment │  │
  │  │ Data Segment │  │              │  │ Data Segment │  │
  │  │ Registers    │  │              │  │ Registers    │  │
  │  │ Stack        │  │              │  │ Stack        │  │
  │  │ PC (Counter) │  │              │  │ PC (Counter) │  │
  │  └──────────────┘  │              │  └──────────────┘  │
  └────────────────────┘              └────────────────────┘
       NOT shared ←────────────────────→ NOT shared
```

### Configuring Heap Size Per Process

```bash
java -Xms256m -Xmx2g Main
#     ↑ min heap     ↑ max heap
```

| Flag | Purpose |
|---|---|
| `-Xms` | Initial (minimum) heap size |
| `-Xmx` | Maximum heap size |

> If process needs more than max heap → `OutOfMemoryError`, even if total JVM heap is larger.

## Shared vs Thread-Local Memory

```
  JVM Instance (one process)
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │   SHARED among all threads                      │
  │   ┌──────────────┬──────────────┬────────────┐  │
  │   │ Code Segment │ Data Segment │    Heap    │  │
  │   │ (read-only)  │ (read/write) │ (read/write)│ │
  │   └──────────────┴──────────────┴────────────┘  │
  │                                                 │
  │   LOCAL to each thread                          │
  │   ┌──────────┐  ┌──────────┐  ┌──────────┐     │
  │   │ Thread 1 │  │ Thread 2 │  │ Thread 3 │     │
  │   │ Stack    │  │ Stack    │  │ Stack    │     │
  │   │ Register │  │ Register │  │ Register │     │
  │   │ PC       │  │ PC       │  │ PC       │     │
  │   └──────────┘  └──────────┘  └──────────┘     │
  └─────────────────────────────────────────────────┘
```

## Code Segment, Data Segment, Heap

| Memory Area | Stores | Shared? | Writable? |
|---|---|---|---|
| **Code Segment** | Compiled bytecode / machine code | ✅ Shared | ❌ Read-only |
| **Data Segment** | Global & static variables | ✅ Shared | ✅ Read/Write — needs synchronization |
| **Heap** | Objects created with `new` | ✅ Shared (within process) | ✅ Read/Write — needs synchronization |

> Heap is shared among threads of the **same process**, but **NOT** across processes.

## Register, Stack, Program Counter

Each thread has its **own** copy of these — not shared.

| Memory Area | Purpose |
|---|---|
| **Stack** | Method calls, local variables, method frame |
| **Register** | Intermediate values during execution; helps JIT optimizer; critical for context switching |
| **Program Counter (PC)** | Points to the current instruction in code segment that the thread must execute next; increments after each instruction |

```
  Code Segment (machine code):
  ┌─────────────────────────────────────────────┐
  │ instruction 0 │ instruction 1 │ instr 2 ... │
  └───────┬───────────────┬─────────────────────┘
          ↑               ↑
       Thread 1 PC     Thread 2 PC
```

## Execution Flow & Context Switching

### Full Flow: From Source to CPU

```
  1. javac Main.java  →  bytecode
  2. java Main        →  process created + JVM instance
  3. JVM interpreter/JIT → bytecode → machine code → stored in Code Segment
  4. JVM creates threads → each gets Stack, Register, PC
  5. PC points to starting instruction in Code Segment
  6. OS scheduler assigns thread to CPU core
  7. CPU reads instruction from Code Segment via PC, executes it
```

### Context Switching

When CPU has fewer cores than threads, it **time-slices** between threads.

```
  CPU Core 1 (single core, 3 threads):

  Time 1: Thread-1 runs (50% done) → saves state to Register
  Time 2: Thread-2 runs (70% done) → saves state to Register
  Time 3: Thread-3 runs (30% done) → saves state to Register
  Time 4: Thread-1 resumes from Register → continues from 50%
  ...
```

| Concept | Detail |
|---|---|
| **Context Switch** | CPU saves current thread's state to its Register, loads next thread's state |
| **Register's role** | Stores intermediate results — enables resume from where thread left off |
| **More threads than cores** | Context switching occurs — looks parallel but isn't truly parallel |
| **Threads ≤ cores** | True parallelism — no context switching needed |

> **CPU idle during context switch** — saving old data, loading new thread. More threads → more switching → wasted CPU time.

## Multithreading vs Multitasking

| Feature | Multitasking | Multithreading |
|---|---|---|
| **Unit** | Multiple processes | Multiple threads within one process |
| **Resource sharing** | ❌ No — each process has own memory | ✅ Yes — threads share heap, code, data |
| **Example** | Running Chrome + IntelliJ at same time | Multiple threads in a single Java app |
| **Communication** | Expensive (IPC) | Cheap (shared memory) |

### Benefits of Multithreading

| Benefit | Detail |
|---|---|
| Performance | Task parallelism — divide work across threads |
| Responsiveness | Faster response to users |
| Resource sharing | Threads share memory — less overhead than processes |

### Challenges

| Challenge | Detail |
|---|---|
| Concurrency issues | Deadlocks, data inconsistency on shared resources |
| Synchronization overhead | Locks, synchronized blocks |
| Debugging difficulty | Hard to reproduce & test concurrent bugs |

## Summary

| Concept | Key Point |
|---|---|
| **Process** | Instance of executing program — own JVM instance, own memory |
| **Thread** | Smallest CPU execution unit — lightweight, shares process resources |
| **Shared memory** | Code Segment (read-only), Data Segment, Heap |
| **Thread-local** | Stack, Register, Program Counter |
| **Context switching** | CPU swaps threads — uses Register to save/restore state |
| **True parallelism** | Only when threads ≤ CPU cores |
| **Multitasking** | Multiple processes — no shared resources |
| **Multithreading** | Multiple threads in one process — shared resources |
