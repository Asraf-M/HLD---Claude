## Why Generics?

Without generics, you use `Object` (parent of all) — but this forces **type casting** and loses type safety.

```java
// Without Generics — uses Object
class Print {
    Object value;
    void setValue(Object v) { this.value = v; }
    Object getValue() { return this.value; }
}

Print p = new Print();
p.setValue(1);
p.setValue("hello");            // anything goes — no control!

int num = (int) p.getValue();  // ❌ must typecast — error-prone
// What if it's a String? → ClassCastException at runtime!
```

> **Generics** solve this by enforcing types at **compile time**, eliminating casting and catching errors early.

## Generic Class

Use the **diamond syntax** `<T>` after the class name. `T` is a **type parameter** — a placeholder replaced with an actual type at object creation.

```java
class Print<T> {
    T value;
    
    void setValue(T v) { this.value = v; }
    T getValue() { return this.value; }
}

// At object creation — specify the type
Print<Integer> intPrint = new Print<>();
intPrint.setValue(1);            // ✅ only Integer allowed
intPrint.setValue("hello");      // ❌ compile-time error!
int result = intPrint.getValue(); // ✅ no casting needed!

Print<String> strPrint = new Print<>();
strPrint.setValue("hello");      // ✅ only String allowed
```

> `T` can be replaced with **any non-primitive** type (not `int`, `float` — use `Integer`, `Float` wrappers).

## Inheritance with Generic Classes

### Non-Generic Subclass

Must specify the type **at extend time**:

```java
class ColorPrint extends Print<String> { }  // locked to String

ColorPrint cp = new ColorPrint();
cp.setValue("Red");  // ✅ String only
```

### Generic Subclass

Keeps generics — type decided at **object creation**:

```java
class ColorPrint<T> extends Print<T> { }  // still generic

ColorPrint<String> cp = new ColorPrint<>();
cp.setValue("Red");  // ✅
```

## Multiple Type Parameters

```java
class Pair<K, V> {
    K key;
    V value;

    void put(K key, V value) {
        this.key = key;
        this.value = value;
    }
}

Pair<String, Integer> pair = new Pair<>();
pair.put("age", 25);    // K=String, V=Integer
```

> You can have as many type parameters as needed: `<T1, T2, T3, ...>`

## Generic Methods

Make only **one method** generic (not the whole class):

```java
class Utility {
    // Generic method — type parameter before return type
    <T> void printValue(T value) {
        System.out.println(value);
    }
}

Utility u = new Utility();
u.printValue(42);          // T = Integer
u.printValue("hello");     // T = String
u.printValue(new Bus());   // T = Bus
```

### Syntax

```
<T> returnType methodName(T param) { ... }
 ↑                         ↑
 type parameter             usage
 (before return type)
```

> Type parameter scope is limited to **that method only**.

## Raw Types

Creating a generic class object **without specifying the type parameter**:

```java
Print rawObj = new Print();       // raw type — no <T> specified
rawObj.setValue(1);                // accepts anything
rawObj.setValue("hello");          // no compile error

// Internally: compiler treats T as Object
```

> **Avoid raw types** — they bypass compile-time type safety.

## Bounded Generics

Restrict **which types** can be used as type parameters.

### Upper Bound (`extends`)

Accept the specified class **and all its subclasses**:

```java
class Print<T extends Number> {
    T value;
    void setValue(T v) { this.value = v; }
}

Print<Integer> p1 = new Print<>();  // ✅ Integer extends Number
Print<Double> p2 = new Print<>();   // ✅ Double extends Number
Print<String> p3 = new Print<>();   // ❌ String is NOT a Number
```

```
          Object
            │
          Number    ← upper bound
         /   |   \
   Integer Double Float  ← all allowed
   
   String ← NOT under Number → rejected
```

> **Note:** In generics, `extends` is used for both classes AND interfaces (don't use `implements`).

### Multi-Bound

Require **a class AND one or more interfaces**:

```java
class Print<T extends ParentClass & Interface1 & Interface2> { ... }
```

| Position | Must be |
|---|---|
| First | Class (concrete/abstract) |
| After `&` | Interfaces (any number) |

```java
// Only accepts types that extend ParentClass AND implement both interfaces
Print<A> p = new Print<>();

// A must: extend ParentClass + implement Interface1 + implement Interface2
```

## Wildcards

Used with **collections** to add flexibility to method parameters.

### The Problem

```java
List<Vehicle> vehicleList = new ArrayList<>();
List<Bus> busList = new ArrayList<>();

vehicleList = busList;  // ❌ INVALID!
// List<Bus> is NOT a subtype of List<Vehicle>
```

> Even though `Bus extends Vehicle`, `List<Bus>` does NOT extend `List<Vehicle>`.

### Upper Bound Wildcard (`? extends`)

Accept the class **and all subclasses**:

```java
void printValues(List<? extends Vehicle> list) { ... }

printValues(vehicleList);  // ✅ List<Vehicle>
printValues(busList);      // ✅ List<Bus> — Bus extends Vehicle
```

### Lower Bound Wildcard (`? super`)

Accept the class **and all superclasses**:

```java
void printValues(List<? super Vehicle> list) { ... }

printValues(vehicleList);  // ✅ List<Vehicle>
printValues(objectList);   // ✅ List<Object> — Object is super of Vehicle
printValues(busList);      // ❌ Bus is below Vehicle
```

### Unbounded Wildcard (`?`)

Accept **any type** — useful when working only with `Object` methods:

```java
void printValues(List<?> list) { ... }

printValues(intList);      // ✅ List<Integer>
printValues(strList);      // ✅ List<String>
printValues(anyList);      // ✅ anything
```

### Wildcards vs Generic Type Parameters

| Feature | Wildcard (`?`) | Generic Type (`T`) |
|---|---|---|
| Multiple params same type | ❌ No enforcement | ✅ `T` ensures same type |
| Lower bound (`super`) | ✅ Supported | ❌ Not supported |
| Multiple type params | ❌ Only one `?` | ✅ `<K, V, T>` |
| Flexibility | More flexible | More restrictive/safe |

```java
// Wildcard: source and dest can be DIFFERENT subtypes of Number
void compute(List<? extends Number> src, List<? extends Number> dest) { }
compute(integerList, floatList);  // ✅ different types OK

// Generic: source and dest MUST be SAME type
<T extends Number> void compute(List<T> src, List<T> dest) { }
compute(integerList, floatList);  // ❌ different types rejected
```

## Type Erasure

At **compile time**, all generic type information is **erased** and replaced with concrete types in the bytecode.

| Source Code | Bytecode (after erasure) |
|---|---|
| `class Print<T> { T value; }` | `class Print { Object value; }` |
| `class Print<T extends Number> { T value; }` | `class Print { Number value; }` |
| `<T> void print(T v)` | `void print(Object v)` |
| `<T extends Bus> void print(T v)` | `void print(Bus v)` |

> Generics are a **compile-time** feature. At runtime, the JVM sees no generics — only raw types.

## Summary

| Concept | Key Point |
|---|---|
| **Why Generics** | Type safety at compile time, no casting |
| **`<T>`** | Type parameter — placeholder for actual type |
| **Non-primitive only** | `int` ❌, `Integer` ✅ |
| **Generic Class** | `class Name<T> { }` |
| **Generic Method** | `<T> returnType name(T param)` |
| **Raw Type** | No type param → treated as `Object` (avoid!) |
| **Upper Bound** | `T extends X` → X and subclasses |
| **Multi-Bound** | `T extends Class & Interface1 & Interface2` |
| **`? extends`** | Upper bound wildcard |
| **`? super`** | Lower bound wildcard |
| **`?`** | Unbounded wildcard |
| **Type Erasure** | Generics removed at compile time → replaced with bounds/Object |
