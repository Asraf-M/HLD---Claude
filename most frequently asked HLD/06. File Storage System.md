# System Design: File Storage System (Beginner-Friendly Guide)

---

## What Are We Building?

Think of Amazon S3 or Google Cloud Storage — a massively scalable, reliable object storage service where users can:
- Upload files of any size (KB to TB)
- Download files from anywhere globally
- Organize files into "buckets"
- Access files programmatically via API
- Version files (keep history)
- Organize files with lifecycle policies (auto-delete old versions)
- Control who can access files

The interesting engineering challenges hidden in this simple flow:
- **Scale** — exabytes (1 million terabytes) of data across millions of users; can't store on single machine
- **Durability** — "11 nines" (99.999999999% uptime) means at most 31 seconds of data loss per year across the entire planet
- **Latency** — user in Tokyo should download file as fast as user in New York (requires global CDN)
- **Cost optimization** — storing old data on expensive SSDs wastes billions; move to cheaper storage (Glacier) automatically
- **Replication** — if data center burns down, data must exist in another country
- **Consistency** — user uploads file, should be readable immediately by anyone who knows the URL
- **Metadata** — index millions of files per bucket so users can search/list quickly

---

## Scope: Beyond Upload/Download

This is **NOT** just an upload/download system. This design covers a complete distributed object storage platform with:

**Core Operations:**
- Upload (multipart, chunking, parallel transfers)
- Download (streaming, range requests, CDN integration)
- List/search (bucket browsing, full-text search)
- Delete with versioning support

**Storage & Durability:**
- Distributed storage across 3+ data centers
- 3-way replication for hot data (fast access, high durability)
- Erasure coding (4+2) for warm/cold data (cost optimization)
- 11-nines durability (99.999999999%) = 31 seconds max data loss per year
- Immutable backups and disaster recovery

**Metadata & Indexing:**
- PostgreSQL for structured metadata (objects, buckets, versions)
- Elasticsearch for full-text search (by filename, tags, content-type)
- Redis caching for frequently accessed object metadata (1-7200s TTL)
- Secondary indexes for fast queries by owner, timestamp, size

**Advanced Features:**
- **Versioning:** Keep full history, restore any previous version
- **Lifecycle Policies:** Auto-transition cold (Standard → IA → Glacier → Archive)
- **Access Control:** Bucket policies, object ACLs, presigned URLs (time-limited sharing)
- **Encryption:** At-rest (AES-256) and in-transit (TLS)
- **Multi-region failover:** RTO < 1 minute, RPO = 0 (zero data loss)

**Infrastructure & Scale:**
- Global CDN (1000+ edge locations worldwide)
- Tiered storage (hot SSD $0.023/GB → warm HDD $0.0125/GB → cold Glacier $0.004/GB)
- 1+ exabyte capacity, 1M concurrent uploads, 10M concurrent downloads
- 100+ billion metadata records queryable instantly
- Automatic cost optimization (older data moves to cheaper tiers)

## Step 3: Database Design — Why Hybrid?

We need **5 different databases** because each has different strengths. No single database is optimal for all use cases.

| Requirement | Solution | Why |
|-------------|----------|-----|
| Object metadata (queryable) | PostgreSQL (SQL) | ACID guarantees, complex queries, indexing |
| Versioning & history | PostgreSQL + time-series DB | Query by date, retention policies, compliance |
| Geographic replication | Distributed SQL (multi-master) | Sync across regions, failover, consistency |
| Fast full-text search | Elasticsearch | Full-text queries (filename, tags), aggregations |
| Hot-object caching | Redis | Sub-millisecond lookups, LRU eviction |
| Actual file data | Distributed file system | Petabyte scale, replication, erasure coding |

### 3. Redis: Hot-Object Caching

**Purpose:** Cache frequently accessed object metadata to avoid PostgreSQL queries (< 1ms latency).

**Cache Keys:**

```redis
-- Object metadata cache (1-2 hour TTL)
SET "object:bucket:1:2026-06-18/database-dump.sql" \
  '{
    "version_id": "v847291048372",
    "size": 2147483648,
    "etag": "7d8a3e9f2c1b4a6d5e9f8c2b1a3e5d7f",
    "storage_class": "STANDARD",
    "location": ["node-12:us-east-1", "node-31:eu-west-1"],
    "created_at": "2026-06-18T10:35:00Z"
  }' \
  EX 7200

-- Bucket listing page cache (10 min TTL)
SET "bucket:listing:1:page:1" \
  '[
    {"key": "2026-06-18/database-dump.sql", "size": 2147483648, "modified": "2026-06-18T10:35:00Z"},
    {"key": "2026-06-17/database-dump.sql", "size": 2147483648, "modified": "2026-06-17T10:35:00Z"}
  ]' \
  EX 600

-- Owner's bucket list (1 hour TTL)
SET "user:456:buckets" \
  '[
    {"name": "my-company-backups", "objects_count": 1243, "storage_used": 2147483648},
    {"name": "photos-public", "objects_count": 5432, "storage_used": 536870912}
  ]' \
  EX 3600

-- Hot object counter (for analytics)
INCR "downloads:bucket:1:object:database-dump.sql"
SET "downloads:bucket:1:object:database-dump.sql" 1543 EX 86400
```

**Tradeoffs:**
- ✅ Sub-millisecond reads (< 1ms), reduces load on PostgreSQL, simple key-value
- ❌ Memory-limited (can't cache all 100B objects), eviction on memory pressure, eventual consistency (stale cache OK)
- ⚠️ **Cache invalidation pattern**: On write, delete cache key; next read from DB repopulates cache

### 5. Distributed File System: Actual File Data

**Purpose:** Store the actual binary file data across multiple nodes with replication/erasure coding.

**Important clarification (Distributed FS vs S3-like service):**

Distributed FS is the **data layer**, not the full product by itself.

- SSD/HDD/Glacier are storage media/tier choices.
- Distributed FS handles chunk placement, replication/erasure coding, and recovery.
- S3-like behavior (buckets, object keys, ACLs, versioning, lifecycle, presigned URLs) is provided by upper layers.

Think in 3 layers:

1. **Object/API layer (what users see):** Buckets, object keys, versioning, access control, lifecycle policies.
2. **Metadata/control layer:** Tracks object-to-chunk map, ownership, encryption metadata, policy state.
3. **Data layer (Distributed FS):** Stores chunk bytes on nodes across regions/racks using replication or erasure coding.

So this design is not "just SSD/HDD/Glacier"; it is a complete S3-like object storage system built on top of distributed storage media.

**How data is stored across different servers:**

1. Upload service splits the file into fixed-size chunks (for example, 500 MB each).
2. Placement service chooses nodes using rules:
  - Do not place two chunks of same stripe in same rack.
  - Spread chunks across regions for disaster tolerance.
  - Prefer nodes with free space + lower load.
3. For erasure coding 4+2, system writes 4 data chunks + 2 parity chunks to 6 different nodes.
4. Each node stores only its chunk and checksum; no node needs full file copy.

**Conceptual Layout:**

```
Object: "2026-06-18/database-dump.sql" (2 GB)

Storage layout (Erasure coding 4+2):
├── Data Chunk-0 (500 MB) → Storage Node 12 (us-east-1, Rack A)
├── Data Chunk-1 (500 MB) → Storage Node 18 (us-east-1, Rack B)
├── Data Chunk-2 (500 MB) → Storage Node 5 (eu-west-1, Rack A)
├── Data Chunk-3 (500 MB) → Storage Node 22 (ap-south-1, Rack A)
├── Parity Chunk-0 (500 MB) → Storage Node 31 (eu-west-1, Rack B)
└── Parity Chunk-1 (500 MB) → Storage Node 47 (ap-south-1, Rack B)

Recovery: Need any 4 of 6 chunks to reconstruct
Cost: 1.5x storage (6 chunks / 4 original)
```

**How we track where each chunk lives (metadata tracking):**

The system keeps a chunk-manifest record in metadata DB.

**Object Manifest (one row per object version):**

| object_id | version_id | coding_scheme | stripe_count | chunk_size_mb | min_chunks_to_read | status |
|-----------|------------|---------------|--------------|---------------|--------------------|--------|
| 1 | v847291048372 | EC-4+2 | 1 | 500 | 4 | ACTIVE |

**Chunk Location Map (one row per chunk):**

| object_id | version_id | stripe_id | chunk_id | chunk_type | node_id | region | rack | checksum | state |
|-----------|------------|-----------|----------|------------|---------|--------|------|----------|-------|
| 1 | v847291048372 | 0 | 0 | DATA | 12 | us-east-1 | A | sha256:aa11 | HEALTHY |
| 1 | v847291048372 | 0 | 1 | DATA | 18 | us-east-1 | B | sha256:bb22 | HEALTHY |
| 1 | v847291048372 | 0 | 2 | DATA | 5 | eu-west-1 | A | sha256:cc33 | HEALTHY |
| 1 | v847291048372 | 0 | 3 | DATA | 22 | ap-south-1 | A | sha256:dd44 | HEALTHY |
| 1 | v847291048372 | 0 | 4 | PARITY | 31 | eu-west-1 | B | sha256:ee55 | HEALTHY |
| 1 | v847291048372 | 0 | 5 | PARITY | 47 | ap-south-1 | B | sha256:ff66 | HEALTHY |

**How read/recovery uses tracking data:**

1. Client asks for object.
2. Metadata service reads manifest + chunk location map.
3. Downloader chooses nearest healthy chunks (any 4 of 6 for EC-4+2).
4. If a chunk is missing/unhealthy, parity reconstruction is triggered.
5. Repair job writes rebuilt chunk to a new node and updates chunk map.

**Failure example (Node 22 down):**
- Chunk `chunk_id=3` unavailable.
- Reader fetches chunks 0,1,2,4 and reconstructs missing data chunk 3.
- Background repair writes rebuilt chunk 3 to Node 53.
- Metadata row is updated: `node_id=53`, `state=HEALTHY`, `last_repaired_at=<timestamp>`.

**Tradeoffs:**
- ✅ Petabyte-scale, fault-tolerant (survive node/rack/DC failure), cost-efficient (erasure coding)
- ❌ Complex to implement (chunk coordination, parity calc), slower writes (parity overhead), complex recovery
- ⚠️ **Read path**: Fetch any 4 chunks from nearest nodes in parallel, reconstruct if needed

### Hybrid Database Summary: Why We Need All 5?

| Database | Use Case | Example Query |
|----------|----------|---|
| **PostgreSQL** | Metadata CRUD, strong consistency, source of truth | "Get all versions of object X in bucket Y, ordered by date" |
| **NoSQL (Optional: DynamoDB/Cassandra)** | Massive key-value metadata at extreme scale | "Get object metadata by (bucket_id, object_key) in <10ms" |
| **Elasticsearch** | Full-text search, aggregations, analytics | "Find all SQL files > 1 GB created in last 7 days" |
| **Redis** | Hot object cache, fast reads, analytics counters | "Get metadata for /2026-06-18/database-dump.sql" |
| **Time-Series DB** | Versioning history, lifecycle transitions, audit | "Find objects created 30 days ago eligible for warm tier" |
| **Distributed FS** | Binary data storage, replication, durability | "Retrieve chunks for 2 GB object from nodes 12,18,5,22" |

**Why PostgreSQL as source of truth instead of NoSQL in this design?**
- We need strict relational guarantees for metadata workflows (ACL + versioning + lifecycle state updates together).
- SQL constraints and transactional updates reduce metadata corruption risk.
- Operationally simpler for interview-scale explanations when strong consistency is a top requirement.

#### Why Strict Relational Guarantees Matter

Metadata is the system's truth layer. If metadata is wrong, data may exist but users can't safely find/use it. Here's why strict consistency is required:

**1. One user action updates multiple metadata records together**

Example: Finalize an object upload
```
BEGIN TRANSACTION
  INSERT INTO objects (object_id, bucket_id, object_key, version_id, size, ...)
  UPDATE objects SET is_current_version=true WHERE version_id=$1
  UPDATE buckets SET storage_used_bytes = storage_used_bytes + 2147483648
  INSERT INTO object_versions (object_id, version_id, size, created_at, ...)
COMMIT
```

If any step fails after others succeed → metadata corruption. Transactions prevent this.

**2. Invariants must never break**

| Invariant | Why it matters | Impact of violation |
|-----------|---|---|
| Only one "current version" per object key | Applications assume latest version is unique | Multiple "latest" versions cause conflicts |
| Bucket usage = sum of active object sizes | Billing, quotas, and storage planning | Under-billing, over-provisioning |
| Delete marker ordering is consistent | Understand if object is deleted or not | Object appears both deleted and active |
| ACL does not temporarily expose private objects | Security guarantee | Unauthorized read window possible |

Relational constraints + transactions enforce these.

**3. Concurrency is high and conflicting**

Two clients can update/delete same object simultaneously:
- Without strong consistency → lost updates, duplicate versions, stale ACL.
- With strong consistency → conflicts detected, retried safely.

**4. Security-critical correctness**

Permission changes must be immediate and atomic:
- Revoke user access to object.
- If ACL update is eventual consistency, revoked user may still read for a window.
- With strong consistency: revoke is instantaneous + transactional.

**5. Billing/compliance correctness**

Retention and delete schedules depend on exact metadata state:
- Lifecycle rule: "Delete objects > 365 days old."
- Wrong ordering or partial updates → violate legal retention policy.
- Strong consistency ensures correct delete schedules.

**What can go wrong without strict relational guarantees:**

| Scenario | What happens | Risk |
|----------|---|---|
| Object upload completes in storage, version pointer update fails | File exists but appears "missing" | Data loss perception |
| ACL update succeeds, search index lags | Unauthorized read still possible | Security breach |
| Delete marker written, lifecycle state not updated | Object appears both deleted and active | Billing/retention violation |
| Two concurrent deletes partially applied | Orphaned version records, dangling references | Metadata corruption |

**Rule of thumb:**
- Use strong relational guarantees for control-plane (ownership, ACL, version state, lifecycle).
- Use eventual consistency for non-critical paths (search index, analytics counters, cache).

**When NoSQL is a better choice:**
- Metadata traffic is so high that SQL sharding/joins become the bottleneck.
- Access pattern is mostly key-based reads/writes (minimal relational joins).
- You can accept eventual consistency for non-critical metadata paths.

**NoSQL tradeoff (why not default to it):**
- ✅ Very high write throughput, linear horizontal scaling, predictable key-based latency.
- ❌ Weaker cross-row transactional guarantees, harder ad-hoc relational queries, more app-side logic for consistency.

**Why not use just one?**
- PostgreSQL alone: Can't do fast full-text search, 100B objects slow
- NoSQL alone: Great for key-value scale, but poor fit for rich relational constraints + multi-entity transactions
- Elasticsearch alone: Not ACID, no multi-key transactions, eventual consistency
- Redis alone: Too small (memory-limited), no durability
- Time-Series alone: Not built for metadata queries
- Distributed FS alone: No metadata indexing, can't search by filename

## Step 5: Replication & Durability

### 3-Way Replication
```
Original data → Replica in different rack
              → Replica in different region

Ensures:
- Survive single data center failure
- Survive regional disaster
- Durability: 99.999999999%
Cost: 3x storage
```

### Erasure Coding (More Efficient)
```
Data split: D1, D2, D3, D4 (4 blocks)
Parity added: P1, P2 (2 blocks)
Total stored: 6 blocks

Recover if 2 blocks lost (need 4 of 6)
Cost: 1.5x storage (6/4)
vs 3x with replication
```

#### Erasure Coding in 30 Seconds (Beginner)

Think of it as "safety with less storage":

1. Split one file into data pieces.
2. Add a few extra parity pieces (computed with math).
3. Store all pieces on different servers.
4. If some servers fail, rebuild missing pieces from the remaining ones.

**Example (4+2):**
- 4 data chunks: D1, D2, D3, D4
- 2 parity chunks: P1, P2
- Total stored chunks: 6
- You only need **any 4 of 6** to recover the full file.

**Why use this?**
- Cheaper than 3 copies: 1.5x storage vs 3x
- Still highly durable
- Best for warm/cold data where cost matters more than fastest writes

**Tradeoff:**
- Writes/rebuilds are more complex (extra math)
- But storage savings are huge at large scale

#### How Recovery Actually Works (4+2)

In 4+2 erasure coding, each stripe has 4 data chunks and 2 parity chunks.

For one stripe:
- Data chunks: D1, D2, D3, D4
- Parity chunks: P1, P2

Parity chunks are math equations built from the data chunks:

$$
P1 = c_{11}D1 + c_{12}D2 + c_{13}D3 + c_{14}D4
$$

$$
P2 = c_{21}D1 + c_{22}D2 + c_{23}D3 + c_{24}D4
$$

All operations are done in finite-field arithmetic (not normal integer math).

#### Case A: One Chunk Missing

If D3 is missing, and D1, D2, D4, P1 are available:

$$
D3 = (P1 - c_{11}D1 - c_{12}D2 - c_{14}D4) / c_{13}
$$

So one missing chunk is recovered directly from one parity equation.

#### Case B: Two Chunks Missing

If D2 and D4 are missing, and D1, D3, P1, P2 are available:

1. Move known terms to the other side:

$$
P1' = P1 - c_{11}D1 - c_{13}D3
$$

$$
P2' = P2 - c_{21}D1 - c_{23}D3
$$

2. Solve the 2-equation system:

$$
P1' = c_{12}D2 + c_{14}D4
$$

$$
P2' = c_{22}D2 + c_{24}D4
$$

Now there are 2 unknowns (D2, D4) and 2 independent equations, so both chunks are recoverable.

#### Easy Intuition (XOR Mental Model)

Single-parity idea:

$$
P = D1 \oplus D2 \oplus D3 \oplus D4
$$

If D3 is missing:

$$
D3 = P \oplus D1 \oplus D2 \oplus D4
$$

Numeric XOR example:
- D1 = 5, D2 = 9, D3 = 12, D4 = 3
- P = 5 xor 9 xor 12 xor 3 = 3
- If D3 is lost: D3 = 3 xor 5 xor 9 xor 3 = 12

Why 4+2 can recover two losses:
- One parity equation can recover only one unknown.
- Two parity equations can recover up to two unknowns (if equations are independent).

## Interview Cheat Sheet Q&A

**Q: Why multipart upload when single upload is simpler?**  
A: Single upload fails on large files or unstable networks. Multipart: split 1GB into 10 × 100MB chunks. If chunk 5 fails, retry only chunk 5 (not all 10). Upload chunks in parallel: 8x faster. Also, metadata server tracks progress: "9 of 10 chunks received."

**Q: Why 3-way replication instead of erasure coding?**  
A: Replication is simpler. Erasure coding saves space (1.5x vs 3x) but slower writes (must calculate parity) and slower rebuilds (more complex). S3 uses replication for regular files, erasure coding for archival tiers.

**Q: How to handle if user's region data center burns down?**  
A: Data already replicated to other regions. API detects region down, routes to secondary region. Apps shouldn't notice (maybe 1-2 seconds delay). RTO (recovery time) < 1 minute. RPO (data loss) = 0.

**Q: Can we compress files to save cost?**  
A: Yes, but transparent compression is risky. If compression algorithm has bug, file corrupts on decompression. Better: recommend users compress before upload. S3 lets admins set storage policies: "Auto-compress PDF files" with caveats.

**Q: What if a file is accessed 1 million times in 1 day?**  
A: Without CDN, 1M downloads all hit main data center, network saturates. With CDN: file cached at 1000+ edge locations worldwide. User in Tokyo downloads from Tokyo edge (cached locally). Reduces bandwidth by 100x.

**Q: How to prevent someone from accessing another user's file?**  
A: ACL on each object. Default: private (only owner). Owner can make public or share via presigned URL (time-limited, signature).

## Full Flow (Start to End)

### Happy Path
1. Client request enters API Gateway and is authenticated/authorized.
2. Orchestrator service validates input and routing context.
3. Core service executes primary business logic and required checks.
4. Read path uses cache first; fallback goes to durable database/store.
5. Write path updates fast layer first (where applicable) and publishes async events.
6. Downstream consumers persist durable state and trigger secondary effects.
7. Response is returned to client with final status and metadata.

### Failure and Retry Paths
1. Cache miss: read from durable store, then repopulate cache.
2. Dependency timeout: retry with backoff or circuit-breaker fallback.
3. Async event failure: retry queue and dead-letter queue (DLQ) handling.
4. Duplicate request: idempotency key returns prior successful outcome.
5. Concurrent updates: version/lock conflict triggers re-read and safe retry.

### End-State Guarantees
- Low-latency user operations on the hot path.
- Durable correctness in the source-of-truth datastore.
- Eventual consistency for non-critical async side effects.
- Strict correctness at critical boundaries (commit/payment/finalization).

## Step 4: Data Organization

### What is a Bucket?

A **bucket** is a top-level container/namespace for storing objects (files). Think of it as a project folder with its own configuration and access control.

**Key Characteristics:**

1. **Globally unique name** — bucket names must be unique across ALL users worldwide
   - Example: `my-company-backups` (once taken, nobody else can use it)
   - Naming rules: 3-63 characters, lowercase, numbers, hyphens only
   - Cannot start/end with hyphen, cannot be IP address format

2. **Top-level organizational unit** — all files live inside buckets
   - Cannot store files "outside" a bucket
   - One account can have multiple buckets for different purposes
   - Example: `photos-2024`, `backups-prod`, `logs-archive`

3. **Configuration per bucket** — settings apply to all files inside
   - Versioning: enable/disable keeping old versions
   - Encryption: enable/disable at-rest encryption
   - Lifecycle policies: "move to Glacier after 90 days", "delete after 1 year"
   - Access control: public, private, or shared with specific users

4. **Separate ownership & permissions** — each bucket has an owner
   - Owner controls who can upload, download, delete, list
   - Owner can make bucket public (everyone reads) or private (only owner)
   - Can grant read-only or read-write access to specific users

**Real-world example:**

```
User "john@company.com" has 4 buckets:

1. my-company-backups (Private)
   ├── 2026-06-22/database-dump.sql (2 GB)
   ├── 2026-06-21/database-dump.sql (2 GB)
   └── 2026-06-20/database-dump.sql (2 GB)
   Policy: Keep all versions for 1 year, move to Glacier after 90 days

2. photos-public (Public, anyone can download)
   ├── 2026-06-18/vacation/beach.jpg
   ├── 2026-06-18/vacation/sunset.jpg
   └── 2026-05-01/family/dinner.jpg
   Policy: Versioning enabled, delete old versions after 30 days

3. logs-prod-us-east (Private, shared with ops-team)
   ├── 2026-06-22/access.log (500 MB)
   ├── 2026-06-22/error.log (50 MB)
   └── 2026-06-21/access.log (500 MB)
   Policy: Auto-delete after 90 days (compliance)

4. temp-uploads (Private, auto-cleanup)
   ├── user-uploads/123/large-video.mp4 (in progress)
   ├── user-uploads/456/dataset.zip (in progress)
   Policy: Delete incomplete uploads after 7 days
```

**Why buckets instead of simple folders?**

| Aspect | Bucket Approach | Simple Folder Approach |
|--------|-----------------|----------------------|
| **Namespace** | Global uniqueness; no conflicts | Same filename OK in different folders |
| **Policy** | One policy per bucket | Would need per-folder config |
| **Billing** | Track cost per bucket easily | Track cost harder (nested folders) |
| **Access Control** | Bucket-level ACLs + object-level | Only object-level (complex) |
| **Scalability** | Can replicate bucket independently | Must replicate entire tree |
| **Data Isolation** | Clean separation for different purposes | Risk of accidental mixing |

## Step 5: Upload Flow - Multipart Upload

### Simple Upload (< 5GB)
```
PUT /bucket/key
Content: File data
Response: ETag, Version ID
```

### Multipart Upload (> 5GB)

```
1. Initiate:
   POST /bucket/key?uploads
   Response: UploadID

2. Upload Parts (parallel):
   PUT /bucket/key?uploadId=abc&partNumber=1
   PUT /bucket/key?uploadId=abc&partNumber=2
   PUT /bucket/key?uploadId=abc&partNumber=3
   Response: ETags for each part

3. Complete Upload:
   POST /bucket/key?uploadId=abc
   Body: List of part ETags
   Response: Final ETag, Version ID

Advantages:
- Parallel uploads (faster)
- Resumable (can restart failed parts)
- Better error handling
```

### Upload Optimization
```
Parallel Uploads:
- Split file into chunks (5-100MB each)
- Upload 4-8 chunks concurrently
- Combine parts server-side
- Result: 4-8x faster upload

Resume Capability:
- Client tracks uploaded parts
- On failure, retry failed parts
- Don't re-upload successful parts
- Save bandwidth for large files
```

## Step 6: Data Durability - Replication & Erasure Coding

### Replication Strategy
```
3-Way Replication (simpler but expensive):
Original → Replica 1 (different rack)
        → Replica 2 (different region)

Write to master, replicate to followers
Durability: 99.999999999% (11 9s)

Cost: 3x storage
```

### Erasure Coding (Reed-Solomon)

```
More efficient than replication
Data + Parity blocks

Example (4+2):
Data:   D1  D2  D3  D4
Parity: P1  P2

Total 6 blocks stored
Recover original if any 4 blocks available
Lose 2 blocks, still recover

Cost: 1.5x storage (6/4)
vs 3x with replication

Trade-off: Slower writes (need parity calc)
         Faster reads (can use any 4 blocks)
```

### Hybrid Approach
```
Hot tier: Recent, frequently accessed
→ 3-way replication (fast access)

Warm tier: Medium access frequency
→ Erasure coding (balanced cost/speed)

Cold tier: Archive, rarely accessed
→ Erasure coding with geographically dispersed

Auto-transition based on access patterns
```

### Storage Nodes: Distributed Chunk Layout
```
Object: 2024-06-18/database-dump.sql (2 GB)
Split into 4 chunks × 500 MB each with erasure coding (4+2):

Original chunks (stored across 4 nodes):
Chunk-0 (500 MB) → Storage Node 12 (Rack A, DC1)
Chunk-1 (500 MB) → Storage Node 18 (Rack B, DC1)
Chunk-2 (500 MB) → Storage Node 5 (Rack C, DC2)
Chunk-3 (500 MB) → Storage Node 22 (Rack D, DC2)

Parity chunks (redundancy, Reed-Solomon):
Parity-0 (500 MB) → Storage Node 31 (Rack E, DC3)
Parity-1 (500 MB) → Storage Node 47 (Rack F, DC3)

Recovery property:
- Recover original if any 4 of 6 chunks available
- Node-12 dies? Use Nodes 18,5,22,31,47 (any 4 of 5) → recover
- Entire DC2 dies? Still have chunks 0,1,4,5 → recover
```

### Replication & Geo-Distribution
```
Object: 2024-06-18/database-dump.sql

Hot tier (Recent, Replication):  3-way replication
├─ Master copy    → Data Center US-EAST-1 (Primary)
├─ Replica copy 1 → Data Center US-WEST-1 (Cross-region)
└─ Replica copy 2 → Data Center EU-WEST-1 (Cross-ocean)
   Master serves all reads; followers are read-only hot backups
   Write path: Update master → async replicate to followers
   Durability: Any 1 DC dies, 2 copies survive

Warm tier (30-90 days):  Erasure coding (4+2)
├─ DC US-EAST-1: Chunks 0,1
├─ DC EU-WEST-1: Chunks 2,3
├─ DC AP-SOUTH-1: Parity 0,1
   Any 4 chunks available → full recovery
   Cost: 1.5x storage vs 3x with replication

Cold tier (Archive, 90+ days):  Erasure coding + offline tape backup
├─ DC US-EAST-1: 2 erasure-coded copies distributed
├─ Off-site tape: Full copy (immutable, air-gapped)
   Recovery time: 12-24 hours (acceptable for archive)
```

## Step 13: Key Design Decisions & Tradeoffs

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Multipart upload | Resume from failure; parallel uploads = 4-8x faster | More complex client logic; server-side state management |
| Erasure coding over replication | 1.5x storage vs 3x; cheaper | Slower writes (parity calculation); complex recovery |
| S3-style bucket organization | Universally familiar; good namespace isolation | Global uniqueness constraint; no nested buckets |
| Tiered storage (hot/warm/cold) | Cost savings (older data cheaper storage) | Complexity; slower access to cold tier; cross-tier movement |
| Eventual consistency | Higher availability; no coordination overhead | Brief stale reads; hard to debug |

## Interview Cheat Sheet Q&A

**Q: Why allow multipart upload when single-request upload is simpler?**  
A: Single upload fails on large files (network timeout, server restart). With multipart, if chunk 3 of 10 fails, retry only chunk 3, not all 10. Also, upload chunks in parallel: chunk 1 on connection 1, chunk 2 on connection 2, etc. 4 parallel = ~4x faster.

**Q: Why use erasure coding instead of simple 3-way replication?**  
A: Replication wastes space. Need 3 copies for durability. Erasure coding (e.g., 4+2): split file into 4 data blocks + 2 parity blocks. Need only 4 of 6 to recover. Storage = 1.5x, not 3x. Over millions of files, this saves billions of dollars. Tradeoff: writes need parity calculation (slower), but writes are rare compared to reads.

**Q: What if a user deletes a file, then wants it back in 3 months?**  
A: Without versioning, it's gone forever. With versioning, old versions are retained (and charged). User can restore. Tradeoff: costs more (storing multiple versions). Best practice: keep versions for 30 days for free, charge per version after that.

**Q: Can we serve files directly from S3 without a CDN?**  
A: Technically yes. User downloads directly from S3. Works fine for a few hundred users. But millions of concurrent downloads hit rate limits and need DDoS protection. CDN has thousands of edge servers, so requests are distributed, and popular files are cached locally at CDN edges.

**Q: How do we prevent someone from sharing a private file's URL?**  
A: Use presigned URLs that expire. "GET /file?signature=xyz&expires=1624050000". After expiry, URL is invalid even if shared. Also, use authentication: only the file owner can generate presigned URLs.

**Q: What if metadata database (file listing) is corrupted?**  
A: Data is still safe in storage, just orphaned. Rebuild metadata by scanning storage: "What data blocks exist?" → infer original files. Takes time but recovers data. This is why immutable infrastructure is important.

## Start-to-End Flow with Tradeoffs (Quick Revision)

### UPLOAD SCENARIO: 2 GB Video File

| Step | What Happens | Tradeoff |
|------|--------------|----------|
| 1. Client initiates multipart upload | Calls `POST /bucket/video.mp4?uploads` → Server creates uploadID, returns in ~50ms | Trade-off: Extra RPC adds latency but enables resume on failure |
| 2. Client splits into 4×500MB chunks | Local compute cost before network; each chunk hashed for integrity | Trade-off: Chunking adds client-side complexity but enables 4-8x parallel speedup |
| 3. Upload chunks in parallel (4 connections) | Chunk-0 → Node-12, Chunk-1 → Node-18, Chunk-2 → Node-5, Chunk-3 → Node-22 simultaneously | Trade-off: 4 parallel ≈ 4x faster upload (2 GB in 5 min vs 20 min single), but network overhead + connection management complexity |
| 4. Server stores original + parity blocks | Erasure coding (4+2): write Chunks 0-3 to data nodes, compute + write Parity-0,1 to separate nodes | Trade-off: Parity calculation adds ~50ms per chunk, but storage cost drops 2x vs replication (1.5x vs 3x) |
| 5. Async replicate across regions | After local durability ✓, background job: ship copy to DC-WEST + DC-EU | Trade-off: Replication delay (eventual consistency window ~30s), but survive region failure |
| 6. Update metadata DB | INSERT into objects table + update buckets.storage_used | Trade-off: DB write latency (5-10ms), but enables bucket listing/search |
| 7. Publish event to Elasticsearch indexer | Async event: "new object with tags + content-type" | Trade-off: Event delay (1-5s), eventual consistency in search, but scales to millions of objects |
| 8. Return to client: Upload complete | Response includes version_id + etag (for integrity check) | Trade-off: All prior steps must succeed; slow if network to distant DC |
| 9. Cleanup multipart state after 7 days | Async job: delete incomplete uploads + temp blocks | Trade-off: Stale state cleanup delay means orphaned blocks briefly consume storage, but simplicity > strict cleanup |
| **Rule of thumb:** | Prioritize durability (replicate + erasure code) > speed (3-5 sec end-to-end OK for 2 GB) | Upload success = data persisted; latency = tradeoff for reliability |

### DOWNLOAD SCENARIO: Retrieve 2 GB Video

| Step | What Happens | Tradeoff |
|------|--------------|----------|
| 1. Client requests object | `GET /bucket/video.mp4` → API Gateway validates auth + rate limit | Trade-off: Auth check adds 10-50ms but prevents unauthorized access |
| 2. Check metadata cache (Redis) | Lookup object version + location → hit in ~1ms (hot objects) | Trade-off: Cache hit is fast but stale for ~1 hour; miss requires DB lookup (10ms) |
| 3. Route to nearest storage node | If user in Tokyo, route to DC-AP; if user in USA, route to DC-US | Trade-off: Geo-routing optimizes latency (~50ms from local DC vs 200ms from distant) but requires request dispatcher |
| 4. Stream chunks via CDN or direct download | For popular files (10K+ downloads): cache at CDN edge (100ms latency globally) | Trade-off: CDN cache saves bandwidth (1M downloads → 1 source fetch + 1M edge hits) but adds CDN cost |
| 5. Reconstruct if using erasure coding | Only if DC partial failure; if 4+ of 6 chunks available → reconstruct on-the-fly | Trade-off: Reconstruction adds CPU cost (~100ms) but prevents complete failure |
| 6. Decompress if object was compressed | Optional; if metadata.compression='gzip' → decompress as stream | Trade-off: Decompression CPU cost (~200ms for 2 GB) but saves 60% bandwidth for text |
| 7. Return with proper headers | Content-Type, Content-Length, ETag, Cache-Control, Age | Trade-off: Headers reduce latency (helps client-side caching) but are metadata overhead |
| 8. Log download event (async) | Fire background event for analytics + billing | Trade-off: Async logging doesn't block response (fast) but eventual consistency for metrics |
| 9. Range request support for large files | If network interrupts at byte 1GB, resume from byte 1GB | Trade-off: Range requests require partial chunk downloads but enable resume |
| **Rule of thumb:** | Optimize for reads (cache + CDN) because reads >> writes; downloads are latency-sensitive | Read success = data delivered correctly; latency = key SLA (< 100ms target) |
