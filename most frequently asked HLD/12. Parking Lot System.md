# System Design: Distributed Parking Lot System (Beginner-Friendly Guide)

---

## What Are We Building?

Imagine a system managing parking across a city (like ParkWhiz or SpotHero) — a distributed system that:
- Shows available parking spaces near you in real-time
- Reserves a spot with the ability to guide you to it
- Charges you based on duration
- Integrates with 1000s of parking lots across the city
- Handles entry/exit at toll booth barriers

The interesting engineering challenges hidden in this flow:
- **Consistency** — if two people try to reserve the last available spot, only one should succeed
- **Real-time inventory** — parking availability must update instantly (not "sometimes")
- **Geographic distribution** — coordinate across 1000 parking lots, each with separate infrastructure
- **Payment at exit** — must work offline (tolls often lose internet); reconcile later
- **Overbooking** — parking lots intentionally allow >100% occupancy (expect no-shows); manage this
- **Scale** — handle 10K+ parking exits per hour (peak times); billing must be fast

## Step 3: Database Choice

| Consideration | SQL | NoSQL |
|---------------|-----|-------|
| Parking availability (real-time) | ✅ Good | ✅ Good |
| Transactions (charge accurately) | ✅ ACID critical | ❌ No ACID |
| Historical data (audit) | ✅ Perfect | ❌ Limited |
| Geospatial queries | ✅ With PostGIS | ✅ Good |
| Entry/exit events (high write) | ❌ Bottleneck | ✅ Better |

**Decision: Hybrid**
- **PostgreSQL** for inventory, reservations, users (ACID needed)
- **Redis** for real-time availability (fast updates, caching)
- **Cassandra/DynamoDB** for events (entry/exit log, high writes)

## Step 5: Space Allocation & Reservation

### Inventory Model

```
Parking Lot (lot_001):
├── Level 1: 100 spaces
│   ├── Car: 60 spaces (50 available, 10 occupied)
│   └── Motorcycle: 40 spaces (38 available, 2 occupied)
├── Level 2: 100 spaces
├── Level 3: 100 spaces
└── Reserved: 200 spaces

Real-time inventory (Redis):
Key: "lot:lot_001:level:1:car:available"
Value: 50

Key: "lot:lot_001:level:1:motorcycle:available"
Value: 38
```

### Reservation Flow

```
1. Customer searches:
   "I need a spot for 2 hours"
   
2. Check availability:
   SELECT available_spaces FROM lot_inventory
   WHERE lot_id = 'lot_001' AND vehicle_type = 'CAR'
   
3. IF available >= 1:
   a) Create reservation (PENDING):
      INSERT INTO reservations (...) VALUES (...)
      WITH idempotencyKey to prevent duplicates
   b) Hold space (decrement available):
      UPDATE lot_inventory SET available_spaces -= 1
   c) Return reservation_id to customer
   
4. IF customer doesn't arrive within 30 min:
   a) Reservation expires
   b) Release space: available_spaces += 1
   
5. Customer enters (confirms reservation):
   a) Scan license plate via ANPR
   b) Match to reservation
   c) Update status: CONFIRMED
   d) Record entry time
```

### Handling Double Booking

```
Two customers try to reserve last available space:

Customer A: 
  1. Reads available_spaces = 1
  2. Tries to INSERT reservation
  3. UPDATE available_spaces = 0
  ✅ Success

Customer B:
  1. Reads available_spaces = 1 (stale read)
  2. Tries to INSERT reservation
  3. UPDATE available_spaces = 0
  BUT: Database constraint fails (available >= 0)
  ❌ Fails with error: "No spaces available"

Customer B sees: "Sorry, lot is now full"
```

## Step 6: Entry & Exit Processing

### Entry Processing

```
Customer arrives at barrier:

1. ANPR camera reads license plate: ABC123
2. Look up in reservations:
   SELECT * FROM reservations WHERE licensePlate = 'ABC123'
   
3. If found (reserved):
   a) Check if expired (30 min buffer): NO
   b) Lift barrier automatically
   c) Record entry: INSERT INTO entry_log (...)
   d) Update status: CONFIRMED → PARKED
   e) Start billing clock
   
4. If not found (no reservation):
   a) Check if lot has capacity
   b) If yes: Allow entry (create on-the-fly reservation)
   c) If no: Display "Lot full"
```

### Exit Processing

```
Customer at toll booth barrier:

1. ANPR reads: ABC123
2. Look up parked vehicle:
   SELECT * FROM parked_vehicles WHERE licensePlate = 'ABC123'
   
3. Calculate charges:
   duration = exit_time - entry_time
   rate = parking_lot.hourly_rate
   total = ceil(duration / 60) * rate  // Round up to nearest hour
   
   Example: 2 hours 15 minutes = 3 hours charge
   3 hours × $5/hour = $15
   
4. Process payment:
   POST /payment/charge
   {
     "amount": 15.00,
     "paymentMethod": vehicle.paymentMethod
   }
   
5. If payment succeeds:
   a) Lift barrier
   b) Record exit
   c) Free up space: available_spaces += 1
   d) Send receipt to customer
   e) Update status: PARKED → EXITED
   
6. If payment fails:
   a) Don't lift barrier
   b) Show message: "Payment failed. Call 1-800..."
   c) Alert lot operator (manual intervention)
```

## Step 7: Pricing Models

### Simple Hourly Rate

```
Monday-Friday: $5/hour
Weekend: $3/hour
Monthly pass: $100/month (unlimited)

Calculation at exit:
duration = ceil((exit_time - entry_time) / 60 minutes)
cost = duration * hourly_rate
```

### Dynamic Pricing

```
During peak hours (9am-12pm, 5pm-8pm):
- Occupancy > 90%
- Increase rate by 1.5x

Off-peak (10pm-6am):
- Occupancy < 30%
- Decrease rate by 0.7x

Benefits:
- Encourage off-peak parking
- Maximize revenue during peak
- Balance demand with supply
```

### Validation & Fraud Prevention

```
Suspicious pattern detected:
- 10 entries/exits in 1 minute (impossible)
- Negative duration (exit before entry)
- Customer charged $500 (unusually high)

Action:
- Flag transaction for review
- Require manual approval
- Alert lot operator
- Don't lift barrier until confirmed
```

## Step 8: Distributed System Considerations

### Cross-Lot Management

```
Parking system across 1000 lots:

City Center:
├── Lot A: 500 spaces (managed by lot_server_001)
├── Lot B: 500 spaces (managed by lot_server_002)
└── Lot C: 500 spaces (managed by lot_server_003)

API layer:
- Route requests to correct lot server
- Shard by lot_id: hash(lot_id) % 1000

Centralized:
- Reservation database (shared)
- Payment service (shared)
- ANPR result aggregation (shared)

Per-lot:
- Inventory cache (Redis)
- Entry/exit log (local + sync to center)
```

### Eventual Consistency

```
Entry recorded at lot_001:
1. Local log: IMMEDIATE (offline-safe)
2. Central DB: 5-10 sec delay (async sync)

Benefit: Barrier never blocks waiting for network
Tradeoff: Temporary inconsistency (same space double-booked), auto-corrected

If network fails for 1 hour:
- Lot still functions (barriers work)
- Manual reconciliation after recovery
```

## Step 10: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Separate entry and exit processing | Different latency requirements (entry quick; exit involves payment) | More services to manage |
| ANPR for automated entry | Faster entry; no manual payment; better tracking | ANPR accuracy ~95% (misreads happen) |
| Eventual consistency (async sync) | Barriers never blocked by network | Brief inconsistency; requires reconciliation |
| Redis cache for availability | Sub-millisecond lookup for 100K+ queries | Must keep in sync with DB; cache misses |
| Hold space on reservation | Prevent overbooking (customer gets promised spot) | ~10% of reservations never used (overbooking) |

## Interview Cheat Sheet Q&A

**Q: What if ANPR reads license plate incorrectly?**  
A: 95% accuracy, so 5% misread. Wrong plate → look up reservation → not found → manual intervention. Operator manually confirms vehicle, allows exit. Or: display sign "Plate not recognized, enter plate manually?" to driver.

**Q: What if customer never leaves and space is occupied forever?**  
A: Set max stay limit (e.g., 24 hours). After 24h, automatically release space, charge maximum (e.g., $120). Send notification: "Long stay detected. Vehicle will be towed if not retrieved." Prevents lost cars and clears stuck spaces.

**Q: How do we handle payment failure at exit?**  
A: Don't lift barrier. Show message: "Payment failed. Retry or contact support." Options: (1) Retry with different card; (2) Call support; (3) Leave vehicle, tow later (rare). Ensure customer can talk to someone.

**Q: What if lot is full but someone has a reservation?**  
A: Reservations guarantee space, so total_reserved + total_occupied can exceed 100%. If customer without reservation arrives: "Lot full, but you can make reservation for later time?" or "Try nearby lot X (2km away, has space)."

**Q: Can we oversell (sell 110% capacity)?**  
A: Yes, like airlines. If 100 spaces, allow 110 reservations (assume 10% no-show). If all 110 show up → some can't park → compensation. Risk: reputation damage. Use conservative overbooking % (e.g., 5-10%).

**Q: Why not just use a single database for all lots?**  
A: Scaling: 1000 lots × 500 transactions/sec/lot = 500K transactions/sec. Single DB can't handle. Shard by lot → each DB handles 500 tx/sec (manageable). Trade-off: cross-lot transactions harder (e.g., "Show all available spaces in downtown" = query 10 shards).

- ✅ Real-time space availability tracking

- ✅ Fast entry/exit processing (<10 sec)
- ✅ Accurate billing and payment processing
- ✅ ANPR integration for automation
- ✅ Distributed architecture across multiple lots
- ✅ Eventual consistency with reconciliation
- ✅ Overbooking support for revenue optimization
- ✅ Monitoring and operational alerts
