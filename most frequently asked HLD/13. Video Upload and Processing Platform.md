# System Design: Video Upload & Processing Platform (Beginner-Friendly Guide)

---

## What Are We Building?

Imagine YouTube — a video upload and processing platform where:
- Users upload videos (from 1GB to 100GB)
- Platform encodes videos to multiple qualities (240p, 360p, 720p, 1080p)
- Videos are stored and served globally
- Users can watch within hours of upload (not days)
- System handles failures and retries gracefully

The interesting engineering challenges hidden in this flow:
- **Huge file uploads** — uploading 100GB over internet is unreliable; network fails halfway
- **Resumable uploads** — if upload fails at 99GB, don't force re-uploading all 100GB
- **Parallel encoding** — encode 240p, 360p, 720p simultaneously to save time
- **Reliable processing** — if encoding worker crashes, reschedule job on another worker
- **Cost optimization** — video storage is expensive; delete original after encoding; use efficient codecs
- **Processing speed** — users expect video available quickly (2-4 hours, not 24 hours)

## Step 3: Database Choice

| Consideration | SQL | NoSQL |
|---------------|-----|-------|
| Metadata (title, description) | ✅ Relational | ✅ OK |
| Encoding jobs (status tracking) | ✅ Transaction needed | ❌ No rollback |
| Video statistics (views, likes) | ✅ Good | ✅ Better for counters |
| Audit trail (processing history) | ✅ Perfect | ❌ Eventual consistency |

**Decision: Hybrid**
- **PostgreSQL** for video metadata, encoding jobs, users
- **Redis** for job queue (fast writes)
- **S3/Blob** for actual video storage
- **Cassandra** for view statistics (high write rate, eventual consistency OK)

## Step 5: Resumable Multipart Upload

### Upload Initialization

```
1. Client POSTs to /upload/init
   Server generates: uploadId = "upload_abc123"
   Server calculates: chunkSize = 1GB, totalChunks = 5

2. Client receives upload parameters
3. Client prepares chunks: [1GB, 1GB, 1GB, 1GB, 972MB]
```

### Parallel Chunk Upload

```
Client uploads chunks in parallel (3 simultaneous):

Chunk 1 (0-1GB):     ▓▓▓▓▓▓▓▓  DONE (5 sec)
Chunk 2 (1-2GB):     ▓▓▓▓▓    IN_PROGRESS
Chunk 3 (2-3GB):     ▓▓▓▓▓▓▓▓  DONE (5 sec)
Chunk 4 (3-4GB):     ▓▓▓      IN_PROGRESS
Chunk 5 (4-4.95GB):  ▓▓▓▓▓▓▓▓  DONE (3 sec)

Total time with 3x parallelism: ~20 seconds
Without parallelism: ~35 seconds

Bandwidth usage:
Upload speed: 50 MB/sec × 3 = 150 MB/sec (reasonable)
```

### Resumable on Failure

```
Upload fails at chunk 3 (network cuts out):
Chunks completed: 1, 2, 4, 5
Chunk 3: 50% done

Client retries:
1. POST /upload/resume?uploadId=abc123
2. Server returns: "Chunks 1,2,4,5 received. Retry chunk 3"
3. Client resends only chunk 3
4. Loss: 50% of 1 chunk = 0.5GB (not 5GB)

vs. No resume: Re-upload entire 5GB file
```

### Checksum Verification

```
For each chunk:
1. Client calculates: sha256(chunk_data)
2. Sends with chunk: "X-Checksum-SHA256: abc123..."
3. Server calculates same hash
4. If mismatch: Reject chunk, ask for retry
5. Prevents corruption from network errors
```

## Step 6: Video Processing Pipeline

### Encoding Jobs

```
Original uploaded file:
- input.mp4 (5GB, 4K resolution, H.264 codec)

Create encoding jobs:
Job 1: Encode to 240p (500kbps)  ← Low bandwidth
Job 2: Encode to 360p (1Mbps)    ← Mobile
Job 3: Encode to 720p (2.5Mbps)  ← Desktop
Job 4: Encode to 1080p (5Mbps)   ← High quality
Job 5: Extract audio (128kbps MP3) ← Audio only

Each job sent to queue independently.
Workers process in parallel → 4 videos created simultaneously.
```

### Job Workflow

```
1. Preprocessing (5 min):
   - Analyze video: resolution, codec, duration, frame rate
   - Plan encoding strategy

2. Parallel encoding (varies):
   - 240p: 15 minutes (low resolution, small file)
   - 360p: 25 minutes
   - 720p: 50 minutes  ← Longest
   - 1080p: 80 minutes ← Even longer
   - Total: Run in parallel, take max = 80 minutes

3. Post-processing (5 min):
   - Generate thumbnail
   - Generate preview (10-second clip)
   - Verify output files

4. Upload to storage (10 min):
   - Upload 4 encoded videos to S3
   - Update database with file locations
   - Update status to READY

Total time: 5 + 80 + 5 + 10 = 100 minutes (~2 hours)
```

## Step 8: Failure Handling

### Retry Strategy

```
Job fails (codec error, disk full, etc.):

Retry 1: Immediately (5 seconds)
  - Transient error? Try again
  
Retry 2: After 1 minute
  - Temporary resource contention
  
Retry 3: After 15 minutes
  - Wait for system recovery
  
Fail: After 3 retries
  - Mark as FAILED
  - Send user notification
  - Alert ops team

Exponential backoff prevents hammering failing system.
```

### Dead Letter Queue

```
Job fails 3 times:
1. Move to dead-letter queue
2. Human review: "Why did it fail?"
3. Common issues:
   - Video format unsupported (codec not installed)
   - Disk out of space
   - Worker crash (transient; should retry)
   
4. Fix and re-queue:
   - If codec issue: Install codec, re-queue
   - If space issue: Add more disk, re-queue
   - If crash: Already auto-retried
```

## Step 9: Optimization Techniques

### Parallel Encoding

```
Traditional (sequential):
- Encode 240p (15 min) → 360p (25 min) → 720p (50 min) → 1080p (80 min)
- Total: 170 minutes

Parallel (same machine):
- Use all 16 cores
- Encode all 4 simultaneously
- Total: 80 minutes (2.1x faster)
- Resource: High CPU usage for 80 minutes (acceptable)
```

### Codec Selection

```
Trade-offs:

H.264 (Standard):
- Widely compatible (plays everywhere)
- File size: 1GB for 2-hour 1080p video
- Encoding time: Fast

H.265 (Modern):
- 50% better compression (0.5GB for same 2-hour video)
- Encoding time: 3x slower
- Compatibility: Newer devices only

Use both:
- Encode in H.264 (for broad compatibility)
- Encode in H.265 (for mobile, savings)
- Client app chooses based on device

Cost savings from H.265:
- Storage: 50% reduction
- Bandwidth: 50% reduction
- Encoding: 3x more expensive
- Net: Still worth it for large videos
```

### Storage Analysis

```
5GB original video
After encoding to 4 qualities:

240p: 200 MB (4% of original)

With original: 7.55GB per video

Cost reduction strategies:

3. Archive after 1 year to Glacier: $0.004/month
```

## Step 12: Key Design Decisions

| Decision | Why? | Tradeoff |
|----------|------|----------|
| Multipart chunked upload | Resume on failure; parallel upload = 2-3x faster | Client complexity; server state tracking |
| Parallel encoding (4 qualities) | 4x faster overall (80 min vs 170 min) | High CPU usage; might overheat server |
| Job queue (Kafka) | Jobs not lost if worker crashes; retry automatically | Added complexity; eventual consistency |
| Exponential backoff retry | Prevents overwhelming failed system; gives time to recover | Failed jobs delayed hours before giving up |
| Hybrid codecs (H.264 + H.265) | Broad compatibility + storage savings | Double encoding cost; manual logic |

## Interview Cheat Sheet Q&A

**Q: Why split into chunks instead of streaming entire file?**  
A: Network fails. Uploading 5GB over internet fails ~50% of the time. With chunks: if upload fails at chunk 4 (of 5), only chunk 4 retries, not all 5 chunks. Parallel: upload chunks simultaneously = 3x faster upload speed.

**Q: Why separate upload and encoding instead of encoding-while-uploading?**  
A: Encoding uses full CPU. While user uploads (network I/O), CPU is idle. Better: (1) upload fast (quick, simple), store; (2) encode later (use full CPU). Also, user might cancel upload; don't waste CPU if upload abandoned.

**Q: What if encoding fails after 1 hour of processing?**  
A: Retry automatically (with backoff). If fails 3 times: mark as FAILED, notify user "Encoding failed, contact support." Common reason: unsupported video format. Solution: convert format first, re-queue.

**Q: Can we cache partially-encoded videos?**  
A: No, incomplete video is useless. But after encoding completes: cache in CDN. Next user who watches = served from CDN (fast). First user waits for full encode-to-CDN pipeline (slow).

**Q: Why not just use a simple script to encode videos sequentially?**  
A: Sequential: one video at a time, takes days. Queue-based system: 100 workers × 4 jobs each = 400 videos simultaneously. Handles scale. Also, workers are ephemeral; if one fails, job moves to another.

**Q: How do we handle videos larger than RAM (100GB)?**  
A: Stream processing. Don't load entire video into memory. Read chunks (1GB), encode, write output, discard input chunk. Repeat. More complex code, but enables large videos.

## Step 7: Error Handling & Retries

### Failure Scenarios

```
1. Upload Failure:
   - Network timeout: Retry from last chunk
   - Storage unavailable: Retry up to 5 times
   - Validation failure: Notify user, no retry

2. Processing Failure:
   - Encoding error: Log error, mark as failed
   - Worker crash: Job reassigned to another worker
   - Storage write failure: Retry 3 times
   - Timeout (> 12 hours): Mark as failed
```

### Retry Logic

```
On encoding failure:
1. Log error details
2. Increment retry counter
3. If retries < maxRetries:
   - Add back to queue with exponential backoff
   - Wait: 5min, 15min, 1hr, 3hr
4. If retries >= maxRetries:
   - Mark as FAILED
   - Notify user
   - Alert operations team
```

### Dead Letter Queue

```
Failed jobs with max retries:
- Move to dead letter queue
- Manual review by engineers
- Debug and fix root cause
- Re-queue if fixed
```

## Step 8: Optimization Techniques

### Adaptive Bitrate Encoding

```
Instead of fixed bitrate:
1. Analyze source video
2. Determine optimal bitrate per quality
3. Encode adaptive:
   - High motion scenes: higher bitrate
   - Low motion scenes: lower bitrate
   - Saves ~30% bandwidth

Example:
- Standard 720p: 2500kbps
- Action movie 720p: 3500kbps
- Talking head 720p: 1800kbps
```

### Parallel Encoding

```
Encode multiple qualities simultaneously:
- Multi-core CPU: Encode 4 qualities in parallel
- GPU: Hardware acceleration for encoding
- Distributed: Encode 240p on worker1, 360p on worker2
- Result: 4x faster processing

Trade-off:
- More resources needed
- Higher cost
- Faster content availability (better user experience)
```

### Smart Caching

```
Popular videos cached at CDN edge:
- Detect popular videos (trending)
- Pre-encode to all qualities
- Cache in CDN edge locations
- Serve locally without transcoding

Cache layers:
1. S3 (original source)
2. Cloudfront (global CDN)
3. Local CDN edge servers
4. Browser cache (24 hours)
```

```
- Encoding cost
- File size

Example:

- Mobile: H.264 (broader support)

```

| Exponential backoff retry | Give service time to recover; prevent hammering | Failed jobs take hours to fully exhaust retries |
| Separate upload and processing | Upload fast (seconds), process slower (hours) | User sees "processing" state; can't watch immediately |
| Dead letter queue for failures | Manual review; prevent data loss | Requires human review; slow resolution |

## Interview Cheat Sheet Q&A

**Q: Why split into chunks instead of streaming the entire file?**  
A: Network is unreliable. If uploading a 50GB file and network fails at 49GB, you've wasted 49GB of bandwidth and must restart. With chunks, only the failed chunk restarts (~2GB). Also, parallel chunks: send chunk 1 and chunk 2 simultaneously = 2x faster.

**Q: Why separate upload and encoding instead of encoding-while-uploading?**  
A: Encoding is CPU-intensive. While user is uploading (slow network), you can't spare CPU to encode. Upload first (simple, fast), store, then encode (use full CPU later). Also, user might cancel upload; don't waste CPU encoding if video is abandoned.

**Q: What if encoding fails after 12 hours (timeout)?**  
A: Mark as FAILED. Show user: "Processing failed. Please re-upload." Collect failure metrics: which format fails most? Maybe there's a bug. Also, offer manual support: "Contact us if this persists."

**Q: Can we cache partially-encoded videos?**  
A: No, incomplete videos are useless. But after encoding completes, you can cache in CDN. The next person who watches that video gets it from CDN (fast). First viewer waits for full encode-to-CDN (slow).

**Q: Why use H.264 instead of H.265 (better compression)?**  
A: Older devices don't support H.265. Users with old phones can't watch H.265 videos. H.264 is "safe" (broad support). For new videos, encode in both: H.264 for compatibility, H.265 for mobile efficiency. Client app chooses based on device.

**Q: How do we handle videos that are 200GB (raw 4K)?**  
A: Upload in chunks (1GB each, 200 chunks). Chunks go to temporary storage. When all 200 arrive, verify all checksums, then move to permanent storage. If chunk 50 is corrupted, user re-uploads only chunk 50. Cost: 50GB wasted bandwidth, not 200GB.
