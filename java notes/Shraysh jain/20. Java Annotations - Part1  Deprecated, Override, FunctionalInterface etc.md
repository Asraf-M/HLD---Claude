## What is an Annotation?

Annotations are **metadata** — extra information added to Java code (classes, methods, fields, parameters).

```java
@Override                      // ← annotation (metadata)
public void fly() { ... }
```

| Property | Detail |
|---|---|
| **Purpose** | Provide information to compiler / runtime |
| **Optional** | Won't break anything if removed |
| **How to read** | Via **Reflection** at runtime |
| **Where to apply** | Classes, methods, fields, parameters, constructors, etc. |
| **Syntax** | `@AnnotationName` |

### How It Works

```
  You add: @Override on method
       ↓
  Compiler reads metadata
       ↓
  Adds logic: "Check parent class has same method"
       ↓
  If not found → Compilation Error
```

## 1. @Deprecated

Marks a class, method, or field as **no longer recommended** for use.

```java
class Mobile {
    @Deprecated
    public void oldMethod() { ... }
    
    public void newMethod() { ... }  // use this instead
}

Mobile m = new Mobile();
m.oldMethod();    // ⚠️ Warning: deprecated API usage
```

| Property | Detail |
|---|---|
| Effect | Shows **warning** to users of the deprecated element |
| Can apply to | Constructor, field, local variable, method, package, parameter, type |
| Meaning | No further development — use the alternative |

## 2. @Override

Tells compiler to verify the method **correctly overrides** a parent method.

```java
class Eagle implements Bird {
    @Override
    public void fly() { ... }       // ✅ matches parent
    
    @Override
    public void flyOne() { ... }    // ❌ Compilation Error — no flyOne() in parent!
}
```

| Property | Detail |
|---|---|
| Can apply to | Methods only |
| Effect | Compile-time check — method must exist in parent |

---

## 3. @SuppressWarnings

Tells compiler to **ignore specific warnings**.

```java
@SuppressWarnings("deprecation")       // suppress deprecation warning
public void test() {
    mobile.oldMethod();                 // no warning shown
}

@SuppressWarnings({"deprecation", "unused"})  // suppress multiple
public void test2() { ... }

@SuppressWarnings("all")               // suppress ALL warnings (use carefully!)
public void test3() { ... }
```

| Property | Detail |
|---|---|
| Can apply to | Field, method, parameter, constructor, local variable, type |
| Common values | `"deprecation"`, `"unused"`, `"unchecked"`, `"all"` |
| Caution | Don't suppress warnings blindly — they can prevent runtime errors |

## 4. @FunctionalInterface

Restricts interface to **exactly one abstract method**.

```java
@FunctionalInterface
interface Bird {
    void fly();                    // ✅ one abstract
    default void eat() { }         // OK — not abstract
    static void info() { }         // OK — not abstract
}

@FunctionalInterface
interface Bird {
    void fly();
    void eat();                    // ❌ Compilation Error — two abstracts!
}
```

| Property | Detail |
|---|---|
| Can apply to | Type (class/interface/enum) — typically interfaces |
| Effect | Compile error if more than one abstract method |

## 5. @SafeVarargs

Suppresses **heap pollution warnings** on methods with varargs parameters.

### What is Heap Pollution?

When an object of type A holds a reference to an object of type B:

```java
List<Integer> intList = ...;
Object[] objArray = intList.toArray();
objArray[0] = "hello";            // heap pollution! Integer list has a String
```

### How Varargs Enable Heap Pollution

```java
void printValues(List<Integer>... lists) {    // varargs
    Object[] objArray = lists;                 // valid — arrays are covariant
    List<String> strList = List.of("hello");
    objArray[0] = strList;                     // heap pollution!
    // lists[0] is now List<String> but typed as List<Integer>
}
```

### Suppressing the Warning

```java
@SafeVarargs                     // "I've verified no heap pollution occurs"
static void safeMethod(List<Integer>... lists) {
    // safe usage only
}
```

| Rule | Detail |
|---|---|
| Can apply to | Methods (`static` or `final`) and constructors with varargs |
| Java 9+ | Also allowed on `private` methods |
| Why `static`/`final`/`private`? | These can't be overridden — prevents accidental removal of the annotation in subclasses |

## Summary

| Annotation | Purpose | Applies To |
|---|---|---|
| `@Deprecated` | Mark as "don't use" — shows warning | Class, method, field, etc. |
| `@Override` | Verify correct override at compile time | Methods only |
| `@SuppressWarnings` | Hide specific compiler warnings | Field, method, type, etc. |
| `@FunctionalInterface` | Enforce single abstract method | Interfaces |
| `@SafeVarargs` | Suppress heap pollution warning | Static/final/private methods with varargs |
