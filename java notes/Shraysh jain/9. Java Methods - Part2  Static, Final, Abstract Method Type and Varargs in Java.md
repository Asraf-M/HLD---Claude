## 1. Static Method

A method declared with the `static` keyword is **associated with the class**, not with any object.

### How It Works

```java
class A {
    void method1() { ... }           // instance method — per object
    static void method2() { ... }    // static method — per class
}

A obj1 = new A();
A obj2 = new A();
```

```
┌──────────────────────────── Class A ─────────────────────────────┐
│                                                                   │
│   static method2()  ← shared by ALL objects (belongs to class)   │
│                                                                   │
│   ┌─────── obj1 ───────┐     ┌─────── obj2 ───────┐             │
│   │   method1()        │     │   method1()        │              │
│   └────────────────────┘     └────────────────────┘              │
└───────────────────────────────────────────────────────────────────┘
```

### Calling Static vs Instance Methods

```java
// Instance method — needs object
obj1.method1();
obj2.method1();

// Static method — called via class name (no object needed)
A.method2();
```

### Critical Rule: Static Cannot Access Non-Static

```java
class Calculation {
    int stockPrice = 20;             // instance variable
    static int carPrice = 40;        // static (class) variable

    static int getPriceOfPen() {
        // carPrice → ✅ static can access static
        // stockPrice → ❌ CANNOT access (which object's stockPrice?)
        return carPrice;
    }
}
```

**Why?** A static method doesn't know which object's instance variable to use — it's not tied to any object.

```
  Static method:  "I belong to the class. I don't know obj1 vs obj2."
  
  stockPrice → exists in obj1 AND obj2 separately → ambiguous!
  carPrice   → only ONE copy for the class → no ambiguity ✅
```

> **Same applies to methods:** A static method cannot call a non-static method directly. You must specify the object: `obj1.print()`.

### Static Method Cannot Be Overridden

```java
class Person {
    static void profession() { System.out.println("Person"); }
}
class Doctor extends Person {
    static void profession() { System.out.println("Doctor"); }
    // This is METHOD HIDING, not overriding!
}

Person.profession();   // "Person"
Doctor.profession();   // "Doctor"
// No dynamic binding — each is linked to its class at compile time
```

> **Why?** Overriding uses dynamic binding (runtime, based on object type). Static methods are resolved at **compile time** via class name — no objects involved.

### When to Declare a Method Static?

| Criteria | Static? |
|---|---|
| Does NOT use instance variables | ✅ Yes |
| Only computes based on arguments | ✅ Yes |
| Modifies object state (instance variables) | ❌ No |
| Depends on instance variables | ❌ No |

```java
// ✅ Good static candidate — pure computation, no state
static int sum(int a, int b) {
    return a + b;
}

// ❌ Bad static candidate — uses and modifies instance variable
int sum2(int a, int b) {
    carPrice = carPrice + a + b;  // modifies state!
    return carPrice;
}
```

> **Real-world example:** Factory Design Pattern uses static methods — computation depends only on arguments.

## 2. Final Method

A `final` method **cannot be overridden** by subclasses.

```java
class Person {
    public final void profession() {
        System.out.println("I am a Person");
    }
}

class Doctor extends Person {
    @Override
    public void profession() { ... }  // ❌ Compilation Error!
}
```

**Why?** If you can't change the implementation, overriding is pointless — the inherited method is already available.

> Use `final` when you want to **lock down** a method's behavior and prevent subclasses from altering it.

## 3. Abstract Method

An abstract method has **only a declaration** (no body). It must be inside an **abstract class**, and the **child class must provide the implementation**.

```java
abstract class Car {
    abstract void pressBrake();        // declaration only — no body
    
    void getWheels() {                 // concrete method — has body
        System.out.println("4 wheels");
    }
}

class Audi extends Car {
    @Override
    void pressBrake() {                // child MUST implement
        System.out.println("Audi brake applied");
    }
}
```

| Rule | Detail |
|---|---|
| Keyword | `abstract` before return type |
| Body | **None** — only declaration |
| Where | Only in `abstract` class |
| Implementation | **Mandatory** in child (concrete) class |

## 4. Variable Arguments (Varargs)

When you **don't know how many arguments** will be passed, use varargs (`...`) instead of creating multiple overloaded methods.

```java
// ❌ Without varargs — many overloaded methods needed
int sum(int a) { ... }
int sum(int a, int b) { ... }
int sum(int a, int b, int c) { ... }
// ... and so on

// ✅ With varargs — single method handles any count
int sum(int... numbers) {
    int total = 0;
    for (int n : numbers) {
        total += n;
    }
    return total;
}
```

### Usage

```java
Calculation calc = new Calculation();
calc.sum(1, 2, 3);           // 3 args → works
calc.sum(3, 8, 9, 10);       // 4 args → works
calc.sum(1, 2, 3, 4, 5, 6);  // 6 args → works
```

### Rules

| Rule | Detail |
|---|---|
| **Only one** varargs per method | `void test(int... a, int... b)` → ❌ Invalid |
| Must be **last** parameter | `void test(int... a, int b)` → ❌ Invalid |
| Correct order | `void test(int b, int... a)` → ✅ Valid |

```java
// ✅ Correct — varargs is last
int sum(int fixed, int... variable) {
    int total = fixed;
    for (int v : variable) total += v;
    return total;
}
```

> **Why last?** Because varargs accepts "any number" — if placed first, Java can't determine where to stop assigning values to it.
